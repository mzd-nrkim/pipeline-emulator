# 캔버스 노드 배포상태 표시 — deployStatus 필드 + 상태 배지/grayout 차등

> 상태: 미시작 (plan-review 대기)

infra 뷰에서 `es`·`kibana`처럼 **실제 배포 상태가 다른 노드**가 현재 `outOfTeamScope: true` 하나로만 구분되어 **모두 같은 회색**으로 보인다. 그러나 두 노드의 성격은 다르다 — es는 `docker-compose.yml`에 **실재**(외부 팀 소관)하고, kibana는 compose에 **없음**(미배포/예정)이다. `outOfTeamScope`는 "조직 소관" 축이지 "배포 상태" 축이 아니라, 이 둘이 한 필드에 뭉개져 있다.

본 계획은 **배포 상태 축을 별도 필드(`deployStatus`)로 분리**하고, 그 값에 따라 상태 배지(예정/없음)와 grayout 강도를 차등 렌더하는 **일반 기능**을 구현한다. 특정 노드(kibana 등)에 이 상태를 지정하는 것은 **각 노드 계획의 책임**이며, 본 계획은 필드·렌더 뼈대만 제공한다(어떤 노드도 상태 미지정이어도 성립 = 회귀 0).

> 관련 계획: `infra_뷰_kibana_허위의존_처리.md`(G-3)가 본 계획을 **선행 의존**한다 — kibana의 "예정 컴포넌트" 표현을 주석이 아니라 본 계획의 `deployStatus: 'planned'` 배지로 실현한다. mysql 라벨 교정(G-4, `displayNameOverride`)은 **별 축**이라 본 계획과 무관·독립.

### 두 축 분리 (핵심 설계)

- **조직 축 `outOfTeamScope`**(기존 유지): "우리 팀 소관 밖" — muted 회색.
- **배포 축 `deployStatus`**(신규): `'active'`(기본·배포됨) | `'planned'`(예정) | `'absent'`(없음). 배지 + grayout 강도로 표현.
- 두 축은 **직교** — es = `active` + `outOfTeamScope`(실재·외부팀 회색), kibana = `planned`(또는 `absent`) + `outOfTeamScope`(예정 배지). 이로써 두 노드가 **다르게** 보인다.

## 목표

- `ToolNode`에 `deployStatus?: 'active'|'planned'|'absent'` 옵셔널 필드가 추가되고, 미지정 시 `active`로 취급되어 **기존 전 노드 회귀 0**이다.
- `ToolFlowNode`가 `deployStatus`에 따라 상태 배지(`planned`→기존 `PlannedBadge` 재사용, `absent`→"없음" 배지)를 렌더하고, grayout 강도를 차등(`active`=불투명, `planned`=현행 muted, `absent`=더 흐림/점선)한다.
- `outOfTeamScope`(조직 축)와 `deployStatus`(배포 축)가 **독립적으로 조합**되어, es(active·외부팀)와 kibana(planned·외부팀)가 시각적으로 구분된다.
- `data` 채널·핸들·채널 필터·`FlowNode.data`의 기존 필드·기타 노드 렌더는 **불변**(신규 필드 추가만).

## 접근 방법

1. **타입 필드 추가** (`$lib/api/types.ts`): `ToolNode`에 `deployStatus?: 'active' | 'planned' | 'absent'` 옵셔널 필드. 미지정 폴백 = `active`.

2. **buildNodesAndEdges가 FlowNode.data로 전달** (`buildNodesAndEdges.ts`): `FlowNode.data`에 `deployStatus` 추가(`n.deployStatus ?? 'active'`). 기존 `isInfra`·`outOfTeamScope` 전달 패턴(213행) 그대로 확장. `FlowNode` 인터페이스(83-100행)에도 옵셔널 필드 추가.

3. **ToolFlowNode 렌더 분기** (`ToolFlowNode.svelte`):
   - `deployStatus`를 `$derived`로 읽는다(기존 `outOfTeamScope` 패턴 16행처럼).
   - **배지**: `planned` → 기존 `PlannedBadge` 재사용(import), `absent` → "없음" 텍스트 배지(PlannedBadge 변형 or 인라인). 카드 우상단/라벨 옆 배치(trigger-badge 패턴 참고 47행).
   - **grayout 차등**: 기존 `.out-of-scope`(outOfTeamScope 전용)는 유지하고, `deployStatus`용 클래스(`.status-planned`/`.status-absent`)를 추가해 opacity·점선 테두리 차등. 두 축 클래스가 **중첩 적용** 가능하도록 CSS 독립 유지(es는 out-of-scope만, kibana는 out-of-scope + status-planned).

4. **데이터 지정은 본 계획 범위 밖**: 어느 노드가 planned/absent인지는 각 노드 계획(kibana 계획 등)이 `topology.ts`에서 지정한다. 본 계획은 **필드·렌더만** 구현하고, 데모/검증용으로만 필요 시 임시 확인(실제 값 커밋은 각 계획).

## 실행 시 필수 고려사항

### 옵셔널 필드 + active 폴백 → 회귀 0

`deployStatus` 미지정 노드는 전부 `active`로 취급되어 기존 렌더와 동일하다. 신규 배지·status 클래스는 `planned`/`absent`에만 적용 → 현재 topology의 모든 노드(값 미지정)는 시각 변화 0. 본 계획 단독 머지 시 **화면상 변화 없음**이 정상(기능 뼈대만).

### 두 축 CSS 독립 (중첩 안전)

`outOfTeamScope`(`.out-of-scope`)와 `deployStatus`(`.status-*`)는 서로 다른 클래스로, opacity가 곱연산되지 않도록 주의(둘 다 opacity 낮추면 과하게 흐려짐). grayout은 **한 축에서만** 적용하거나(권장: deployStatus가 지정되면 그쪽이 opacity 주도, outOfTeamScope는 배경 tint/라벨색만), 값 조합 매트릭스를 plan-review에서 확정한다.

### PlannedBadge 재사용 vs 신규

`planned`은 기존 `PlannedBadge.svelte`("예정") 그대로 재사용. `absent`("없음")은 PlannedBadge에 prop(라벨) 추가 or 소형 신규 컴포넌트. **기본안: PlannedBadge에 `label` prop 추가**해 "예정"/"없음" 공용화(신규 컴포넌트 최소화). 단 PlannedBadge가 다른 곳(Stage/Dimension/PiiCount)에서 무인자로 쓰이므로 **기본값 "예정" 유지**해 기존 사용처 회귀 0.

### Node 정적/e2e 게이트 위치 (Z-post)

`frontend/package.json` 의존(check/build/vitest/playwright)은 gitignored `node_modules` 필요 → 워크트리 신뢰 불가 → **머지 후 원본 main(Z-post)**. DB·마이그레이션 무관.
