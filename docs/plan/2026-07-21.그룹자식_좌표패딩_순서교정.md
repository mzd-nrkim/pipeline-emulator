# 그룹 자식 좌표 패딩·순서 교정

> 상태: 미시작

playwright 스크린샷으로 `/sample/pipeline` 데이터흐름 뷰를 확인한 결과 두 가지 렌더 버그가 발견됐다. (1) 그룹 자식 노드의 y=0이라 그룹 박스 상단 테두리에 노드가 겹쳐 보임 — `PAD_TOP` 오프셋 누락. (2) docling이 아닌 presidio가 가장 왼쪽에 표시됨 — `topology.ts`에서 presidio(L128)가 docling(L142)보다 먼저 정의돼 있어 배열 순서(filter 순)로 idx가 결정되기 때문. DAG 깊이 기준 정렬 후 `PAD_X + idx * COL_GAP` 좌표를 사용해야 한다.

---

## 목표

- 그룹 자식 노드가 `position.x = PAD_X + idx * COL_GAP`, `position.y = PAD_TOP`으로 렌더된다.
- DAG 깊이(절대 position.x 기준 오름차순) 정렬 후 idx를 부여해 배열 정의 순서에 무관하게 docling이 가장 왼쪽에 위치한다.
- 단위테스트가 새 좌표 기대값·순서 검증을 포함해 통과한다.

## 접근 방법

1. `buildNodesAndEdges.ts` 그룹 자식 좌표 재배치 블록(현재 L232-235)을 수정:
   - `childNodes`를 원래 `position.x` 기준 오름차순 정렬(DAG 깊이 = 위상정렬 결과)
   - 정렬 후 `{ x: PAD_X + idx * COL_GAP, y: PAD_TOP }`으로 상대좌표 배정
   - `absMinX / absMinY`는 정렬 전 원래 절대좌표에서 이미 캡처됨 → 그룹 박스 위치 계산에 영향 없음
2. `buildNodesAndEdges.test.ts` E-1 기대값 업데이트:
   - `position.x` 기대값: `0 → PAD_X`, `COL_GAP → PAD_X+COL_GAP`, `2*COL_GAP → PAD_X+2*COL_GAP`
   - `position.y` 기대값: `0 → PAD_TOP(60)`
   - 순서 회귀 테스트 추가: `groupTopology` 픽스처에서 presidio를 docling보다 **앞에** 정의해 실제 topology.ts 버그 재현, 정렬 후 docling이 x=PAD_X(idx=0)인지 단언
