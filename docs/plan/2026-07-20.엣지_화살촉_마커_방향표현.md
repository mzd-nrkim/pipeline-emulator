# 엣지 화살촉·마커 방향 표현

pipeline-emulator 그래프의 엣지는 현재 **화살촉이 없다**. `buildNodesAndEdges.ts`의 `FlowEdge`(라인 102–108)는 `id·source·target·animated·label·sourceHandle`만 담고 `markerEnd`가 없으며, 두 뷰(`ToolCanvasView.svelte:41`, `PipelineGraphView.svelte:77`)의 `<SvelteFlow>`에도 `defaultEdgeOptions`·`markerEnd` 지정이 없다. `@xyflow/svelte` v1 기본 엣지는 `markerEnd`를 명시해야만 화살촉이 렌더되므로, 지금은 **화살촉 없는 베지어 곡선**만 그려지고 방향감은 좌(target)/우(source) 핸들 배치와 data 뷰의 `animated` 점선으로만 전달된다. 본 계획은 엣지의 방향 표현 정책(화살촉 추가 여부·마커 종류·엣지 타입·조건 라벨)을 확정하고 일괄 적용한다.

---

## 목표

- 엣지 방향 표현 정책이 **명시적으로 결정**된다: 화살촉 추가 여부(n8n은 미사용 → 미추가가 정합적일 수 있음), 추가 시 마커 종류(`Arrow` vs `ArrowClosed`)·크기·색.
- 결정이 "추가"이면 두 뷰의 모든 엣지에 `markerEnd`가 **일괄·일관** 적용된다(뷰·노드별 누락 없음).
- route 노드의 **다중 출력 엣지**(`sourceHandle`)도 동일 마커 정책을 따른다.
- 엣지 `condition` 라벨(현재 `label = e.condition`)과 마커가 **시각적으로 충돌하지 않는다**.
- 마커 색이 DESIGN.md 토큰·라이트/다크 모드에서 자연스럽게 보인다(하드코딩 색 지양).
- 기존 e2e(`.svelte-flow__edge*` 셀렉터·route-split)가 **회귀 없이** 통과한다.

## 접근 방법

1. **정책 결정 (선행)**: 실제 n8n은 화살촉을 쓰지 않고 입력 핸들에 곡선이 플러그처럼 꽂힌다. 두 가지 방향 —
   - **(A) 미추가 유지 (n8n 정합)**: 화살촉 없이 방향은 좌/우 핸들·`animated`로 전달. 이 경우 본 계획은 "결정·문서화 + 방향감 보강(옵션)"에 그친다.
   - **(B) 화살촉 추가 (가독성 우선)**: 노드가 좌우로 촘촘할 때 방향 판별을 돕기 위해 target 끝에 `ArrowClosed` 마커. n8n 룩에서 살짝 벗어남.
   - 기본 권장: **(A)**. 단 실제 그래프에서 방향 혼동이 관찰되면 (B)로 전환. → D-0에서 확정.

2. **일괄 적용 지점 선택 (B 결정 시)**: 엣지마다 `markerEnd`를 다는 대신 `<SvelteFlow defaultEdgeOptions={{ markerEnd: { type: MarkerType.ArrowClosed, width, height, color } }}>`로 **뷰 레벨 1곳**에 적용하는 것을 기본으로 한다(누락·불일치 방지). 두 뷰(`ToolCanvasView`·`PipelineGraphView`)에 동일 옵션 적용. `MarkerType`은 `@xyflow/svelte`에서 export됨(확인 완료). 노드/엣지별로 마커를 달리해야 하면 그때만 `buildNodesAndEdges.ts`의 `FlowEdge`에 `markerEnd` 필드를 추가한다.

3. **마커 색·크기**: `color`는 `--border`/`--text-muted` 계열 CSS 변수를 참조(하드코딩 hex 금지). SVG 마커는 CSS 변수 상속에 제약이 있을 수 있으므로, 안 되면 라이트/다크 각각에 맞는 중립 색을 `defaultEdgeOptions`에 상수로 두되 DESIGN.md 팔레트에서 채택. `animated`(data 뷰) 엣지와 정적 엣지의 마커가 동일하게 보이는지 확인.

4. **condition 라벨 공존**: route 분기 엣지는 `label`(condition)과 마커가 target 근처에 겹칠 수 있다. 라벨 위치(`labelBgStyle`/기본 중앙)와 마커(target 끝)가 겹치지 않는지 실측. 필요 시 엣지 타입을 `smoothstep`으로 바꿔 분기 가독성을 높이는 것도 검토(D-0 종속, 선택).

5. **엣지 타입 검토 (선택)**: 현재 기본 `bezier`. n8n은 부드러운 베지어를 쓰므로 유지가 자연스럽다. LR 흐름에서 분기 가독성이 나쁘면 `smoothstep`/`step` 대안을 D-0에서 함께 결정(범위 넘치면 별도).

6. **회귀 방어**: e2e가 `.svelte-flow__edge`·`.svelte-flow__edge-path` 등 xyflow 내부 클래스에 의존. `markerEnd` 추가는 DOM에 `<marker>` def와 `marker-end` 속성을 더할 뿐 엣지 개수·클래스는 불변이므로 회귀 위험은 낮다. Z-post playwright로 실측 확인.

## 실행 시 필수 고려사항

### D-0 결정 항목 (착수 전 확정)

1. **화살촉 추가 여부**: (A) 미추가 유지 [기본] vs (B) ArrowClosed 추가.
2. (B 시) **마커 종류·크기·색**: `Arrow`(열림) vs `ArrowClosed`(채움), `width/height`, `color`(중립 토큰).
3. **엣지 타입**: `bezier` 유지 [기본] vs `smoothstep` 전환.
4. **적용 범위**: 두 뷰 모두 동일 정책(기본) vs 뷰별 차등.

> (A) 확정 시 R 그룹 대부분 SKIP, 본 계획은 "결정 문서화 + 회귀 없음 확인"으로 축소된다.

### 뷰 2곳 일관성

`ToolCanvasView.svelte`·`PipelineGraphView.svelte`가 각각 `<SvelteFlow>`를 갖는다. 마커 정책을 한쪽만 적용하면 뷰 간 불일치가 생긴다 → **두 뷰 동시 적용**을 원칙으로, 공통 옵션 상수를 한 곳(예: 유틸/상수 모듈)에 정의해 재사용하는 것을 검토.

### data 계약 불변 우선

`defaultEdgeOptions`(뷰 레벨)로 해결되면 `buildNodesAndEdges.ts`·단위테스트(`buildNodesAndEdges.test.ts`)를 건드리지 않는다 → 회귀 표면 최소. `FlowEdge`에 `markerEnd` 필드를 신설하는 건 엣지별 차등이 꼭 필요할 때만.

### 다크 모드 마커 가시성

흰 배경·어두운 배경 모두에서 마커가 보여야 한다. 하드코딩 `#000`/`#fff` 금지. SVG `<marker>`의 `fill`이 CSS 변수를 못 받으면 중립 회색(양쪽 대비 확보) 채택.

### Node 정적/e2e 게이트 위치 (Z-post)

`npm run check`·`build`·`vitest`·`playwright`는 gitignored `node_modules` 의존 → 워크트리(Z-pre) 신뢰 불가 → **머지 후 원본 main(Z-post)** 실행. DB·마이그레이션 변경 없음(순수 프론트 렌더).
