# 실행주체 구조 캔버스 재설계 (Airflow 오케스트레이션 + MySQL 데이터 버스)

> 상태: 통테통과-완료
> 선행 의존: `2026-07-20.데이터흐름_토폴로지_정합성_교정.md`(배선교정) 머지 후 착수 — 본 계획은 교정된 흐름 위에 시각 구조를 재설계한다.

현재 캔버스는 파이프라인을 **선형 peer 체인**(`s3-bronze → airflow → presidio → docling → kure → valkey → {es, mysql}`)으로 그린다. 그러나 실제 실행 구조는 다르다(근거: `dags/*.py` 전수 확인).

- **Airflow가 실행 엔진**이다. `silver_1_structuring`·`silver_2_masking`·`gold_3_chunking`·`gold_4_enrichment`·`gold_5_field_mapping` 5개 DAG를 스케줄·실행하며, 각 `@task`가 Airflow worker(LocalExecutor=스케줄러 / CeleryExecutor=Valkey 브로커+워커)에서 돈다. presidio·docling·kure는 airflow와 **peer가 아니라 그 안에서 도는 스테이지**다.
- **스테이지 간 데이터는 MySQL medallion 테이블로 오간다.** 각 DAG가 MySQL 테이블 read→write: `bronze_* → silver_structured_documents → silver_masked_documents → gold_chunked_documents → gold_enriched_documents → gold_staged_documents`(근거: `gold_3_chunking.py:52,96`·`gold_4_enrichment.py:58,125`·`gold_5_field_mapping.py:70,148`·`silver_2_masking.py:51,74`). 노드끼리 직접 데이터를 넘기지 않는다.
- **스테이지가 부르는 도구**: presidio=worker 내 라이브러리(`silver_2_masking.py:65` 인프로세스, 별도 서비스 아님), docling=**미사용**(`silver_1_structuring.py`에 import 없음, 라벨 허위), mock-api=진짜 외부 HTTP(`gold_3`/`gold_4`가 `http://mock-api:8000` 호출).

본 계획은 캔버스를 이 **실행주체 구조**에 맞게 재설계한다 — Airflow를 5개 스테이지를 감싸는 오케스트레이션 경계로, MySQL medallion을 스테이지 간 데이터 버스로 표현한다.

## 현재 캔버스의 3대 허구 (교정 대상)

1. **Airflow가 "한 스텝"으로 그려짐** → 실제로는 전 transform 스테이지를 실행하는 엔진. 스테이지를 감싸는 경계(오케스트레이션 컨테이너)여야 함.
2. **MySQL이 "종착 아카이브"로 그려짐** → 실제로는 스테이지 간 데이터 버스(medallion 테이블). bronze/silver/gold 계층이 데이터를 릴레이.
3. **presidio/docling이 독립 서비스 노드** → 실제로는 airflow worker 내부 로직(docling 미사용). mock-api만 외부 서비스.

## 목표

- 캔버스가 Airflow를 **5개 스테이지(silver_1·silver_2·gold_3·gold_4·gold_5)를 감싸는 오케스트레이션 경계**로 표현한다(스테이지가 airflow 내부에 있음이 시각적으로 명확).
- 스테이지 간 데이터 흐름이 **MySQL medallion 테이블(bronze→silver→gold)을 경유**함이 드러난다(직접 hand-off 허구 제거).
- 스테이지↔도구 관계가 정확하다: presidio=airflow 내부 라이브러리, docling 라벨=실제 구조화 명칭으로 교체(미사용 도구명 제거), mock-api=외부 HTTP 호출 대상.
- Valkey는 배선교정 계획대로 인프라(Celery 브로커/CDC 싱크)로 유지(본 계획에서 데이터 스테이지로 되돌리지 않음).
- 재설계 후에도 fan-in(수집 3종)·infra 뷰·nodeclick 드릴다운이 동작하고, 변경된 노드/엣지 구조에 맞춰 e2e 기대값이 재작성된다.

## D-0 결정 항목 (착수 전 확정 — 시각 메타포가 핵심)

1. **Airflow 경계 표현 방식** — (a) **xyflow group/parent 노드**(subflow: `parentId`+`extent:'parent'`로 스테이지를 airflow 컨테이너 자식으로) vs (b) **배경 사각형/레인 주석**(스테이지는 일반 노드, 뒤에 "Airflow" 경계 박스만) vs (c) **접기/펼치기 그룹**(airflow 노드 클릭 시 스테이지 확장). → **기본안 (a) xyflow subflow** — 가장 사실적이나 `@xyflow/svelte` group 노드 지원·핸들/엣지 상호작용 실측 필요(작업목록 I-1). 리스크 크면 (b)로 강등.
2. **MySQL 데이터 버스 표현** — (a) medallion 테이블을 **중간 store 노드**로 명시(bronze/silver_structured/silver_masked/gold_chunked/gold_enriched/gold_staged — 노드 수 급증) vs (b) **엣지 라벨**로 경유 테이블 표기(노드 유지, 엣지에 테이블명) vs (c) **단일 MySQL 버스 노드**에 각 스테이지가 read/write 엣지. → **기본안 (b) 엣지 라벨** — 노드 폭증 없이 데이터 버스 사실 전달. 사실성 더 필요하면 (c) 버스 노드.
3. **docling 라벨 교체** — `node-docling`을 실제 구조화 명칭으로(예: "Structuring"/"JSON Schema 변환"). tool 카탈로그 `docling-langchain` 엔트리 처리(라벨만 교체 vs 카탈로그 엔트리 개명). → 기본안: 노드 라벨/표시만 교체, 카탈로그 id는 회귀 최소화 위해 유지하되 displayName 정정.
4. **presidio/외부 표기** — presidio=airflow 내부(경계 안), mock-api=외부(경계 밖 + 스테이지에서 HTTP 엣지). 시각적으로 내부/외부 구분 방식 확정(경계 박스 안/밖 배치 + dependency 채널).

## 접근 방법

1. **xyflow subflow 실측 (I-1)**: `@xyflow/svelte` 현재 버전에서 group/parent 노드(`parentId`, `extent:'parent'`, `type:'group'`)와 그 안 자식 노드의 핸들·엣지·nodeclick 동작을 실측한다. `ToolCanvasView`/`PipelineGraphView`가 subflow를 렌더 가능한지 확인. 불가/고위험이면 D-0.1을 (b) 배경 경계 박스로 강등.
2. **토폴로지 모델 확장 (`topology.ts` + `api/types.ts`)**: 노드에 그룹 소속(`parentId`/`group:'airflow'`) 필드, 엣지에 경유 테이블 라벨(`viaTable?`) 필드 추가(D-0.2 (b) 채택 시). CanvasTopology 타입 확장. 배선교정 계획이 정리한 edges 위에 그룹/라벨 메타 부가.
3. **build 계층 반영 (`buildNodesAndEdges.ts`)**: 그룹 노드 생성·자식 `parentId` 매핑·엣지 라벨 주입. `data` 계약에 그룹/라벨 필드 추가(신규 필드는 하위호환 optional).
4. **그룹/스테이지 렌더 (`ToolFlowNode.svelte` + 신규 group 노드 컴포넌트)**: Airflow 경계 컨테이너 노드 컴포넌트 신설(제목 "Airflow (CeleryExecutor)", 내부에 스테이지 배치). 스테이지 노드는 기존 `ToolFlowNode` 재사용하되 경계 안 배치. 외부 서비스(mock-api)는 경계 밖 + HTTP dependency 엣지.
5. **뷰 반영 (`ToolCanvasView`·`PipelineGraphView`)**: subflow/경계 렌더, 레이아웃(스테이지가 경계 안 가로 배치), MySQL 버스 라벨/노드 표시. 데이터뷰/인프라뷰 각각 경계 표현 적용 범위 확정.
6. **docling 라벨·presidio 내부화 (D-0.3/0.4)**: docling 노드 라벨 교체, presidio를 경계 안 스테이지로, mock-api를 외부 호출 대상으로 재배치.
7. **회귀 방어·기대값 재작성**: 노드 구조(그룹/자식)·엣지·핸들이 크게 바뀌므로 `buildNodesAndEdges.test.ts`·`graph-lr-handle`·`route-split`·`infra-view`·`node-detail-modal` 기대값을 **재설계 구조 기준으로 재작성**(단순 통과 아님). nodeclick 드릴다운이 경계 안 자식 노드에서도 동작하는지 실측.

## 실행 시 필수 고려사항

### 배선교정 계획과의 관계 (선행 의존)

본 계획은 배선교정(Valkey 인프라화·silver 순서·gold_4 편입)이 **머지된 뒤** 착수한다. 배선교정이 edges 정합성을 확보하고, 본 계획은 그 위에 **시각 구조(그룹/버스)**를 얹는다. 두 계획을 동시에 진행하면 `topology.ts` 충돌 → 순차 진행.

### 아이콘 교체 계획(①)과의 동일 파일 편집 충돌

아이콘 교체 계획(`2026-07-20.toolflownode_브랜드로고_lucide폴백_아이콘.md`)이 `ToolFlowNode.svelte`의 `.node-icon` 렌더(R-1/R-2)를 수정하고, 본 계획도 같은 파일의 displayName·presidio 표기(R-2)를 수정한다. **동일 파일 병렬 편집 충돌**이 발생하므로, 두 계획을 함께 실행할 경우 `ToolFlowNode.svelte`를 건드리는 Phase는 병렬로 돌리지 말고 순차 배치한다(아이콘 계획을 먼저 머지한 뒤 본 계획 착수 권장 — 아이콘 계획은 배선교정과 독립이므로 순서 자유).

### xyflow subflow 리스크 (기본안 강등 경로)

group/parent 노드는 xyflow 버전별 지원·핸들 좌표계·엣지 라우팅이 까다롭다. I-1 실측에서 자식 노드 핸들/nodeclick/엣지가 불안정하면 **D-0.1을 배경 경계 박스(b)로 강등**한다(스테이지는 일반 노드, airflow는 뒤 배경 rect + 라벨). 사실성은 약간 낮지만 회귀 위험이 훨씬 작다.

> **plan-review 실측(2026-07-20)**: `@xyflow/svelte@1.6.2`는 group 노드를 **지원한다** — `@xyflow/system`의 NodeBase에 `parentId?`·`extent?:'parent'` 필드 존재, `dist/lib/components/nodes/GroupNode.svelte`(빌트인 GroupNode 컴포넌트) 존재. 단 **현재 코드베이스는 미사용**(ToolCanvasView `nodeTypes={{ tool }}`만 등록, ToolNode에 parentId 없음). 따라서 I-1은 "지원 여부"가 아니라 **자식 핸들 좌표계·nodeclick 이벤트 버블링·엣지 라우팅의 실동작**을 실측하는 것이 핵심. 빌트인 GroupNode 재사용 vs 신규 AirflowGroupNode(R-1) 커스텀 여부도 I-1에서 판단.

### PipelineGraphView는 CanvasTopology 미사용 — 재설계 범위 대상 아님 (범위 명확화)

plan-review 실측 결과, `PipelineGraphView.svelte`는 `ToolCanvasView`와 달리 **CanvasTopology/buildNodesAndEdges를 쓰지 않고 별도 Stage 데이터 모델(Bronze/Silver/Gold/Serving 레이어)로 독자 렌더**한다. 따라서 Airflow 경계·MySQL 버스·group 노드 재설계는 **ToolCanvasView(+buildNodesAndEdges+topology.ts) 단일 축**에 적용된다. I-2·V-1의 "두 뷰" 표현은 정확히는 **ToolCanvasView 데이터뷰/인프라뷰 두 채널**을 의미하며, PipelineGraphView는 이번 재설계에서 **무변경**(또는 별도 계획). I-2 실측 시 PipelineGraphView가 topology 그룹 필드에 영향받지 않음을 확인하고, V-1의 subflow 등록은 ToolCanvasView `<SvelteFlow>`에만 적용한다.

> **정정 (plan-review 실측 2026-07-20)**: PipelineGraphView가 `StageNode.svelte`로 렌더한다는 서술은 부정확하다. 실제로는 `PipelineGraphView.svelte:4,81`에서 **`ToolFlowNode`를 import해 `nodeTypes={{ tool: ToolFlowNode }}`로 렌더**한다(노드 data는 Stage 모델에서 `{ label: stage.name, planned, animated }`만 주입, PipelineGraphView.svelte:39). `StageNode.svelte`는 파일로 존재하나 이 뷰에서 미사용(사실상 dead). **함의**: topology/buildNodesAndEdges 기반 group·viaTable 재설계(M·V-1)는 PipelineGraphView에 영향 없으나, **R-2가 `ToolFlowNode.svelte` 렌더 자체(예: displayName 표시 로직 :57/:116)를 건드리면 공유 컴포넌트라 PipelineGraphView에도 반영된다**. R-2는 docling displayName을 `toolCatalog.ts`/`displayNameOverride`(데이터)로 정정하고 ToolFlowNode 렌더 로직은 무변경으로 유지해 PipelineGraphView 무변경 계약을 지킨다 — ToolFlowNode.svelte 마크업/스타일을 직접 수정할 경우 PipelineGraphView 회귀(핸들·모달)도 함께 확인한다.

### 아이콘·아이콘 렌더 현황 (충돌 재확인)

`ToolFlowNode.svelte`의 아이콘 렌더는 이미 `nodeIcon.ts:resolveNodeIcon()`(brand/lucide/emoji 분기, `.node-icon` 클래스)로 동적화되어 있고 `SI_ICONS`에 langchain 엔트리 존재. R-2가 손대는 것은 아이콘이 아니라 **displayName 렌더(`ToolFlowNode.svelte:119` `resolvedDisplayName`)**이므로, 아이콘 계획(`.node-icon` 렌더 R-1/R-2)과 실제 편집 라인이 다를 수 있으나 **같은 파일**이라 여전히 순차 배치 원칙 유지.

### 노드 수 폭증 경계

D-0.2 (a) medallion 테이블 노드화는 노드가 6개+ 늘어 캔버스가 복잡해진다. 기본안 (b) 엣지 라벨을 우선하고, 사용자가 테이블 가시성을 명시 요구할 때만 (a)/(c)로 확대.

### e2e 대규모 회귀 (구조 변경)

노드/엣지 구조가 근본적으로 바뀌므로 기존 e2e의 핸들 개수·노드 셀렉터·모달 드릴다운이 대부분 영향받는다. **회귀 통과가 아니라 재설계 구조 기준 기대값 재작성**이 핵심 작업이다. 특히 `graph-lr-handle`(핸들 개수)·`node-detail-modal`(경계 안 자식 클릭)·`infra-view`(경계와 dependency 채널 공존)를 재실측.

### 카탈로그·계약 하위호환

`toolCatalog`·`buildNodesAndEdges` `data` 계약에 그룹/라벨 필드를 추가할 때 optional로 두어, 그룹 미지정 노드(수집·싱크)는 기존 렌더 유지. docling 카탈로그 id 유지(displayName만 정정)로 셀렉터 회귀 최소화.

### Node 게이트 위치 (Z-post)

`frontend/package.json` 존재 → `check`/`build`/`vitest`/`playwright`는 워크트리 신뢰 불가 → 머지 후 원본 main(Z-post). DB·마이그레이션 무관(mock 데이터 + 뷰 렌더).
