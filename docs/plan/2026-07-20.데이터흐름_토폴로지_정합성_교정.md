# 데이터흐름 토폴로지 정합성 교정 (Valkey 위치·silver 순서·gold_4 누락)

> 상태: 통테통과-완료

`frontend/src/lib/mock/topology.ts`의 샘플 캔버스 토폴로지(주석상 "hyundaimotor-lllm 파이프라인 반영")가 **실제 파이프라인 정의(`dags/` DAG 파일 + `docker-compose.yml` + `.env.example` + CLAUDE.md 스테이지 표)와 3건 불일치**한다. 데이터흐름도를 검증한 결과 아래 근거로 확정된 문제다. 본 계획은 토폴로지를 실제 파이프라인 시맨틱에 맞춰 교정한다.

## 검증 근거 (SSOT 대조)

실제 스테이지 순서 (CLAUDE.md:135-139 · `dags/*.py`):

```
silver_1_structuring(docling) → silver_2_masking(presidio) → gold_3_chunking(kure) → gold_4_enrichment(mock-api) → gold_5_field_mapping(es)
```

Valkey 실제 역할 (`docker-compose.yml` · `.env.example`):
- Debezium CDC **Redis Stream 싱크** — `.env.example:39`("Debezium이 생성하는 Redis Stream 키 패턴: pipeline.<db>.<table>"), `docker-compose.yml:201`(`DEBEZIUM_SINK_TYPE: redis`).
- Airflow **Celery 브로커** — `docker-compose.yml:165`(`CELERY_BROKER_URL=redis://valkey:6379/1`), `.env.example:53`. (profiles: [celery], 기본 off)
- transform DAG 데이터 경로에는 Valkey **미등장** — 전 스테이지 MySQL 테이블 경유(`gold_5_field_mapping.py:54-174` MySQL read→insert).

## 확정 문제 3건

### P1. Valkey를 선형 데이터 스테이지로 오배치 (`topology.ts:151-161, 253-257`)

- 현재: `node-valkey`(role `broker`)가 `kure → valkey → {es, mysql}` 데이터 체인 중간에 위치.
- 실제: Valkey는 (a) 수집 단 Debezium의 CDC 스트림 싱크, (b) Airflow Celery 브로커 인프라. KURE(임베딩)와 싱크 사이의 transform 데이터 스테이지가 **아니다**.
- 영향: 데이터뷰가 "임베딩 결과가 Valkey를 거쳐 ES/MySQL로 분기"한다는 **허위 흐름**을 보여준다.

### P2. silver_1 / silver_2 실행 순서 반전 (`topology.ts:250-252`)

- 현재: `airflow → presidio(silver_2_masking) → docling(silver_1_structuring) → kure`.
- 실제: 구조화(silver_1)가 마스킹(silver_2) 앞 → `docling → presidio` 순이어야 함. `gold_3_chunking.py:3`이 "silver_masked_documents 읽어 청킹"이라 마스킹이 청킹 직전임도 교차확인됨.

### P3. gold_4_enrichment(mock-api) 데이터 흐름 누락 (`topology.ts:227-237, 272`)

- 현재: `node-mock-api`(dagId `gold_4_enrichment`)가 airflow로 가는 **dependency 엣지로만** 존재. 데이터 체인은 gold_3→gold_5로 점프하며 enrichment를 건너뜀.
- 실제: `gold_3_chunking → gold_4_enrichment → gold_5_field_mapping`. mock-api가 데이터 체인에 있어야 함.

- 데이터뷰 흐름이 실제 DAG 스테이지 순서(`silver_1 → silver_2 → gold_3 → gold_4 → gold_5`)와 일치한다.

## 실행 시 필수 고려사항

### mock 토폴로지 = 시각화 SSOT

`topology.ts`는 실행 로직이 아니라 캔버스 시각화용 샘플 데이터다. 따라서 교정은 **런타임 파이프라인 동작에 영향 없고**, 데이터뷰/인프라뷰 렌더 및 관련 e2e에만 영향을 준다. 단 주석 다이어그램이 SSOT 역할을 하므로 코드-주석 동기화가 필수다.

### 실제 파이프라인 확인 의존성

D-0.1(Valkey가 스테이지 간 실데이터 버퍼인지)·D-0.2(MySQL 아카이브 분기점)는 **실제 hyundaimotor-lllm 파이프라인 지식**이 필요하다. DAG 파일로 유추 가능한 범위(`gold_5_field_mapping.py`가 MySQL read→insert, Valkey 미등장)를 넘어서면 사용자 확인 후 확정한다. 근거 없이 추정 배선하지 않는다.

### 엣지 개수 변동 → e2e 기대값 갱신

엣지를 추가·제거·재배선하면 `graph-lr-handle.spec.ts`의 핸들 개수(route 다중출력 등)·`buildNodesAndEdges.test.ts`의 노드/엣지 카운트가 바뀐다. 회귀 "통과"가 아니라 **교정 흐름 기준으로 기대값을 재작성**한 뒤 통과 확인한다(수치 근거 동반).

### infra 뷰 dependency 채널 무결성

Valkey를 dependency 노드로 전환하면 infra 뷰(`infra-view.spec.ts`)의 dependency 채널 렌더·식별성에 영향. 기존 `mysql-container`/`seaweedfs`/`mock-api` dependency 노드 패턴과 동일 규약을 따르는지 확인.

### Node 게이트 위치 (Z-post)

`frontend/package.json` 존재 → `check`/`build`/`vitest`/`playwright`는 워크트리 신뢰 불가 → 머지 후 원본 main(Z-post) 실행. DB·마이그레이션 무관(mock 데이터 편집).
