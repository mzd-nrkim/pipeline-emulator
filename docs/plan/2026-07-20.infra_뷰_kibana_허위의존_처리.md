# infra 뷰 `es→kibana` 허위 의존 처리 — compose ground truth 정합

> 상태: 미시작 (plan-review 대기)
> 분리 출처: `2026-07-20.infra_뷰_의존성_배선_교정.md`의 D-3에서 분리(사용자 확정 2026-07-20).

`frontend/src/lib/mock/topology.ts`의 dependency 채널 엣지 `node-es → node-kibana`(260행)는 infra 뷰에 kibana를 elasticsearch 의존 컴포넌트로 그린다. 그러나 `docker-compose.yml` 서비스 목록(seaweedfs·mysql·airflow·mock-api·elasticsearch·ui-backend·valkey·debezium·zookeeper·nifi)에 **kibana 서비스가 없다** → 실제 컨테이너 없는 **허위 의존**이다. 형제 계획(dependency 배선 교정)이 확립한 **compose ground truth 원칙**과 어긋나는 마지막 잔여 항목이다.

단 `node-es`·`node-kibana`는 `outOfTeamScope: true`(예정/외부 관측 계층 표식)를 이미 갖고 있어, "아직 배포되지 않은 예정 컴포넌트"로 의도적으로 남겼을 여지가 있다. 본 계획은 이 엣지를 **제거할지(a) / 예정 컴포넌트로 명문화해 유지할지(b)** 를 결정하고 반영한다.

> 형제 계획은 `es→kibana`를 **status quo(불변)** 로 두기로 했으므로, 본 계획이 이 엣지의 유일한 처리 주체다. 두 계획의 실행 순서는 무관(독립 파일 라인이 아니라 동일 파일이나 서로 다른 엣지/영역을 만지므로, 동시 실행 시 topology.ts 편집 충돌만 직렬화하면 됨).

> **선행 의존(신규)**: `노드_배포상태_표시_deployStatus.md`(P). 본 계획의 기본안 (b)는 kibana의 "예정 컴포넌트"를 **주석이 아니라 P의 `deployStatus: 'planned'` 상태 배지**로 실현한다 → P 머지 후 착수. P가 `deployStatus` 필드·배지 렌더를 제공하고, 본 계획은 `node-kibana`에 값만 지정한다.

### 결정 D-3-a vs D-3-b (본 계획 착수 전 확정)

- **(a) 제거**: `es→kibana` 엣지 + `node-kibana` 노드를 topology.ts에서 삭제. compose에 없는 것은 그래프에도 없다 — ground truth 원칙 엄격 적용. infra 뷰에서 kibana 사라짐. **data 뷰에도 kibana가 dependency 전용이라 원래 미표시** → data 뷰 영향 0.
- **(b) 유지 + 명문화**: 엣지·노드 유지하되, kibana가 compose 미포함 **예정/외부 관측 컴포넌트**임을 근거 주석으로 명문화. `outOfTeamScope: true`가 시각적으로 이미 구분(muted 렌더). "누락 아니냐" 재질문 차단.

> **기본안: (b) 유지 + 명문화.** 근거: (1) `outOfTeamScope` 표식이 이미 "팀 범위 밖/예정" 의미로 존재하고 muted 렌더로 시각 구분됨 — 허위가 아니라 "예정"으로 해석 가능. (2) 관측(Kibana) 계층은 ES 배포 후 자연스러운 후속이라 로드맵 상 유효. (3) 제거는 정보 손실이나, 명문화는 근거를 남기며 향후 실제 배포 시 재작업 불필요. plan-review에서 사용자 최종 확정.

## 목표

- `es→kibana` dependency 엣지의 상태가 **compose ground truth 원칙과 명시적으로 정합**한다(허위 방치 0).
- (기본안 b) kibana가 compose 미포함 **예정/외부 관측 컴포넌트**임이 topology.ts 근거 주석으로 명문화되어, 향후 "누락/허위 아니냐" 재질문이 차단된다.
- (대안 a 선택 시) `es→kibana` 엣지 + `node-kibana` 노드가 제거되고, infra 뷰에서 kibana가 사라지되 다른 노드 고아화는 0이다.
- `data` 채널·기존 카탈로그(`kibana` 엔트리 포함)·`ToolFlowNode` 렌더 계약은 **불변**(회귀 0).
- 형제 계획(dependency 배선 교정)과 topology.ts 편집이 충돌 없이 정합한다.

## 실행 시 필수 고려사항

### data 채널·카탈로그·렌더 불변

kibana는 dependency 채널 전용(data 엣지 0)이라 data 뷰 회귀 표면이 없다. `ToolFlowNode`·`toolCatalog`의 `kibana` 엔트리는 (b)에서 완전 불변, (a)에서도 카탈로그 엔트리는 유지 기본. 핸들·채널 필터 계약 불변.

### 형제 계획과의 topology.ts 편집 충돌

형제 계획(T-1~T-5)과 본 계획이 같은 `topology.ts`를 만진다. 서로 다른 엣지/영역이나 **동일 파일**이므로 동시 워크트리 병렬 시 머지 충돌 가능 → plan-run에서 **직렬화**(한 계획 머지 후 다른 계획 착수)하거나 한 Phase로 통합. 상단 SSOT 주석은 양쪽이 모두 갱신하므로 특히 주의.

### 고아 노드 처리 (a 경로 한정)

(a) 제거 시 kibana는 `es→kibana` 하나로만 연결 → 엣지 제거로 자동 고아화·숨김(기존 동작). **es는 다른 dependency 엣지(형제 계획 신규 `es→airflow`)로 연결 유지**되므로 es 고아화 0. 단 형제 계획 미적용 상태에서 본 계획 (a)만 단독 적용 시 es의 dependency 연결이 줄어드는지 실측 확인.

### Node 정적/e2e 게이트 위치 (Z-post)

`frontend/package.json` 의존(check/build/vitest/playwright)은 gitignored `node_modules` 필요 → 워크트리 신뢰 불가 → **머지 후 원본 main(Z-post)** 실행. DB·마이그레이션 무관(순수 mock 데이터).
