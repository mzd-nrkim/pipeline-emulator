# 노드 설정 저장(setNodeConfig) 배선

> 상태: 통테통과-완료

파이프라인 노드 상세 모달의 설정 폼(`frontend/src/routes/[mode=mode]/pipeline/+page.svelte:246-299`)은 `localConfig`에 바인딩된 text/number/select/boolean 입력을 제공하고, `applyMode`가 `readonly`가 아닌 필드(`runtime`/`restart`/`code`)는 실제로 편집 가능하다(`isDisabled = applyMode === 'readonly'`, line 252). 그러나 **편집값을 서버로 보내는 "적용/저장" 버튼이 없고 `setNodeConfig()` 호출부가 코드 전체에 0건**이라 모든 편집이 유실된다. 어댑터 `setNodeConfig`(mock/real)와 백엔드 `POST /nodes/{id}/config`(변수·is_paused 처리)는 이미 구현돼 있고, `ToolCanvasView`는 `setNodeConfig` prop 타입을 선언하지만 파이프라인 페이지가 이를 전달하지 않는다 — 배선만 누락된 상태다.

> 스코프 경계: "재실행" 버튼(별도 계획서 완료)·SSE 실시간 진행(별도 계획서)과 분리. 본 계획은 설정 저장 배선에 집중한다.

---

## 목표

- 노드 설정 모달에 "적용" 버튼이 생겨, 편집한 `localConfig`가 `currentAdapter.setNodeConfig(nodeId, config)`로 서버에 전송된다.
- 저장 성공 시 사용자에게 성공 피드백이 노출되고, 실패 시 오류가 노출된다(무음 실패 금지).
- `applyMode` 시맨틱이 UI 동작과 일치한다: **`runtime`만 실제 저장 대상**(백엔드가 변수/변경 수락), `restart`·`code`는 백엔드가 readonly로 취급하므로 편집 자체를 막거나 저장 대상에서 제외한다 — 현재 "편집 가능한데 저장 안 됨" 모순을 해소한다.
- `ToolCanvasView`의 `setNodeConfig` prop 미전달 상태를 정리한다(캔버스 경유 저장이 필요 없으면 prop 제거, 필요하면 전달).

## 접근 방법

1. **저장 핸들러 신설**: 페이지에 `handleApplyConfig()`를 추가한다. `selectedNode`와 편집된 `localConfig`에서 **`applyMode === 'runtime'`인 필드만 추려** payload를 구성하고 `currentAdapter.setNodeConfig(selectedNode.id, payload)`를 호출한다. 성공/실패 상태(`configSaved`/`configError`)를 신설해 기존 `triggeredRunId`/`triggerError` 패턴과 대칭으로 처리한다.
2. **UI 배선**: 설정 폼 하단(조작 영역, line 302 부근)에 "적용" 버튼을 추가하고 `onclick={handleApplyConfig}` 배선. 저장 가능한 `runtime` 필드가 하나도 없으면 버튼을 숨기거나 비활성화한다.
3. **applyMode 모순 해소**: `restart`·`code` 필드의 `isDisabled` 판정을 `applyMode !== 'runtime'`로 좁혀(또는 저장 payload에서 제외) "편집되지만 저장 안 됨"을 제거한다. 배지 문구(`🔵 코드수정`/`🟡 재기동필요`)는 유지하되 입력은 읽기전용으로 통일. — 정확한 정책(입력 비활성 vs 활성+저장제외)은 plan-review에서 확정.
4. **캔버스 prop 정리**: `ToolCanvasView`가 `setNodeConfig`를 실제로 쓰지 않으면 prop 타입 선언(line 13)을 제거해 죽은 계약을 없앤다. 저장을 캔버스 경유로 할 계획이 없으면 모달-only 경로로 확정.
5. **mock/real 대칭 확인**: mock `setNodeConfig`는 noop이지만 성공 반환하므로 mock 모드에서도 "적용" 성공 피드백이 뜨는지 확인. real은 백엔드 응답의 `detail` 오류를 그대로 노출.

## 실행 시 필수 고려사항

- **동일 파일 편집 충돌 (병렬 금지)**: 본 계획은 `frontend/src/routes/[mode=mode]/pipeline/+page.svelte`를 편집한다. 같은 세트의 다른 계획서(재실행 버튼·SSE 구독·executions 뷰)도 **같은 파일**을 편집하므로, 이 계획들은 **병렬 워크트리로 동시에 돌리면 머지 충돌**한다. 순차 실행+순차 머지(한 계획 머지 후 다음 계획 rebase)로 진행한다.
- **정책 미선택 근거 (A-1)**: `restart`·`code` 필드를 (1)입력 비활성 통일 vs (2)입력 활성+저장 payload 제외 중 택일. 실물(폼 렌더 코드·배지 문구)을 확인한 뒤 결정한다 — 권장 기본값으로 자율 처리 금지. 사용자 편집 기대치(값을 눈으로 확인만 하고 싶은지)에 따라 갈린다.
- **테스트 하네스 전제**: e2e는 `--mode test`로 dev 서버를 자체 기동해 real 백엔드 헬스체크가 항상 실패 → **mock 모드**로 동작한다. mock `setNodeConfig`는 noop이지만 성공을 반환하므로, mock e2e는 "적용→성공 피드백" 경로만 검증 가능하고 실제 서버 반영은 검증 대상이 아니다(그건 real 수동 확인 몫).
- **회귀 범위**: `configSaved`/`configError` 상태를 `triggeredRunId`/`triggerError`와 대칭으로 추가하므로, 모달 열림·노드 전환 시 두 상태 리셋 로직이 기존 트리거 상태 리셋과 충돌하지 않는지 확인한다.

## Verification

- **UI 수동 확인**: mock 모드에서 `runtime` 필드 값을 바꾸고 "적용"을 누르면 성공 피드백이 뜬다. `readonly`/`restart`/`code` 필드는 편집되지 않거나 저장에 포함되지 않는다.
- **real 모드**: 백엔드 기동 시 "적용" → `POST /nodes/{id}/config` 200, 잘못된 요청 시 백엔드 `detail` 오류가 UI에 노출된다.
- **정적 검증**: `npm run check`(svelte-check) 통과 — `setNodeConfig` 반환 타입(`void | Record`) 정합.
- **모순 해소 확인**: 코드상 "편집 가능하지만 저장 경로 없는" 필드가 0이 된다.
- **회귀 없음**: 트리거 버튼·재실행 버튼 등 기존 인터랙션 정상.
