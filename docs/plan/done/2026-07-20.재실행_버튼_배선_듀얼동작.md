# 파이프라인 "재실행" 버튼 배선 — 듀얼 동작

> 상태: 통테통과-완료

파이프라인 화면 상단 실행 조작 패널의 "재실행" 버튼(`frontend/src/routes/[mode=mode]/pipeline/+page.svelte:121-127`)은 `964c8ba` 커밋에서 정적 UI 목업으로 들어온 뒤 **한 번도 `onclick`이 배선된 적이 없어** 눌러도 아무 일도 일어나지 않는다. 하위 계층(어댑터 `triggerNode`(mock/real), 백엔드 `POST /nodes/{id}/trigger`)은 이미 구현돼 있고, 노드 상세 모달의 "트리거" 버튼은 `handleTrigger()`로 정상 동작한다 — 상단 버튼의 UI 배선만 누락된 상태다. 이 계획은 상단 버튼을 **노드 선택 상태에 따라 라벨·동작이 바뀌는 듀얼 동작**으로 배선한다.

> 스코프 경계: SSE 실시간 진행 표시(`subscribePipelineStatus` 미구독, #3)와 노드 설정 저장 폼(`setNodeConfig` 미호출, #2)은 **별도 계획서**로 분리한다. 본 계획은 재실행 버튼 배선에만 집중한다.

---

## 목표

- 상단 "재실행" 버튼에 `onclick` 핸들러가 배선되어 클릭 시 실제 트리거가 발생한다.
- 노드가 선택된 상태(`selectedNode != null`)에서는 버튼이 **선택 노드 재트리거**로 동작하고, 라벨이 선택 맥락을 반영한다(예: "노드 재실행").
- 노드 미선택 상태(`selectedNode == null`)에서는 버튼이 **파이프라인 전체 재실행**(첫 단계 `bronze_raw`→`bronze_0_registration`부터)으로 동작하고, 라벨은 "재실행"을 유지한다.
- 어느 경로든 트리거 성공 시 반환된 `dag_run_id`가 `activeRunId`에 반영되어 상단 "활성 실행" 패널이 갱신된다.
- 트리거 실패 시 사용자에게 오류가 노출된다(무음 실패 금지).

## 접근 방법

1. **전체 재실행 대상 결정 — 신규 백엔드 엔드포인트 불필요**: `STAGE_DAG_MAP`에서 첫 단계 노드 `bronze_raw`가 `bronze_0_registration` DAG에 매핑돼 있다. 전체 파이프라인 재실행은 **기존 `triggerNode('bronze_raw', {})` 재사용**으로 구현한다(Airflow 하류 DAG는 첫 단계 트리거로 연쇄). 어댑터 신규 함수·백엔드 엔드포인트 추가를 피해 표면적을 최소화한다.
   - 가독성을 위해 어댑터에 얇은 래퍼 `triggerPipeline()`를 두는 안을 검토한다(내부적으로 첫 단계 노드를 트리거). mock/real 양쪽에 대칭 구현. — plan-review에서 "래퍼 신설 vs 페이지에서 직접 `triggerNode('bronze_raw')` 호출" 중 택일.
2. **듀얼 동작 핸들러**: 페이지에 `handleRerun()`을 신설한다. `selectedNode`가 있으면 기존 `handleTrigger()` 경로(선택 노드 트리거)를, 없으면 전체 파이프라인 트리거 경로를 타도록 분기한다. 두 경로 모두 성공 시 `activeRunId`를 갱신하고 실패 시 `triggerError`를 세팅한다. 기존 `handleTrigger()`의 상태 관리(`triggeredRunId`/`triggerError` 리셋) 패턴을 재사용해 중복을 최소화한다.
3. **버튼 라벨 반응성**: 버튼 라벨을 `selectedNode ? '노드 재실행' : '재실행'`처럼 파생 상태로 바인딩한다. 접근성 위해 `aria-label`도 동기화. (정확한 문구는 plan-review에서 확정.)
4. **상태 일관성**: 전체 재실행 트리거의 반환 `dag_run_id`도 노드 트리거와 동일하게 `activeRunId`에 반영한다. 모달이 닫힌 상태(미선택)에서도 상단 패널 "활성 실행"이 갱신되는지 확인한다.
5. **회귀 방지**: 모달 내 "트리거" 버튼(`handleTrigger`)의 기존 동작은 변경하지 않는다 — 상단 버튼만 신규 핸들러를 사용한다.

## 실행 시 필수 고려사항

### 미선택 결정 근거 — 래퍼 미신설, 페이지에서 직접 호출 (결정)

`triggerPipeline()` 래퍼 신설안은 **탈락**한다. 근거: (1) 계획서가 "표면적 최소화"를 명시하는데 래퍼는 mock·real 양쪽 대칭 구현 부담만 추가한다. (2) 전체 재실행의 의미는 "첫 단계 노드(`bronze_raw`) 트리거"와 동일해 별도 함수로 추상화할 실익이 없다. → `handleRerun()` 내부에서 `currentAdapter.triggerNode('bronze_raw', {})`를 **직접 호출**한다. (실물 확인: `mock-adapter.ts:31`·`real-adapter.ts:66` 모두 `triggerNode(nodeId, conf): Promise<{ dag_run_id: string }>` 시그니처 존재 — 첫 노드 재사용 가능.)

### 회귀 범위 — 모달 트리거 경로 불변

모달 내 "트리거" 버튼은 `handleTrigger()`(`+page.svelte:41-52`)를 계속 사용한다. 본 계획은 **상단 버튼에만** `handleRerun()`을 배선한다. `handleTrigger()` 코드는 수정하지 않으므로 모달 트리거·`triggeredRunId` 표시 경로에 회귀가 없다. `activeRunId`(`:20`) 갱신 규약은 두 핸들러가 공유한다.

### 테스트 하네스 전제 — e2e는 mock 모드

e2e는 `/sample/pipeline`(mock 모드) 경로에서 실행한다(기존 스펙 `node-detail-modal.spec.ts:5` 패턴). mock `triggerNode` 반환은 `mock-run-${nodeId}-${Date.now()}`(`mock-adapter.ts:32`)라 실제 Airflow·DB 불요 → **teardown 불필요**(휘발 상태만 생성, 영속 side-effect 없음). real 모드 재실행 e2e는 Airflow 기동 전제라 본 계획 범위 밖.

### 실행 순서 — 단일 파일 집중, 직렬 1-Phase

핵심 편집은 `+page.svelte` 단일 파일(핸들러 신설 + 버튼 배선 + 라벨 파생)에 집중된다. 동일 파일이므로 **한 Phase(한 에이전트)로 묶는다**. e2e 스펙은 별도 신규 파일이라 편집 충돌은 없으나 규모가 작아 같은 Phase에 포함해도 무방.

## Verification

- **UI 수동 확인**: mock 모드에서 노드 미선택 상태로 "재실행" 클릭 시 상단 "활성 실행" `RUN_ID`가 `—`에서 `mock-run-bronze_raw-*`로 갱신된다. 노드를 선택하면 버튼 라벨이 "노드 재실행"으로 바뀌고, 클릭 시 해당 노드 run_id로 갱신된다.
- **회귀 없음**: 모달 "트리거" 버튼, 뷰 토글, 연결없는노드 토글 등 기존 인터랙션 정상.

### TC 커버리지 (Right-BICEP · CORRECT)

- **Right(정상 기대)**: 미선택 재실행 → `triggerNode('bronze_raw')` 호출·`activeRunId` 갱신; 선택 재실행 → 선택 노드 트리거.
- **B(경계)**: `selectedNode` null ↔ non-null 경계에서 라벨·동작 전환 정확. 모달 열림/닫힘 상태 모두에서 상단 버튼 동작.
- **I(역/부정)**: `handleTrigger()`(모달) 경로 코드 불변 부정 단언; 상단 버튼 클릭이 모달을 열지 않음.
- **C(교차검증)**: 반환 `dag_run_id` ↔ 상단 `RUN_ID` 표기 1:1. 선택 노드 id ↔ 트리거 대상 id 일치.
- **E(에러)**: 트리거 실패(어댑터 reject) 시 `triggerError` 세팅·무음 실패 금지 단언.
- **Cardinality**: 클릭 1회 = 트리거 1회(중복 발사 없음).
- **Ordering**: 해당 없음(단발 액션, 순서 의존 없음).
- **e2e**: (1) 미선택 재실행 → `activeRunId` 표기 갱신, (2) 노드 선택 시 라벨 "노드 재실행" 변경 + 재트리거 — mock 모드 `/sample/pipeline`. **DB 통합 테스트**: 해당 없음(mock 모드, 실 DB 미접촉).
