# 그래프 엣지 handle LR(좌→우) 변경

> 상태: 통테통과-완료

n8n처럼 좌→우 흐름의 그래프를 목표로 하지만 현재 `@xyflow/svelte`의 `default` 노드 타입이
handle을 노드 **상단(target)·하단(source)**에 배치하여, 엣지가 노드 좌우가 아닌 상하에 닿는
문제가 있다. 노드 좌표는 이미 좌→우(depth·X축)로 배치되어 있어, handle 위치만 바꾸면 된다.

---

## 목표

- 모든 그래프 뷰(Data·Infra·Pipeline)에서 엣지가 노드 **왼쪽(입력)·오른쪽(출력)**에 연결된다
- n8n과 동일한 좌→우 흐름 시각화가 완성된다

## 접근 방법

1. **커스텀 노드 컴포넌트 생성** (`LRFlowNode.svelte`):
   - `Handle type="target" position={Position.Left}` (입력 handle — 좌)
   - `Handle type="source" position={Position.Right}` (출력 handle — 우)
   - `data.label` 표시 — `default` 노드 동작을 그대로 유지하면서 handle만 교체

2. **커스텀 노드 타입 등록**:
   - `ToolCanvasView.svelte`: `nodeTypes={{ lrnode: LRFlowNode }}` 추가
   - `PipelineGraphView.svelte`: 동일하게 `nodeTypes` 등록

3. **노드 타입 변경**:
   - `buildNodesAndEdges.ts` line 171: `type: 'default'` → `type: 'lrnode'`
   - `PipelineGraphView.svelte` 내부 `buildNodesAndEdges` line 37: 동일하게 변경

4. 엣지는 별도 수정 없음 — `source`/`target` ID만 있으면 커스텀 handle을 자동으로 사용함

## 실행 시 필수 고려사항

### 동일 파일 편집 충돌

`PipelineGraphView.svelte`를 수정하는 두 항목(① nodeTypes prop 추가 + LRFlowNode import, ② 내부 `type: 'default'` → `type: 'lrnode'` 변경)은 반드시 **Phase C 단일 에이전트가 순차 처리**해야 한다. plan-run이 별개 Phase로 분리하면 동시 편집 충돌 발생.

### Phase 의존성

- Phase C·D는 Phase A(`LRFlowNode.svelte` 파일 생성)가 완료된 후 실행해야 한다 (import 대상이 없으면 `npm run check` 실패).
- Phase B(`buildNodesAndEdges.ts` 변경)는 Phase A와 병렬 가능하나, Phase A·B 모두 완료되어야 뷰가 정상 렌더링됨.

### LRFlowNode 스타일 전달 방식

`@xyflow/svelte` 0.1.39는 Node 객체의 `style` 필드를 **wrapper div**에 적용하며 커스텀 컴포넌트에 전달하지 않는다. 따라서 `LRFlowNode.svelte` 내부에 별도 스타일을 정의하지 않아도 기존 `getNodeStyle()` 결과가 정상 적용된다.

### Node 프로젝트 정적 검사 위치

`frontend/package.json` 존재 → `npm run check`/build는 Z-pre(워크트리)에서 실행하지 않고 **머지 후 원본 main**에서 실행한다.
