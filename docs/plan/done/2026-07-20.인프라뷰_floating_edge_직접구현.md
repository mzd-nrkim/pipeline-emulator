# 인프라 뷰 Floating Edge 직접 구현

> 상태: 통테통과-완료

인프라 관계도(`view==='infra'`)에서 엣지가 다른 노드를 가로지르는 문제가 있다. 원인은 `ToolFlowNode.svelte`의 핸들이 Left/Right 고정이라 Airflow 같은 허브 노드에 연결되는 bezier 곡선이 다른 노드 위를 통과하기 때문이다. 커스텀 floating edge를 구현해 두 노드 중심점 간 방향벡터와 bbox 교점을 계산하여 노드 경계에서 시작/종료하는 직선 엣지로 교체한다.

---

## 목표

- 인프라 뷰 엣지가 노드 bbox 경계에서 출발·도착하여 **다른 노드를 가로지르지 않는다**.
- 데이터 뷰(`view==='data'`)의 L/R bezier 엣지는 **회귀 없이 그대로** 동작한다.
- force 파라미터 조정으로 노드 간격이 확보되어 **교점 계산의 여유 공간이 늘어난다**.

## 접근 방법

1. **실측(I-1)**: `@xyflow/svelte@1.6.2`의 `useSvelteFlow().getNode(id)` API 가용성 확인. 커스텀 엣지 컴포넌트에서 소스/타겟 노드의 `position`·`width`·`height`에 접근 가능한지 실측. 불가 시 대안(EdgeProps의 `internalsSymbol` 또는 store 직접 접근) 확정.

2. **`InfraFloatingEdge.svelte` 신규** — 커스텀 엣지 컴포넌트:
   - `useSvelteFlow().getNode(source/target)`으로 노드 center 좌표 산출
   - 두 center 간 방향벡터 → 각 노드 bbox(`width=80, height=80`) 경계 교점 계산 (rect-line intersection)
   - 교점 간 SVG `<path>` 직선(`M x1 y1 L x2 y2`)으로 렌더
   - `markerEnd` 화살표(xyflow prop 재사용), stroke dashed 스타일(dependency 표현)
   - 노드 중심이 겹치는 엣지 방어(교점 = center fallback)

3. **`buildNodesAndEdges.ts` 수정**: infra 뷰 엣지에 `type: 'infra-floating'` 추가. data 뷰 엣지는 무변경.

4. **`ToolCanvasView.svelte` 수정**: `edgeTypes={{ 'infra-floating': InfraFloatingEdge }}` 등록. infra/data 뷰 조건 무관하게 edgeTypes를 상시 등록해도 무해(data 뷰 엣지에 해당 type이 없으면 기본 bezier 사용).

5. **`forceLayout.ts` 파라미터 튜닝**: `charge -300 → -600`, `linkDistance 150 → 220`, `center (400,300) → (500,350)`. 노드 간격 확대로 교점 경로의 여유 공간 확보.

6. **`ToolFlowNode.svelte` 핸들 처리**: floating edge는 xyflow 핸들을 사용하지 않으므로, `isInfra=true`일 때 L/R 핸들을 `visibility:hidden`으로 숨기되 DOM에는 유지(e2e 핸들 카운트 테스트 회귀 방어). 단, I-1 실측 결과에 따라 핸들 완전 제거도 가능.

## 실행 시 필수 고려사항

① **회귀 이유·범위 (data 뷰 표면 0)**: `buildNodesAndEdges.ts` edges map(178-201)은 `view==='infra'`일 때만 `type:'infra-floating'`을 부여한다. data 뷰 엣지는 `type` 미부여 → xyflow 기본 bezier 사용 → 회귀 표면 0. edgeTypes 상시 등록도 무해(해당 type 엣지가 없으면 미사용).

② **핸들 카운트 e2e 회귀 (H-1 DOM 유지 필수)**: `e2e/graph-lr-handle.spec.ts`의 Infra 뷰 테스트(22-28행)가 `.svelte-flow__handle-left`·`handle-right` count > 0, `top`·`bottom` count == 0을 단언한다. H-1은 핸들을 `visibility:hidden`으로 **숨기되 DOM/클래스는 유지**해야 이 카운트가 보존된다. 핸들 완전 제거(display:none·조건부 미렌더)는 이 e2e를 깨뜨리므로 금지 — I-1 실측이 완전 제거를 시사해도 e2e 회귀 방어가 우선한다. 또한 floating edge는 source/target을 노드 id로 연결하나 xyflow는 연결 등록에 핸들 DOM을 요구하므로 핸들 유지가 안전하다.

③ **`isInfra` 접근 경로**: `ToolFlowNode.svelte`는 `let { data, selected }: NodeProps = $props()`로 받는다. 최상위 `isInfra` prop이 아니라 **`data.isInfra`**로 접근한다(buildNodesAndEdges.ts:167이 노드 data에 주입). H-1 구현 시 이 경로를 준수.

④ **실행 순서·병렬 가능 여부**:
   - **I-1(실측)은 선행** — F-1이 I-1의 getNode 가용성 결론에 의존한다. I-1 완료 후 F-1 착수.
   - **동일 파일 묶음**: B-1·B-2는 같은 파일(`buildNodesAndEdges.ts`) → 한 에이전트. C-1·C-2는 같은 파일(`ToolCanvasView.svelte`) → 한 에이전트.
   - **독립 병렬 가능**: F-1(신규 파일), P-1(`forceLayout.ts`), H-1(`ToolFlowNode.svelte`)은 서로·B/C와 파일이 겹치지 않아 병렬 가능(단 F-1은 I-1 이후).

⑤ **미선택 결정 근거 (I-1 대안)**: 노드 크기 접근은 `useSvelteFlow().getNode(id)` 우선. 반환값에 `width`/`height`/`position`이 없으면 대안으로 EdgeProps의 `internalsSymbol` 또는 svelte store 직접 접근을 사용. getNode를 1순위로 두는 이유는 공개 API로 버전 안정성이 높기 때문(internalsSymbol은 내부 구현 의존).
