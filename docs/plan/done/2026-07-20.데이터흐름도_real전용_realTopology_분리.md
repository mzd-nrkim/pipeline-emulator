# 데이터흐름도 real 모드 전용 realTopology 분리 — sample=이상 / real=실측

> 상태: 미시작

`/real/pipeline` 데이터흐름도의 노드 config가 실제 인프라(docker-compose·DAG·mock-api·pii_engine)와 다수 어긋난다(사실오류: 임베딩 차원 768→실측 384, index `lllm-docs`→실측 `pdis_cft`, `ml-node-1` 허위, executor CeleryExecutor→실측 SequentialExecutor 등). 원인은 **real 모드도 `mockTopology`를 그대로 반환**하기 때문이다(`real-adapter.ts:61-64`). 이 mockTopology는 목표 프로덕션 아키텍처(hyundaimotor-lllm: KURE·docling·lllm 네이밍)를 모사한 것이라 로컬 emulator 실구현과 별개다.

방향(사용자 확정): **sample(mock) 모드 = 이상 아키텍처(현행 mockTopology 유지), real 모드 = 실측(docker-compose/DAG 실제값)으로 분리**. 어댑터 분기(`+layout.ts:6`)는 이미 모드별로 나뉘어 있어 real-adapter만 별도 `realTopology`를 반환하면 된다.

## 목표

- real 모드(`/real/pipeline`) 데이터흐름도가 **신규 `realTopology`**를 사용해 실측값(index=pdis_cft, dim=384, MiniLM, SequentialExecutor, DB=pipeline_emulator 등)을 표시한다.
- sample 모드(`/sample/pipeline`)는 **현행 `mockTopology`(이상 아키텍처: KURE·docling·lllm·dam)를 무변경**으로 유지한다.
- real 모드에서 실제 ES 색인 DAG(gold_6_es_indexing)가 **신규 `node-es-search` 노드**로 트리거 가능해진다(staging=node-es→gold_5, indexing=node-es-search→gold_6 분리, 결정 A 확정).
- 실측 일치 노드(debezium·seaweedfs·zookeeper·valkey·mock-api)와 렌더 계약(toolCatalog·buildNodesAndEdges·ToolFlowNode)은 **불변**이다.

> **분리 축은 뷰가 아니라 모드다.** 데이터흐름 뷰·인프라 뷰는 별도 토폴로지가 아니라 **하나의 `CanvasTopology`를 채널(`data`/`dependency`)로 필터**한 두 뷰다(`buildNodesAndEdges.ts:45-51` — `e.channels.includes(view==='data'?'data':'dependency')`; plan-review 실측으로 라인 교정, 기존 표기 116-121은 파일 진화로 무효). 따라서 real-adapter가 `realTopology`를 반환하면 real 모드의 **데이터흐름 뷰·인프라 뷰가 동시에** realTopology를 쓴다. 이 계획의 교정 대상은 **노드 config 값 + data 채널 엣지 + 가상 노드**이며, **dependency 채널 엣지(인프라 뷰 배선)는 mockTopology에서 이미 docker-compose depends_on/env 기반 실측 반영**(`topology.ts:276-294`)이라 realTopology에서도 **원칙적으로 동치 유지**한다(dam 제거 등 노드 삭제에 따른 엣지 정리만 반영). 인프라 뷰를 별도로 손대지 않는다.

## 접근 방법

1. **`realTopology.ts` 신규 작성** (`frontend/src/lib/mock/realTopology.ts` 또는 `$lib/api/realTopology.ts`): `mockTopology`를 복제한 뒤 위 실측 표대로 노드 config를 교정한다. 타입은 기존 `CanvasTopology` 재사용(스키마 변경 없음).
2. **real-adapter 배선 교체** (`real-adapter.ts:59-64`): import를 `mockTopology`→`realTopology`로 바꾸고 `fetchCanvasTopology()`가 이를 반환. mock-adapter는 손대지 않는다 → sample 회귀 0.
3. **사실오류 교정(값)**: 임베딩(model/dim/위치), ES(index/mlNode/색인 DAG), executor, PII(recognizer/nlpEngine), DB명 등을 realTopology 노드 config에만 반영.
4. **구조 갭 해소(node-es-search 추가, 결정 A 확정)**: `ui-backend`에 이미 존재하는 `node-es-search→gold_6_es_indexing` 매핑(`airflow.py:37`)에 대응하는 **신규 노드 `node-es-search`를 realTopology에 추가**한다. node-es(gold_5, staging)는 유지하고, data 채널 엣지를 `node-valkey→node-es→node-es-search`(또는 병렬)로 이어 staging·indexing을 분리 표현한다. node-es의 dagId 재지정 안은 채택하지 않는다(staging 트리거 경로 보존).
5. **가상/profile-gated 노드 처리(deployStatus 표기, 결정 B 확정)**: DAM(가상)은 `deployStatus:'planned'`, profile-gated(nifi·debezium·valkey)는 `deployStatus:'absent'`(이미 존재하는 필드, `types.ts:110`; 렌더 grayout은 기존 배지 로직 재사용)로 표기한다. 노드 제거 안은 채택하지 않는다(인프라 존재 가능성 표현 유지).
