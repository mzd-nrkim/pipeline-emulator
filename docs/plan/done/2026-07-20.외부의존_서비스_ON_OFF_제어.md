# 외부 의존 서비스 ON/OFF 제어 — 컨테이너 start/stop 컨트롤 플레인

> 상태: 통테통과-완료
> 성격 변경 결정 — 본 계획은 read-only 시각화 도구를 **부분적 컨트롤 플레인**으로 확장한다(사용자 옵션 b 선택, 2026-07-20).

현재 pipeline-emulator 프론트는 **read-only 시각화**(CLAUDE.md·다수 계획서가 반복 명시)로, 노드는 조회·트리거(Airflow DAG)만 가능하고 인프라 컨테이너 자체를 켜고 끌 수는 없다. 본 계획은 사용자 요청(외부 의존 서비스 상태관리, ON/OFF 포함)에 따라 **각 노드에서 대응 docker 컨테이너를 start/stop**하는 제어 기능을 추가한다. 이는 도구 정체성을 바꾸는 결정이므로, 보안·안전장치를 1급 요구사항으로 다룬다.

> **형제 계획 의존**: `노드_런타임_헬스_신호등_축.md`. 본 계획은 그 계획의 **node→compose service 매핑**과 **runtimeHealth 신호등**을 재사용한다(stop 후 신호등이 즉시 down으로 반영되어야 제어 피드백이 성립). 헬스 계획 선행 또는 동시 진행 권장.

---

## 성격 변경 명문화: read-only → 제한적 read-write

| 축 | 기존 | 본 계획 |
|----|------|---------|
| 노드 트리거 | Airflow DAG trigger (`/nodes/{id}/trigger`) | 유지 |
| 노드 config | runtime 키만 변경(`is_paused`·`variable`) | 유지 |
| **컨테이너 수명주기** | **불가(read-only)** | **start/stop 추가(신규 read-write)** |

docker 컨테이너 제어는 **호스트 docker 데몬 접근**을 요구한다(사실상 root 권한). 따라서 본 계획은 **로컬 개발/데모 환경 전용**이며, 공유·프로덕션 배포 환경에서는 기능 자체를 비활성(flag off)하는 것을 전제로 한다.

## 배경: 실측 근거

| 항목 | 값 | 근거 |
|------|----|----|
| 백엔드 | ui-backend(FastAPI :8001), 노드 엔드포인트 `/nodes/{id}/trigger`·`/config` 존재 | `ui-backend/app/api/nodes.py` |
| docker 접근 | ui-backend에 `docker.sock` **미마운트** — 현재 컨테이너 제어 불가 | `docker-compose.yml:114` |
| 제어 대상 서비스 | seaweedfs·mysql·airflow·mock-api·elasticsearch·valkey·debezium·zookeeper·nifi | `docker-compose.yml` services |
| 자기 참조 위험 | ui-backend 자신도 compose 서비스 → 자기 stop 방지 필요 | ui-backend 서비스 |
| 의존 연쇄 | mysql stop → airflow·debezium 등 다운스트림 영향 | compose `depends_on` |
| node→service 매핑 | 형제 헬스 계획과 공유 SSOT | `노드_런타임_헬스_신호등_축.md` |

## 목표

- ui-backend가 docker 데몬에 접근해 **지정 서비스 컨테이너를 start/stop**하는 엔드포인트(`POST /services/{service}/power` 등)를 제공한다.
- 프론트 노드에서 ON/OFF 제어(토글/컨텍스트 메뉴 + 확인 다이얼로그)로 대응 컨테이너를 켜고 끌 수 있다.
- **안전장치**가 1급으로 구현된다: 제어 허용 서비스 화이트리스트, ui-backend 자기 stop 차단, 확인 절차, 감사 로그, 기능 flag(로컬 전용).
- stop/start 결과가 형제 계획의 **runtimeHealth 신호등에 즉시 반영**되어 제어 피드백 루프가 성립한다.
- 기능 flag off(기본값) 시 프론트·백엔드가 **기존 read-only 동작으로 완전 회귀**한다(회귀 0).

## 접근 방법

1. **결정(성격 변경 승인)**: read-only → 제한적 read-write 확장을 사용자 확정(옵션 b 이미 선택). 기능 flag(`ENABLE_SERVICE_CONTROL`, 기본 off) 설계로 위험 격리.

2. **docker 접근 인프라** (`docker-compose.yml` ui-backend):
   - ui-backend 서비스에 `/var/run/docker.sock:/var/run/docker.sock` 볼륨 마운트(로컬 전용).
   - Python: `docker` SDK(`docker` PyPI) 또는 Engine API 직접 호출. requirements 추가.
   - **보안 경고 명문화**: sock 마운트 = 컨테이너가 호스트 docker 완전 제어 = 사실상 root. 공유 환경 금지.

3. **제어 엔드포인트** (`ui-backend/app/api/services.py` 신규):
   - `POST /services/{service}/power` body `{action: 'start'|'stop'|'restart'}` (또는 `/start`·`/stop` 분리).
   - **화이트리스트**: 제어 허용 서비스 집합 상수. 목록 외·`ui-backend` 자신·핵심 인프라(선택)는 400/403.
   - `GET /services` — 제어 가능 서비스 + 현재 상태(형제 헬스 엔드포인트와 통합 검토).
   - 감사 로그(누가·언제·무엇 stop) 기록.

4. **프론트 제어 UI**:
   - 노드에 ON/OFF 토글 또는 우클릭 컨텍스트 메뉴. 매핑 없는 노드는 제어 UI 미표시(형제 매핑 재사용).
   - stop 등 파괴적 동작 전 **확인 다이얼로그**(의존 연쇄 경고 문구 포함 — "mysql 중지 시 airflow·debezium 영향").
   - `real-adapter.ts`에 `setServicePower(service, action)` 추가.

5. **피드백 루프**: 제어 후 형제 계획의 헬스 폴링이 신호등을 갱신(stop→🔴). 즉시성 위해 제어 응답에 낙관적 상태 반영 + 다음 폴링 확정.

## 실행 시 필수 고려사항

### 보안 = 1급 요구사항 (docker.sock 마운트)

`docker.sock` 마운트는 **컨테이너 탈출·호스트 장악 벡터**다. 필수 방어:
- 기능 flag `ENABLE_SERVICE_CONTROL` 기본 off — 명시적 opt-in 없이는 엔드포인트 404/비활성.
- 제어 화이트리스트 — 임의 컨테이너·임의 docker 명령 불가, 미리 정의된 서비스명 집합만.
- ui-backend **자기 자신 stop 차단**(자기 참조 → 제어 불능 상태 유발).
- 공유/프로덕션 환경 배포 금지 문서화. 로컬 데모 전용 명문화.

### 의존 연쇄 (depends_on)

mysql·valkey·zookeeper 같은 하위 인프라를 stop하면 상위(airflow·debezium·nifi)가 연쇄 실패한다. 확인 다이얼로그에 영향 서비스 경고. 무분별한 stop이 에뮬레이터 전체를 마비시키지 않게 화이트리스트·경고로 방어.

### 형제 헬스 계획과의 통합 (매핑·신호등 공유)

node→service 매핑과 상태 표시는 형제 `노드_런타임_헬스_신호등_축.md`가 SSOT다. 본 계획은 그것을 **재사용**하며 중복 정의하지 않는다. docker 접근 인프라(sock 마운트)는 본 계획이 도입하고 헬스 계획이 공유 — **본 계획의 sock 마운트가 헬스 계획 (b) 경로의 선행**이 될 수 있다. plan-run 시 두 계획 배치 순서 동기화.

**동일 파일 편집 충돌 — 병렬 금지**: #1과 #2는 다음 파일을 공동 편집한다 — `ui-backend/app/main.py`(라우터 등록), `frontend/src/lib/components/ToolFlowNode.svelte`(신호등 dot vs 제어 UI), `frontend/src/lib/api/real-adapter.ts`·`mock-adapter.ts`(fetchServiceHealth vs setServicePower), `ui-backend/app/services/`(매핑 SSOT). 따라서 두 계획은 **병렬 워크트리로 나누지 말고 직렬(동일 브랜치 순차) 또는 단일 워크트리에서 함께** 진행한다. 권장 순서: **#1(헬스+매핑 SSOT+sock 소비) → #2(sock 마운트 도입은 I-1에서, 제어 엔드포인트·UI)**. 단 sock 마운트(I-1)는 #1의 (b) 경로 선행이므로, 실무상 I-1을 두 계획 착수 직전 공유 커밋으로 먼저 반영하는 편이 안전하다.

### 기능 flag off = 완전 회귀

flag off 시 제어 엔드포인트 비활성 + 프론트 제어 UI 미표시 → 도구가 기존 read-only로 완전 복귀. 기본값 off로 "실수로 프로덕션에 제어 노출" 방지.

### 게이트 위치 (Z-post)

docker 실제 start/stop은 live docker 데몬 필요 → 워크트리 불가 → **Z-post(원본 main, docker-compose 기동 상태)**. 화이트리스트·자기참조 차단 등 순수 로직은 pre-gate 단위테스트.

- 기능 flag off 시 도구가 기존 read-only로 완전 회귀한다(회귀 0).
- 파괴적 stop 전 의존 연쇄 경고 다이얼로그가 표시된다.
- `npm run check`·`build`·`vitest`·`playwright` + docker 실연동이 수치 근거와 함께 통과한다.

### TC 커버리지 (Right-BICEP · CORRECT)

- **Right**: 화이트리스트 서비스 start/stop 정상, 헬스 반영.
- **B(경계)**: flag off→엔드포인트 비활성, 화이트리스트 경계(허용/거부).
- **I(역/부정)**: ui-backend 자기 stop 거부, 목록 외 서비스 거부, 임의 docker 명령 불가 부정 단언.
- **C(교차검증)**: `docker ps` 실상태 ↔ 제어 결과 ↔ 신호등 1:1.
- **E(에러)**: docker 데몬 미가용·이미 stop된 서비스 재stop → 명확한 에러 응답(500 남발 아님).
- **Security**: docker.sock 마운트 로컬 전용, flag 기본 off, 감사 로그 존재.
- **Conformance/Reference**: 기존 `/nodes` 엔드포인트·read-only 경로 계약 불변.
- **Existence/Cardinality**: 노드·엣지·기존 API 불변(신규 엔드포인트만 추가).
