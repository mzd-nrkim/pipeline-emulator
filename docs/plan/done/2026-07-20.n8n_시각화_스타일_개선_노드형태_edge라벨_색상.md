PipeScale 캔버스를 n8n에 가까운 시각 경험으로 다듬는다. 직전 아이콘 작업(`done/2026-07-20.toolflownode_브랜드로고_lucide폴백_아이콘.md` + 후속: 아이콘 44px 확대·카드 배경틴트 제거·accent 세로보더 삭제·컬러 로고 유지)의 연장선이며, **read-only 시각화 도구 성격을 유지**한다(편집 기능은 별도 idea 계획 `docs/plan/idea/pipeline-emulator-idea-n8n_인터랙션_편집_확장.md`로 분리). n8n 제안 7항 중 시각화에 해당하는 **1(노드 형태)·2(배경)·3(edge 라벨)·6(유형별 색상)** 만 담는다.

> - **노드 형태(1)**: 현재 아이콘-only 80px 정사각 카드 + 카드 **외부 하단** 라벨. n8n은 아이콘+제목이 한 둥근 사각형 안. → 실질 신규 작업.

## 목표

- 노드가 **아이콘 + 제목을 한 둥근 사각형 안에 담은 n8n형 카드**로 렌더된다(현재의 아이콘-only 정사각 + 외부 라벨 구조를 개선). 핸들 개수·위치와 모달 드릴다운은 회귀 없이 유지된다.
- 분기 edge(`airflow-branch` 등)의 조건 라벨(true/false·match/nomatch)이 **작은 배지 형태로 또렷하게** 읽힌다(현재 부분 구현의 가독성 보강).
- 노드가 **벤더색(기존)에 더해 카테고리·deployStatus에 따라 시각적으로 분별**된다(색상 하드코딩 없이 DESIGN.md oklch 토큰·`--node-accent` 체계 내에서).
- 도트 그리드 배경의 대비·간격이 캔버스 가독성에 맞게 정돈된다(이미 존재 → 튜닝).
- 모든 변경은 `ToolCanvasView`·`PipelineGraphView` 두 뷰에 일관 반영되고, **read-only 계약**(노드 드래그·편집 미도입)은 그대로다.

## 접근 방법

1. **노드 카드 레이아웃 재구성** (`ToolFlowNode.svelte`) — 현재 `.node-card`(80px 정사각, 아이콘 중앙) + `.node-label`(카드 외부 하단)을 **n8n형 단일 카드**로 재편한다. 두 후보를 R-단계에서 실측 비교:
   - (a) **가로형 카드**: 좌측 아이콘 + 우측 제목/벤더 텍스트(n8n 노드에 가장 근접).
   - (b) **세로형 카드**: 아이콘 상단 + 제목 하단을 카드 *내부*로 흡수.
   - 핸들 top 계산이 `--node-card-size` 기준이므로 카드 치수 변경 시 **핸들 위치 공식 동반 수정**(회귀 방어 핵심).
2. **edge 조건 라벨 배지화** — `buildNodesAndEdges.ts`의 `label: e.condition`은 유지하되, 커스텀 edge 라벨 렌더(또는 `labelBgStyle`/`labelStyle`)로 **작은 pill 배지**를 입힌다. true/false 등 조건값별 색(성공=green 계열, 분기=neutral)을 토큰으로 매핑.
3. **카테고리·상태 색 분별** — `toolCatalog`의 `category`와 `deployStatus`를 노드 data로 이미 흘려보내므로(`ToolFlowNode`에서 `d.category`·`deployStatus` 접근 가능), 카드 테두리/미세 배경/라벨 색을 **카테고리·상태 축으로** 분별한다. 벤더 accent(로고 색)와 **역할이 겹치지 않게** 축을 분리(로고=벤더, 형태/보조색=카테고리·상태).
4. **배경 튜닝** — `<Background>` `gap`·`size`·`patternColor` 대비를 n8n 참고 이미지에 맞춰 조정(다크모드 포함). 컴포넌트 신규 추가 없음.
5. **두 뷰 일관성·회귀 방어** — 노드/edge 렌더는 `ToolFlowNode`·공통 edge 경로가 담당하므로 두 뷰 자동 반영. e2e(`graph-lr-handle`·`node-detail-modal`·`route-split`·`infra-view`)가 핸들 개수·모달·분기를 커버 → 카드 치수·핸들 공식 변경의 회귀를 이들로 검증.

## 실행 시 필수 고려사항

- **핸들 위치 회귀**: 핸들 `top: calc(var(--node-card-size) * ...)` 공식이 카드 형태 변경의 최대 회귀 지점. 카드 가로/세로 비가 바뀌면 target(좌1)·source(우≥1) 정렬이 깨진다 — 변경과 동시에 공식·`--node-card-size`/신규 치수 변수 정리, `graph-lr-handle.spec.ts`로 실측.
- **색 축 중복 금지**: 로고=벤더색을 이미 쓰므로 카테고리·상태 색을 또 강한 색면으로 넣으면 과채도. 카테고리·상태는 **테두리 스타일·미세 톤·라벨색** 등 저채도 축으로.
- **하드코딩 색 금지**: DESIGN.md oklch 토큰·`--node-accent`만. 신규 색이 필요하면 토큰 추가 후 참조.
- **read-only 유지**: 노드 드래그 활성화·`+`버튼·편집 UI는 이 계획 범위 밖(idea 계획으로 분리). SvelteFlow `nodesDraggable` 등 편집 옵션 변경 금지.

### 두 뷰가 ToolFlowNode를 공유하되 data shape가 다름 (plan-review 실측 2026-07-20)

`ToolCanvasView`와 `PipelineGraphView` **둘 다** 노드를 `nodeTypes={{ tool: ToolFlowNode }}`로 렌더한다(PipelineGraphView.svelte:4,81 — `ToolFlowNode` import·등록 확인). 따라서:

- **카드 레이아웃(N)은 두 뷰에 자동 반영**된다(공유 컴포넌트). 접근 방법 #5의 "두 뷰 자동 반영"은 카드 마크업 한정으로 **성립**한다.
- **단, 두 뷰의 노드 data shape가 다르다**. ToolCanvasView는 `buildNodesAndEdges`가 `displayName·vendor·category·accent·deployStatus·outputs` 전부 주입하지만, **PipelineGraphView는 `{ label: stage.name, planned, animated }`만 주입**(PipelineGraphView.svelte:39)한다. 그 결과 PipelineGraphView에서 ToolFlowNode는 `displayName`→`'Unnamed'`, `vendor`/`category` 없음, `deployStatus`→`'active'` 폴백으로 렌더된다. **N-2(제목·벤더를 카드 내부로 흡수)는 이 폴백에서 크래시·빈 카드가 되지 않도록 방어**해야 한다(기존에도 카드 외부 라벨이 'Unnamed'였을 것).
- **카테고리·상태 색 분별(C)은 PipelineGraphView엔 비적용**(category/deployStatus data 미주입) — 회귀는 아니지만 색 분별은 ToolCanvasView 채널에만 나타난다.
- **edge 조건 배지(E)도 ToolCanvasView 전용**: PipelineGraphView 엣지는 stage 연결만 있고 `condition`이 없다(PipelineGraphView.svelte:43-54). PipelineGraphView는 이미 `.svelte-flow__edge-text`/`edge-textbg` 스타일(:101-107)을 갖고 있으나 조건 라벨 자체가 없으므로 배지 대상이 아니다.

### Background가 두 뷰에 중복 정의됨 (B-1 동기 필요)

`<Background variant=Dots gap={20} size={1.5} bgColor patternColor>`가 **ToolCanvasView.svelte:44와 PipelineGraphView.svelte:90에 각각 중복 정의**돼 있다. B-1의 대비·간격 튜닝은 **양쪽을 동일 값으로 동기**해야 두 뷰 일관성이 유지된다(한쪽만 바꾸면 뷰 전환 시 배경이 튄다).
