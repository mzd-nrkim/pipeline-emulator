# ToolFlowNode 브랜드 로고 + lucide 폴백 아이콘 교체

> 상태: 통테통과-완료

`done/2026-07-20.toolflownode_실제_n8n_룩_재디자인.md`는 D-0 결정에서 **"아이콘: 이모지 유지(1차) / SVG 도입(후속)"** 으로 확정하며 브랜드 로고 도입을 **명시적으로 후속 계획으로 분리**했다. 그 후속 계획이 본 문서다. 현재 `ToolFlowNode.svelte:43`은 `toolCatalog.icon`의 이모지 문자열(`🔄`·`📡`·`🗄️` 등)을 흰 카드 중앙에 그대로 렌더한다. 실제 n8n은 각 노드에 **해당 서비스의 브랜드 SVG 로고**를 쓰고, 로고가 없는 일반 역할 노드는 **미니멀 라인 아이콘**을 쓴다. 본 계획은 **브랜드 로고 우선 + lucide-svelte 아이콘 폴백** 방식으로 이모지를 교체한다(이모지는 최종 폴백으로만 잔존).

---

## 목표

- 실제 서비스가 있는 툴 노드는 **그 서비스의 브랜드 SVG 로고**로 렌더된다(Apache NiFi·Airflow·Debezium·MySQL·Elasticsearch·Kibana·S3·Valkey·LangChain 등).
- 브랜드 로고가 없거나 부적절한 일반 역할 노드(`dam` 외부 API, `kure-embedding`, `airflow-branch` 분기 등)는 **lucide-svelte 아이콘**으로 렌더된다.
- 브랜드/lucide 어느 쪽도 매핑되지 않은 미지의 툴은 **기존 이모지(`❓` 등)로 폴백**해 렌더 계약이 절대 깨지지 않는다(3단 폴백: brand → lucide → emoji).
- 아이콘 해석 로직이 **뷰(`ToolFlowNode.svelte`)에 흩어지지 않고 단일 리졸버**로 캡슐화된다(`toolId` → 아이콘 종류/자산).
- `toolCatalog`의 기존 `icon`(이모지) 필드와 `data` 렌더 계약은 **하위호환 유지**(최종 폴백으로 계속 사용). 기존 e2e 셀렉터·핸들 개수·모달 드릴다운 **회귀 0**.
- 브랜드 로고 자산의 **출처·라이선스가 명시**되고(설계-port 계획의 "자산·라이선스 불명확 시 대체" 원칙 준수), 다크 모드에서도 로고가 뭉개지지 않는다(단색 로고는 currentColor/틴트, 컬러 로고는 배경 대비 확보).

## 접근 방법

### D-0 결정 항목 (착수 전 확정 필요)

1. **브랜드 로고 자산 소스** — (a) `simple-icons`(브랜드 SVG 패키지, 단색 path, CC0 유사 라이선스지만 상표 가이드 준수 필요) 의존성 추가 vs (b) `frontend/static/logos/<id>.svg`에 **자체 확보 SVG 수기 배치**(출처별 라이선스 개별 확인). → **기본안 (a) simple-icons** — 20+ 벤더 SVG를 수기 확보하는 부담을 없애고 단색 path라 다크모드 틴트가 쉽다. 단 컬러 브랜드감이 약하면 주요 노드만 (b)로 보강.
2. **렌더 방식** — (a) 인라인 `<svg>`(currentColor 틴트·CSS 제어 용이) vs (b) `<img src>`(정적 자산 단순). → **기본안 (a) 인라인** — simple-icons는 path 데이터를 직접 주입 가능, accent 틴트/다크모드 대응이 CSS만으로 된다.
3. **로고 색 처리** — (a) accent 벤더색으로 **단색 틴트**(미니멀·n8n 근접) vs (b) 브랜드 **원색 유지**(식별성↑, 다크모드 대비 이슈). → **기본안 (a) accent 틴트** + 카드 배경 accent 틴트 유지. 원색이 꼭 필요한 로고(예: 멀티컬러 브랜드)는 예외 허용.
4. **lucide 폴백 매핑** — 로고 없는 노드의 카테고리/역할별 lucide 아이콘 확정: `source`→`Radio`/`Download`, `switch`→`GitBranch`/`Split`, 임베딩→`Binary`/`Hash`, 외부 API→`Cloud`/`Globe`, sink→`Database`/`Archive`. → 구현 시 툴별 최적 아이콘 개별 선정(작업목록 M-2).

### 구현 접근

1. **아이콘 리졸버 신설** (`frontend/src/lib/canvas/nodeIcon.ts` 신규): `resolveNodeIcon(toolId: string, fallbackEmoji: string): NodeIconSpec` 형태로, 다음 3단 폴백을 반환한다 —
   - `{ kind: 'brand', slug }`: 브랜드 로고 매핑 테이블에 `toolId`가 있으면.
   - `{ kind: 'lucide', name }`: lucide 매핑 테이블에 있으면.
   - `{ kind: 'emoji', char: fallbackEmoji }`: 둘 다 없으면 기존 이모지.
   매핑 테이블(`BRAND_ICON_MAP`·`LUCIDE_ICON_MAP`)은 이 파일에 `toolId` 키로 정의한다. `toolCatalog`의 `icon`(이모지)은 fallback 인자로만 흘려보내 **catalog 구조 변경 최소화**.

2. **브랜드 매핑 정의** (`BRAND_ICON_MAP`): `toolId` → simple-icons slug(또는 static SVG 경로). 대상 —
   - `apache-nifi`→`apachenifi`, `apache-airflow`/`airflow-branch`→`apacheairflow`, `debezium`→`debezium`(simple-icons 유무 확인, 없으면 static SVG), `mysql`→`mysql`, `elasticsearch`→`elasticsearch`, `kibana`→`kibana`, `s3`→`amazons3`(또는 `amazonaws`), `valkey`→`valkey`(유무 확인), `docling-langchain`→`langchain`, `presidio`→`microsoft`(또는 static). simple-icons에 없는 slug는 **static SVG로 개별 확보**하거나 lucide 폴백으로 강등(M-1에서 실측).

3. **lucide 매핑 정의** (`LUCIDE_ICON_MAP`): 로고 없는/부적절 노드 — `dam`(외부 API)→`Cloud`, `kure-embedding`→`Binary`, `airflow-branch`가 브랜드 대신 분기 표현이 나으면→`GitBranch`. 카테고리 기본 폴백도 둔다(`source`/`task`/`switch`/`sink`).

4. **`ToolFlowNode.svelte` 렌더 분기**: `resolvedIcon`(현재 20행 부근, `$derived(icon || '❓')`) 로직을 `resolveNodeIcon(...)` 호출로 교체하고, `.node-icon` 마크업(현재 47행 부근, `<span class="node-icon">{resolvedIcon}</span>`)을 `spec.kind`에 따라 분기 — brand/lucide는 `<svg>`(또는 lucide 컴포넌트), emoji는 기존 `<span>`. 세 경로 모두 `.node-icon` 크기/중앙정렬/accent 틴트 스타일을 공유(레이아웃·핸들 정렬 불변).

5. **다크모드·틴트**: 단색 브랜드/lucide는 `fill: currentColor` 또는 `color: var(--node-accent)`로 틴트. 카드 배경 accent 틴트(재디자인 계획 R-2 산출물)와 대비 확보. 하드코딩 색 금지(DESIGN.md oklch 토큰·`--node-accent`만).

6. **자산·라이선스 문서화**: simple-icons 도입 시 `package.json` 의존성 + 라이선스 주석, static SVG 확보 시 출처를 `docs/` 메모 또는 파일 헤더 주석에 남긴다(design-port 계획의 "자산 출처 불명확 시 대체" 원칙).

7. **회귀 방어**: `buildNodesAndEdges.ts`의 `data.icon` 계약·핸들 개수·`.svelte-flow__node` 루트 클릭은 불변. e2e는 아이콘을 **텍스트로 단언하지 않으므로**(재디자인 계획 회귀 실측 결과 참조) 영향 낮음 — 단 `.node-icon` 내부가 이모지 텍스트→`<svg>`로 바뀌므로 아이콘 텍스트 매칭 셀렉터가 있으면 M-4에서 실측 조정.

## 실행 시 필수 고려사항

### catalog 구조 변경 최소화

리졸버가 `toolId`를 키로 매핑을 들고 있으므로 `toolCatalog.ts`의 `icon`(이모지) 필드는 **그대로 두고 최종 폴백 인자로만 재사용**한다. catalog에 `logoSlug` 같은 신규 필드를 추가하지 않는 것을 기본으로 하되, 리졸버 매핑을 catalog로 이관하는 게 더 응집도 높다고 판단되면 D-0에서 재검토(기본: 별도 리졸버 파일).

### simple-icons slug 실측

simple-icons에 `debezium`·`valkey`·`presidio`·`kure` 등 마이너 브랜드 slug가 **없을 수 있다.** 착수 시 실제 패키지에서 slug 존재를 grep으로 확인하고(M-1), 없으면 (a) static SVG 확보 또는 (b) lucide 폴백으로 강등한다. **없는 slug를 매핑에 넣으면 런타임에 로고가 사라지므로**, 리졸버는 slug 미존재 시 lucide→emoji로 자동 강등하는 방어 로직을 둔다.

### 번들 크기

simple-icons 전체 import는 번들을 부풀린다. **개별 아이콘 named import**(`import { siMysql } from 'simple-icons/icons'` 형태 또는 tree-shakeable 경로)로 사용 로고만 포함시킨다. lucide-svelte도 개별 컴포넌트 import(전체 배럴 import 금지).

### 두 뷰 공통

아이콘은 `ToolFlowNode.svelte` 단일 컴포넌트가 렌더하므로 `ToolCanvasView`·`PipelineGraphView` 양쪽에 자동 반영된다. 별도 뷰 편집 불필요(재디자인 계획과 동일 스코프).

### Node 게이트 위치 (Z-post)

`frontend/package.json` 존재 → `npm run check`/`build`/`vitest`/`playwright`는 gitignored `node_modules` 의존이라 **워크트리(Z-pre) 신뢰 불가 → 머지 후 원본 main(Z-post)** 실행. DB·마이그레이션 무관(순수 프론트 렌더).
