# PII 통계(pii-stats · PiiCountGrid) 실데이터 연결

> 상태: 통테통과-완료

PII 감지 건수를 다루는 두 자산이 있으나 실제 페이지에 노출되지 않는다:
- 백엔드 집계 엔드포인트 `GET /stages/pii-stats`(`ui-backend/app/api/stages.py` → `get_pii_stats()`)는 **프론트엔드 소비자가 전무**하다(fetch 0건).
- 표시 컴포넌트 `PiiCountGrid.svelte`(props `counts: PiiCount[]`)는 **`/components` 쇼케이스(117-122줄)에서 정적 예시(`examplePii`)로만 렌더**된다.
- 문서별 `Document.piiCounts`는 `types.ts`에 정의되고 mock(`lib/mock/documents.ts`)에만 채워져 있으나, 실제 문서 페이지·노드 상세에 그리드로 표시되지 않는다.

즉 "PII 밀도/감지 건수"라는 에뮬레이터의 핵심 서사가 백엔드·컴포넌트·타입까지 갖춰졌는데 **실제 화면에 연결만 안 된** 상태다.

> 스코프 경계: 다른 미연결 항목(재실행·설정 저장·SSE·executions)과 분리. 본 계획은 PII 통계를 실제 페이지에 노출하는 배선에 집중한다.

---

## 목표

- 어댑터에 `fetchPiiStats()`가 추가되어 백엔드 `/stages/pii-stats`를 호출한다(real: 실제 fetch, mock: 대칭 stub).
- `PiiCountGrid`가 쇼케이스가 아닌 **실제 페이지**(개요/파이프라인 또는 문서 상세)에서 실데이터로 렌더된다.
- 문서별 `piiCounts`(존재 시)가 문서 페이지 또는 노드 상세에서 `PiiCountGrid`로 표시된다.
- 로딩·빈 결과·오류 상태가 처리되고, `planned` PII 유형은 흐리게 표시된다(컴포넌트 기존 규약 활용).

## 접근 방법

1. **집계 어댑터 신설**: `real-adapter.ts`에 `fetchPiiStats()` — `GET ${BASE}/stages/pii-stats`, 비정상 응답 시 오류. `mock-adapter.ts`에 대칭 stub(`PiiCount[]` 결정적 반환). 반환형은 백엔드 `get_pii_stats()` 형태에 맞춰 `PiiCount[]`(또는 stage별 맵)로 `types.ts` 정합 확인.
2. **노출 지점 결정(택1, plan-review 확정)**:
   - (a) **집계 뷰**: 파이프라인/개요 페이지에 "PII 감지 통계" 패널을 추가하고 `fetchPiiStats()` 결과를 `PiiCountGrid`로 렌더. 상단 "투입 볼륨 · PII 밀도 높음"(`pipeline/+page.svelte:112`) 문구를 실수치로 대체하는 방향과 연결.
   - (b) **문서별 뷰**: 문서 페이지/노드 상세에서 선택 문서의 `Document.piiCounts`를 `PiiCountGrid`로 렌더(집계 API 없이도 mock 데이터로 동작).
   - 권장: (a)를 1차(집계 API 실연결이 목적), (b)는 추가 노출로 병행.
3. **로드 경로**: 집계는 해당 페이지 `+page.ts` 로더 또는 온디맨드 `$effect`로 `fetchPiiStats` 호출. 문서별은 이미 로드된 `piiCounts`를 그대로 사용(추가 fetch 불필요).
4. **상태·빈 처리**: `piiStatsLoading`/`piiStatsError`/빈 배열 처리. `PiiCountGrid`의 `planned` 흐림 처리 규약 유지.
5. **mock/real 대칭**: mock stub이 UI 형태를 동일하게 채우도록 실제감 있는 결정적 데이터 제공. real은 백엔드 집계값 반영.

## 실행 시 필수 고려사항

- **노출 지점 미선택 근거 (A-1)**: (a)집계 패널(파이프라인/개요에 `fetchPiiStats` 결과 렌더) / (b)문서별(`Document.piiCounts` 렌더) / 병행 중 택일. 집계 API 실연결이 1차 목적이므로 (a) 권장, (b)는 추가 노출로 병행 가능. **실물(대상 페이지 레이아웃·기존 문구 위치)을 확인한 뒤** 결정한다 — 권장값 자율 확정 금지.
- **동일 파일 편집 충돌 조건부**: (a)를 파이프라인 페이지(`pipeline/+page.svelte:112` "PII 밀도 높음" 문구 대체)에 붙이면 다른 4개 계획과 **같은 파일**을 편집해 충돌한다. 개요 페이지(`[mode=mode]/+page.svelte`)나 문서 페이지에 붙이면 충돌을 피한다. **노출 지점 결정 시 파일 충돌 회피를 고려**한다 — 개요/문서 페이지 선택이 병렬성 측면에서 유리.
- **반환형 선대조 전제 (A-2)**: 백엔드 `get_pii_stats()` 반환이 `PiiCount[]`인지 stage별 맵인지 **먼저 확인**해 `types.ts`의 `PiiCount`와 정합시킨다. `PiiCountGrid`는 `counts: PiiCount[]`를 받으므로 맵이면 변환 필요.
- **기존 자산 재사용**: `PiiCountGrid.svelte`는 이미 `planned` PII 유형 흐림 처리 규약을 가진다 — 신규 렌더에서 이 규약을 유지한다. 쇼케이스(`/components` 117-122) 예시 렌더는 회귀 없이 그대로 둔다.
- **테스트 하네스 전제**: e2e는 `--mode test` mock 모드. mock `fetchPiiStats`가 결정적 stub을 채우므로 "비쇼케이스 경로에서 실데이터 그리드 렌더"를 mock으로 단언 가능.
