# SSE 실시간 파이프라인 상태 구독

> 상태: 통테통과-완료

에뮬레이터는 `subscribePipelineStatus(onChange)` 어댑터(real: `EventSource(/sse/stages)`, mock: noop stub)와 백엔드 SSE 엔드포인트 `GET /sse/stages`를 갖추고 있으나, **이 함수를 구독하는 코드가 프론트엔드 전체에 0건**이다(`frontend/src/lib/api/*-adapter.ts`에 정의만 존재). 그 결과 파이프라인 캔버스는 최초 로드 시점의 정적 스냅샷에 머물고, 노드를 트리거하거나 "재실행"해도 **단계 상태가 실시간으로 진행되지 않는다**. 개요 페이지가 약속하는 "투입 → 6단계 실행 → 결과 확인"(`[mode=mode]/+page.svelte:46`) 경험이 성립하려면 실행 상태의 라이브 반영이 필요하다.

> 스코프 경계: "재실행" 버튼 배선·노드 설정 저장은 각각 별도 계획서. 본 계획은 실행 중 단계 상태를 캔버스에 라이브 반영하는 구독 배선에 집중한다. 단, "재실행 후 진행이 시각적으로 보인다"는 최종 경험은 재실행 계획과 본 계획이 함께 충족한다.

---

## 목표

- 파이프라인 페이지 진입 시 `subscribePipelineStatus`를 구독하고, 이탈 시 정리(cleanup)한다(구독 누수 없음).
- SSE로 도착한 단계 상태 이벤트가 캔버스 노드/단계의 상태(`StatusDot`·`StageNode` 등)에 반영되어, 트리거·재실행 후 진행이 실시간으로 보인다.
- mock 모드는 noop stub이므로 구독해도 안전(오류·누수 없음)하고, real 모드는 백엔드 `/sse/stages`에서 실제 이벤트를 수신한다.
- SSE 오류·연결 종료 시 UI가 깨지지 않는다(graceful degradation — 정적 스냅샷으로 유지).

## 접근 방법

1. **구독 라이프사이클**: 파이프라인 페이지(`[mode=mode]/pipeline/+page.svelte`)에서 `$effect`로 `currentAdapter.subscribePipelineStatus(onEvent)`를 구독하고, effect cleanup에서 반환된 해제 함수를 호출한다. Svelte 5 `$effect` 정리 규약을 따른다.
2. **이벤트 → 상태 반영 모델**: SSE 이벤트 스키마(`types.ts:80` "SSE 이벤트" 타입)를 확인해, 도착 이벤트를 단계 id별 상태 맵(`$state`)으로 병합한다. 캔버스/노드 컴포넌트가 이 맵을 읽어 상태 색상·`StatusDot`를 갱신하도록 파생 바인딩한다. (이벤트 payload가 stage 단위인지 task 단위인지 백엔드 `sse.py`로 확정.)
3. **캔버스 반영 경로**: `ToolCanvasView`/`StageNode`가 상태를 prop으로 받도록 라이브 상태 맵을 주입한다. 기존 `data.stages`(정적 로드)와 라이브 이벤트를 병합하는 우선순위(라이브 > 스냅샷)를 정한다.
4. **오류 처리**: real 어댑터의 `es.onerror`는 이미 `es.close()`한다 — 재구독/백오프가 필요한지 검토(초기 버전은 단순 종료+정적 유지로 스코프 제한 가능). plan-review에서 재연결 정책 확정.
5. **mock 경로**: mock stub은 이벤트를 발생시키지 않으므로, mock 모드 데모용으로 타이머 기반 가짜 진행 이벤트를 방출할지 여부를 결정한다(선택 — 데모 품질 vs 스코프). 기본은 구독만 안전 배선.

## 실행 시 필수 고려사항

- **동일 파일 편집 충돌 (병렬 금지)**: 본 계획은 `frontend/src/routes/[mode=mode]/pipeline/+page.svelte`에 `$effect` 구독을 추가한다. 같은 세트의 재실행 버튼·노드 설정 저장·executions 뷰 계획도 **같은 파일**을 편집하므로 병렬 워크트리 동시 실행 시 충돌한다. 순차 실행+순차 머지로 진행한다.
- **스키마 선확인 전제 (A-1)**: SSE 이벤트 payload가 **stage 단위인지 task 단위인지** 백엔드 `sse.py`와 `types.ts`(line 80 "SSE 이벤트" 타입)를 **먼저 확인**한 뒤 상태 맵 키 구조를 확정한다 — 추측으로 병합 로직을 짜지 않는다. 이 확정이 C(상태 맵)·D(주입) 전제다.
- **재연결 정책 미선택 근거 (C-2)**: real `es.onerror`는 이미 `es.close()`한다. 초기 버전은 **단순 종료+정적 스냅샷 유지**로 스코프 제한한다(백오프/재구독 미도입). 실시간이 끊겨도 페이지가 정적으로 남으면 수용 가능하다는 판단. 재구독이 필요하면 후속 계획으로 분리.
- **테스트 하네스 전제 (e2e mock 한계)**: e2e는 `--mode test`로 mock 모드 강제 → mock `subscribePipelineStatus`는 noop stub이라 **이벤트를 방출하지 않는다**. 따라서 "트리거 후 상태 실시간 변화" 단언은 real SSE 없이는 검증 불가하다. 선택지: (b1) mock 데모 타이머 이벤트를 도입해 e2e로 상태 변화 단언, (b2) e2e는 "구독 배선·cleanup 누수 없음"만 단언하고 실제 진행 반영은 real 수동 확인. B-3(선택 항목)이 이 갈림을 결정한다.
- **구독 누수 회귀**: `$effect` cleanup에서 해제 함수를 반드시 호출 — 페이지 이탈·모드 전환 시 `EventSource`가 닫혀야 한다(devtools Network EventStream 확인). Svelte 5 `$effect` 정리 규약 준수.

## Verification

- **real 모드 수동 확인**: 백엔드 기동 상태에서 노드 트리거/재실행 시 캔버스 단계 상태가 running→completed 순으로 실시간 변화한다.
- **구독 누수 없음**: 페이지 이탈·모드 전환 시 `EventSource`가 `close()`되어 열린 연결이 남지 않는다(devtools Network EventStream 확인).
- **mock 안전성**: mock 모드에서 구독해도 콘솔 오류·누수가 없다.
- **정적 검증**: `npm run check`(svelte-check) 통과 — 이벤트 타입 정합.
- **e2e (post-gate)**: 트리거→상태 변화 스펙이 `playwright` 통과. real SSE가 무거우면 mock 데모 이벤트 기반 단언으로 대체.
- **회귀 없음**: SSE 미수신/오류 시에도 페이지가 정적 스냅샷으로 정상 렌더된다.

### TC 검토 (Right-BICEP · CORRECT)

- **Right(정상)**: 트리거 후 이벤트 도착 → 단계 상태 running→completed 반영 (real 수동 / B-3 도입 시 e2e).
- **B(경계)**: 이벤트 0건(mock noop) → 정적 스냅샷 유지, 오류 없음.
- **I(역)**: 알 수 없는 stage id 이벤트 도착 시 병합 로직이 깨지지 않음(무시 또는 신규 키 추가).
- **C(교차검증)**: mock stub(무이벤트) vs real EventSource 대칭 — 구독 자체는 양쪽 안전.
- **E(오류조건)**: `es.onerror`→`close()` 후에도 페이지 정적 유지(graceful degradation).
- **P(성능/누수)**: 페이지 이탈·모드 전환 시 EventSource close — **연결 누수 없음**(핵심 TC, Z-post).
- **CORRECT-Time/Ordering**: 이벤트 순서 뒤바뀜/지연 도착 시 마지막 상태가 최종 반영(라이브 > 스냅샷). 동일 stage 연속 이벤트 시 최신값 우선.
- **CORRECT-Reference**: 라이브 맵이 없는 stage는 정적 `data.stages` 폴백 참조.
- **Conformance**: 상태 맵 키가 A-1 확정 스키마(stage/task 단위)와 정합.
