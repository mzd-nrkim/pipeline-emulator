# ToolFlowNode 실제 n8n 시각 룩 재디자인

> 상태: 머지완료-통테대기

기존 계획서 `docs/plan/done/2026-07-20.xyflow_v1_업그레이드_n8n스타일_커스텀노드.md`로 만든 `ToolFlowNode`는 "n8n 스타일"을 표방하지만 실제 n8n과 시각 언어가 다르다. 현재 노드는 **벤더색으로 꽉 채운 헤더 바 + 헤더 안 라벨 + 이모지 아이콘 + `role`/`⚡trigger`/`🟢🟡🔵🔴 applyMode`/`status` 배지 행** 구조다(`frontend/src/lib/components/ToolFlowNode.svelte`). 반면 실제 n8n은 **흰/밝은 정사각형 카드 + 중앙 큰 아이콘 + 카드 *바깥 아래* 라벨 + 배지 없는 미니멀 + 은은한 accent(좌측 스트라이프/아이콘색)**다. 본 계획은 도메인 정보(role·applyMode·trigger)를 유지하되, 상시 노출 대신 미니멀하게 재배치해 실제 n8n 룩에 근접시킨다.

---

## 목표

- 노드가 **벤더색 헤더 바 없이** 흰/밝은 카드(`--surface`)로 렌더되고, accent는 **좌측 스트라이프 또는 아이콘 색**으로만 은은하게 드러난다.
- `displayName`·`vendor`가 **카드 내부 헤더가 아니라 카드 바깥 아래**에 중앙 정렬로 표시된다(n8n 라벨 배치).
- 아이콘이 **카드 중앙에 크게** 배치된다(이모지 유지 또는 SVG 브랜드 아이콘 — 결정 항목 D-0).
- `role`·`applyMode`·`trigger` 도메인 정보가 **상시 배지 행이 아니라** hover/선택 시 노출 또는 코너 아이콘/점(dot)으로 미니멀하게 표현된다(정보는 유지, 시각 노이즈 제거).
- 선택된 노드는 실제 n8n처럼 **진한 테두리/외곽 강조**가 생겨 현재 선택 상태가 시각적으로 명확해야 한다.
- 노드뿐 아니라 캔버스도 n8n 룩에 맞게 **점선 배경·회색 베지어 엣지·작은 원형 핸들·컨트롤 버튼 톤**을 조정한다.
- 엣지 화살촉(`markerEnd`) 추가 여부가 **명시적으로 결정**된다(현재 미구현 — n8n도 화살촉 미사용이라 유지가 정합적일 수 있음).
- 좌 target 1개 + 우 source(route 다중) **핸들 개수·nodeclick 드릴다운·기존 e2e 셀렉터가 회귀 없이** 동작한다.

## 접근 방법

1. **레이아웃 재구성 (헤더 바 → 흰 카드 + 라벨 하단)**: `ToolFlowNode.svelte`의 마크업을 (a) 정사각형에 가까운 카드(`--surface` 배경, `--border` 테두리, 은은한 accent 좌측 스트라이프 `border-left` 또는 아이콘 배경 틴트) + (b) 카드 중앙 큰 아이콘 + (c) 카드 **아래** 별도 라벨 블록(displayName 굵게, vendor 작게, 중앙 정렬)으로 재작성한다. 현재 `.node-header{background:var(--node-accent);color:#fff}`를 제거하고 accent는 스트라이프/틴트로만 사용. DESIGN.md 토큰(`--surface`·`--surface-muted`·`--border`·`--primary`) 준수.
   - **핸들 위치 주의**: 라벨을 카드 바깥으로 빼면 컴포넌트 루트 높이가 커진다. `<Handle position=Left/Right>`는 루트 기준 `top:50%`로 배치되므로, **카드 영역과 핸들 세로 중심이 어긋나지 않도록** 카드를 별도 래퍼로 감싸고 핸들 top을 카드 중심에 맞춘다(현재 route 다중출력의 `top:{((i+1)/(n+1))*100}%}` 계산도 카드 기준으로 보정).

2. **도메인 배지 미니멀화**: 현재 상시 노출되는 `.node-badges` 행(role/trigger/applyMode/status)을 다음 중 택1로 전환(D-0 결정) —
   - (기본안) **hover/선택 시에만** 카드 하단/툴팁으로 노출. 평상시엔 `trigger`는 카드 코너 `⚡` 아이콘, `applyMode`는 좌측 스트라이프 **색**으로 인코딩(🟢runtime→녹/🟡restart→황/🔵code→청/🔴readonly→적), `role`은 라벨 vendor 줄에 흡수하거나 생략.
   - 도메인 정보를 완전히 잃지 않도록, hover 시 상세는 반드시 노출한다.
   - hover-only 정보가 키보드 사용자에게 사라지지 않도록 `focus-within` 또는 `aria-label`/`title` fallback을 둔다.

3. **아이콘 처리 (D-0 결정)**: `toolCatalog.icon`은 현재 이모지다(`icon: '🔄'` 등, 주석상 "이모지 또는 약어 배지(svg 경로 폴백)"). 진짜 n8n은 SVG 브랜드 로고를 쓰지만 20+ 벤더 SVG 자산 확보는 범위가 크다 → **1차는 이모지를 흰 카드 중앙에 크게 배치**(저위험·즉시 가능)하고, SVG 도입은 후속 계획으로 분리 권장. 카드 배경에 accent 틴트를 살짝 주면 이모지도 브랜드감이 살아난다.

4. **선택 상태 스타일**: `NodeProps`의 `selected` 값 또는 xyflow가 붙이는 `.svelte-flow__node.selected` 클래스를 사용해 선택 노드에 n8n식 강조 테두리/그림자/핸들 활성색을 적용한다. 카드 안쪽 내용이 움직이지 않도록 `outline` 또는 box-shadow 기반으로 처리하고, 선택 전후 노드 크기가 바뀌지 않게 한다.

5. **캔버스/엣지 룩 보정**: 참고 n8n 이미지는 노드뿐 아니라 캔버스 문법이 강하다. `ToolCanvasView.svelte`와 동일하게 `PipelineGraphView.svelte`도 확인해 `<Background />`의 점 간격/색/크기, `<Controls />` 버튼 톤, edge stroke/width/label 배경을 n8n에 가깝게 조정한다. 다만 기능 계약(`nodes`/`edges` shape)은 건드리지 않는다.

6. **엣지 화살촉 결정 (D-0)**: 현재 `buildNodesAndEdges.ts`의 `FlowEdge`에 `markerEnd`가 없어 화살촉이 렌더되지 않는다(순수 베지어 곡선, 방향은 좌/우 핸들·`animated`로만 전달). n8n도 화살촉을 쓰지 않으므로 **미추가 유지**를 기본안으로 한다. 만약 추가한다면 `<SvelteFlow defaultEdgeOptions={{ markerEnd: { type: MarkerType.ArrowClosed } }}>`(MarkerType은 `@xyflow/svelte` export 확인됨)로 두 뷰에 일괄 적용.

7. **스타일 스코프**: 주 변경은 `ToolFlowNode.svelte`의 마크업·`<style>`에 집중한다. 다만 캔버스 배경/컨트롤/엣지 시각 보정은 `ToolCanvasView.svelte`·`PipelineGraphView.svelte`의 `SvelteFlow` 옵션/전역 xyflow 클래스 스타일에 한해 허용한다. `buildNodesAndEdges.ts`의 `data` 계약(icon·accent·vendor·displayName·role·trigger·applyMode·outputs·outOfTeamScope)은 **그대로 유지**(재디자인은 뷰 레이어). 단 배지→코너아이콘/스트라이프 전환에 필요한 추가 data 필드가 없다면 신설 없이 기존 필드만으로 처리한다.

8. **회귀 방어**: e2e(`graph-lr-handle.spec.ts`의 `.svelte-flow__handle-left/right` 개수, `route-split.spec.ts` 분기, nodeclick 드릴다운, role 레이블 셀렉터)가 노드 내부 텍스트/구조에 의존한다. 라벨을 카드 밖으로 옮기고 배지를 hover화하면 **텍스트 매칭 셀렉터가 깨질 수 있다** → Z-post playwright에서 실측 조정. 핸들 개수(좌1·우≥1)는 **불변 유지**가 목표.

## 실행 시 필수 고려사항

### 단일 파일 집중 + data 계약 불변

핵심 변경은 `ToolFlowNode.svelte` 한 파일이다. `buildNodesAndEdges.ts`의 `data` 필드·`type:'tool'`·핸들 개수 계약은 건드리지 않는 것을 원칙으로 한다(재디자인 = 뷰 레이어). 이로써 두 뷰(`ToolCanvasView`·`PipelineGraphView`)·단위테스트 회귀 표면을 최소화한다.

### D-0 결정 항목 (구현 착수 전 확정)

1. **배지 처리 방식**: hover/선택 노출 vs 코너 아이콘+스트라이프 색 인코딩 vs 혼합. → 기본안: 혼합(trigger=코너아이콘, applyMode=좌측 스트라이프 색, 상세는 hover).
2. **아이콘**: 이모지 유지(1차) vs SVG 도입(후속). → 기본안: 이모지 유지.
3. **엣지 화살촉**: 미추가 유지 vs ArrowClosed 추가. → 기본안: 미추가 유지.
4. **status 배지**: 기존 계획에서 이번 범위 제외됨 → 유지(제외).

### 핸들 세로 정렬 회귀

라벨을 카드 밖 하단으로 빼면 컴포넌트 루트 높이 증가 → 핸들이 카드 중심에서 벗어난다. **단일 root wrapper 안에 카드 래퍼와 라벨 래퍼를 분리**하고, 핸들 `top`을 카드 중심 기준으로 명시 조정한다. route 다중출력 핸들의 `top` 비율 계산도 카드 높이 기준으로 보정(라벨 영역 제외). 카드 높이는 CSS 변수(`--node-card-size` 등)로 고정해 label 길이·hover 상세·선택 outline이 핸들 위치를 밀지 못하게 한다.

### 선택/hover 접근성

hover 시에만 표시되는 상세 정보는 키보드 포커스에서도 동일하게 확인 가능해야 한다. `:hover`와 `:focus-within`을 함께 쓰거나, 시각적으로 접힌 상세 정보를 `aria-label`/`title`로 보강한다. 선택 상태는 색만으로 구분하지 않고 테두리 굵기/outline/box-shadow 중 하나를 함께 사용한다.

### 캔버스 룩 스코프

실제 n8n 룩은 노드 단품보다 캔버스 전체 인상이 크다. `<Background />`는 점선 그리드가 보이도록 색·gap·size를 명시하고, 엣지는 과한 `animated` 인상이 강하면 stroke/opacity/animation 강도를 조정한다. `<Controls />`는 현재 기본 스타일이 튀면 xyflow 클래스 override로 n8n에 가까운 작은 사각 버튼 톤으로 맞춘다. 이 변경은 `ToolCanvasView.svelte`와 `PipelineGraphView.svelte` 양쪽 적용 여부를 확인한다.

### 다크 모드

DESIGN.md는 라이트/다크 토큰을 모두 정의한다(`--surface` 등 oklch 라이트·다크쌍). 흰 카드가 다크 모드에서 `--surface`(다크: `oklch(0.19 ...)`)로 자동 전환되도록 하드코딩 `#fff` 사용을 피하고 토큰만 사용한다(현재 헤더 `color:#fff` 제거로 자연 해소).

### e2e 회귀 표면 실측 (셀렉터 의존성 확인 결과)

plan-review에서 e2e 스펙을 실측한 결과, 노드 재디자인이 깨뜨릴 수 있는 회귀 표면은 **텍스트 셀렉터가 아니라 구조 셀렉터**다:

- `graph-lr-handle.spec.ts`: `.svelte-flow__handle-left`/`-right`/`-top`/`-bottom` **개수**에만 의존 → 라벨/배지 변경과 무관. 핸들 `type`/`position` 계약만 지키면 불변. (좌1·우≥1 유지가 핵심)
- `route-split.spec.ts`: `main`·`h1`(`PipeScale`)·`'백엔드 연결 대기'` 텍스트에 의존 → **노드 내부 텍스트가 아님**, 재디자인 영향 없음.
- `node-detail-modal.spec.ts`: `.svelte-flow__node` 엘리먼트 클릭 → `[role="dialog"]` 모달 검증. 노드 루트 클래스(`.svelte-flow__node`)와 클릭 이벤트 전파에만 의존 → 카드/라벨 래퍼 분리 시 **루트가 여전히 클릭 가능**해야 함(hover 상세 오버레이가 pointer 이벤트를 가로채지 않도록 `pointer-events` 주의).

→ 결론: "텍스트 매칭 셀렉터가 깨질 수 있다"는 접근 방법 8의 우려는 실측상 낮다. **실제 방어 초점은 (a) 핸들 개수/위치 불변, (b) `.svelte-flow__node` 루트 클릭 가능성, (c) hover 오버레이의 pointer-events 간섭 방지**다.

### Node 정적/e2e 게이트 위치 (Z-post)

`frontend/package.json` 존재 → `npm run check`·`build`·`vitest`·`playwright`는 gitignored `node_modules` 의존이라 **워크트리(Z-pre)에서 신뢰 불가 → 머지 후 원본 main(Z-post)** 에서 실행한다. 마이그레이션 SQL·DB 변경 없음(순수 프론트 렌더).
