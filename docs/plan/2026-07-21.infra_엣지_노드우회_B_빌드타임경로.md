# 인프라 뷰 엣지 노드 우회 — Plan B: 빌드타임 routeY + 런타임 커스텀 컴포넌트

> 상태: 대기중
> 선행 계획: `2026-07-21.인프라뷰_LR계층_꺾은선엣지_재설계.md` (머지완료)

## 문제

LR 계층 배치에서 depth gap > 1인 인프라 엣지(`A depth0 → C depth2` 등)가
중간 컬럼(depth1)의 노드 영역을 관통한다.

**근본 원인**: xyflow 내장 `smoothstep`의 `getSmoothStepPath`는 LR(Right→Left) 모드에서
`center.y`를 무시하고 `verticalSplit` 경로를 생성한다. source·target이 동일 y일 때
verticalSplit이 수평선으로 퇴화해 중간 노드를 그대로 가로지른다.
`pathOptions.offset`/`stepPosition`은 수직 굴절점 x 위치만 조정하므로 이 문제를 해결할 수 없다.

## 목표

- depth gap > 1인 인프라 엣지가 중간 컬럼 노드 영역을 통과하지 않는다.
- 경로는 모든 노드 최상단(y < 0) 위를 통과하는 U자형 라우팅이다.
- **xyflow Handle 앵커 유지**: path 시작·끝이 실제 Right/Left 핸들 좌표에 정확히 고정된다.
- depth gap = 1인 단일 컬럼 엣지는 현행 `smoothstep`을 유지한다.
- 기존 `buildNodesAndEdges` 단위테스트 전량 통과, e2e 시각검증 통과.

## 접근 방법

역할을 둘로 나눈다:

| 책임 | 위치 | 이유 |
|------|------|------|
| **우회 y좌표 결정** (`routeY`) | `buildNodesAndEdges.ts` (빌드타임) | 노드 배치는 빌드타임에 확정된다 |
| **실제 SVG path 생성** | `InfraStepEdge.svelte` (렌더타임) | 핸들 픽셀 좌표(`sourceX/Y`, `targetX/Y`)는 xyflow가 렌더 후 `EdgeProps`로 전달 — 빌드타임에는 알 수 없다 |

빌드타임에 SVG path 문자열을 미리 만드는 방식은 **핸들 좌표를 추정값으로 구워버려 실제 핸들과 어긋난다**. 이 방식을 쓰지 않는다.

### 빌드타임: routeY 계산 (buildNodesAndEdges.ts)

depth gap > 1인 엣지에 대해 `data.routeY`만 계산해 담는다.

```
ROUTE_Y = -48   (노드 상단 y=0 기준, 48px 위)
depthGap = depth.get(e.to) - depth.get(e.from)
if depthGap > 1:
  edge.type = 'infra-step'
  edge.data = { routeY: ROUTE_Y }
else:
  edge.type = 'smoothstep'   (현행 유지)
```

### 렌더타임: path 생성 (InfraStepEdge.svelte)

`EdgeProps`에서 받은 실제 핸들 좌표로 6-point 경로를 그린다.

```
입력: sourceX, sourceY, targetX, targetY  ← xyflow가 실제 Handle에서 계산한 좌표
      data.routeY                          ← 빌드타임에 계산한 우회 y
OFF = 24  (핸들→첫 굴절점 간격)

경로:
  M sourceX sourceY           ← Right 핸들 (앵커 고정)
  L (sourceX+OFF) sourceY     ← 오른쪽으로
  L (sourceX+OFF) routeY      ← 위로 (노드 최상단 위)
  L (targetX-OFF) routeY      ← 수평 우회
  L (targetX-OFF) targetY     ← 타겟 y로
  L targetX targetY           ← Left 핸들 (앵커 고정)
```

꺾임 라운딩은 초기엔 L-join으로 단순 구현, 미관 확인 후 `borderRadius=6` 추가.

### Plan A와의 차이

Plan A는 `useNodes()` store를 읽어 중간 노드를 탐색하고 routeY를 런타임에 계산한다.
Plan B는 routeY를 빌드타임에 고정(`-48`)하고 store를 읽지 않는다. 더 단순하지만 토폴로지가 크게 달라지면 직접 조정이 필요하다.
