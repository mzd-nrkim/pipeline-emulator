# [BUG] get_gold_staged_documents() 컬럼 불일치 — 항상 빈 배열 반환

`ui-backend/app/services/mysql_aggregator.py`의 `get_gold_staged_documents()`가 `gold_staged_documents` 테이블에 존재하지 않는 컬럼(`title`, `priority`, `security_class`, `vehicle_model`, `is_masked`, `stage_reached`)을 직접 SELECT해 MySQL 오류가 발생하고 `except Exception: return []`가 이를 흡수, `/documents` 엔드포인트가 항상 빈 배열을 반환하는 결함이다.

---

## 목표

- `GET /documents`가 실제 gold staged 문서 목록을 반환한다
- 각 문서에 `id`, `title`, `priority`, `security`, `vehicleModel`, `masked`, `stageReached`, `piiCounts` 필드가 정확히 채워진다
- 쿼리가 실제 테이블 스키마와 정합하며, MySQL 오류 없이 실행된다

## 접근 방법

1. **실제 스키마 재확인**: `gold_staged_documents` DDL — `staged_id, enriched_id, es_field_info(JSON), role_ids, metadata_tags, pclrty_class, indexing_status`. `title`·`vehicle_model` 등은 `es_field_info` JSON 내부에 존재.
2. **JOIN 체인 재설계**: 단일 `gold_staged_documents` 조회 대신 아래 체인으로 데이터 추출:
   - `gold_staged` → `gold_enriched` (enriched_id) → `gold_chunked` (chunk_id) → `silver_masked` (masked_doc_id) → `silver_structured` (structured_doc_id) → `bronze_document_hub` (document_hub_hash_key)
   - 단, `gold_staged_documents`는 청크 단위(15행)이므로 **문서 단위로 GROUP BY 또는 MIN/MAX** 집계 필요 (5건 문서 = gold 15행)
3. **필드 추출 전략**:
   - `id`: `bronze_document_hub.source_primary_key`
   - `pclrty_class` → `security`
   - `is_masked`, `pii_pattern_types` → `silver_masked_documents`에서 JOIN
   - `vehicle_model` → `gold_staged.es_field_info` JSON 파싱 (MySQL `JSON_EXTRACT` 또는 Python 파싱)
   - `title`: `silver_structured.structured_content` JSON의 `data.title` 또는 `source_primary_key` 기반 생성
4. **piiCounts 포함**: `silver_masked.pii_pattern_types` JSON을 Python에서 파싱해 `[{type, label, count}]` 변환
