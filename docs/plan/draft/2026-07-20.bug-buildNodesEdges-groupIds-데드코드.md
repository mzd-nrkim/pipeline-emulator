# buildNodesAndEdges groupIds 데드코드 + 그룹 ID 하드코딩 제거

> 상태: 통테통과-완료

## 목표

Phase M-3 구현에서 `groupIds` Set이 수집되지만 이후 루프에서 미사용 데드코드로 잔존하고, Airflow 그룹 노드 생성 로직이 `'node-airflow-group'`이라는 고정 문자열에 의존해 다중 그룹 확장이 불가능하다. 데드코드를 제거하고 그룹 ID를 `groupIds` 순회 방식으로 교체한다.

## 실행 시 필수 고려사항

> plan-review 검토 중 실물 코드와 계획 전제가 상충하는 항목이 발견됐다. 구현 전 반드시 실물을 재확인한다.

- **① 필드 불일치 (핵심)**: `groupIds`는 `n.group` 필드에서 수집된다(`buildNodesAndEdges.ts:135-138`, `if ((n as any).group) groupIds.add((n as any).group)`). 그러나 그룹 노드 생성은 `n.parentId === 'node-airflow-group'`로 자식을 필터한다(L206). 즉 **그룹 id 원천(`.group`)과 실제 그룹 소속키(`.parentId`)가 서로 다른 필드**다. 하드코딩을 단순히 `groupIds` 순회로 교체하면, 자식은 여전히 `parentId`로 매칭되는데 그룹 id는 `.group`에서 오므로 **매칭 실패로 그룹 노드가 하나도 안 생길 수 있다**. 올바른 수정은 `groupIds`(=`.group`)가 아니라 **노드들의 distinct `parentId` 집합**을 순회하는 것일 가능성이 높다. topology의 실제 그룹키가 `.group`인지 `.parentId`인지 실물 확인 후 순회 원천을 결정한다 (실물 확인 후 결정).
- **② 기존 그룹 테스트 부재**: `buildNodesAndEdges.test.ts`의 fixture 노드(L27~)에는 `parentId`/`group` 필드가 없어 현재 그룹 노드 생성 경로(`view==='data'` && childNodes>0)가 **테스트되지 않는다**(childNodes 빈 배열). 따라서 계획서의 "기존 그룹 노드 테스트가 통과" 항목은 부정확 — 신규 다중그룹 테스트가 사실상 첫 그룹 커버리지다. 신규 fixture는 그룹 소속을 실제 트리거하는 필드(수정 후 확정된 원천 필드)를 세팅해야 한다.
- **③ 동일 파일 편집 충돌**: `2026-07-20.bug-edge-pill-svg-rx수정` 계획과 **동일 파일 `buildNodesAndEdges.ts`를 편집**한다(본 계획=그룹 노드부 L135-138/L205-225, 상대 계획=엣지 라벨부 L188-198). plan-run에서 **병렬 금지** — 직렬 배치하거나 한 에이전트로 묶어 실행한다.
- **② 패키지매니저**: 이 repo는 `package-lock.json`+`npm`. 계획서의 `pnpm`은 오기 — 게이트는 `cd frontend && npm run test:unit` 사용.

## TC

> 순수 함수 단위테스트. 그룹 노드 생성은 `view==='data'` && 그룹 소속 자식 존재가 트리거.

- [x] Right(정상): 그룹 소속 노드가 있는 data 뷰에서 그룹 id별 group 노드가 생성됨
- [x] B(경계): 그룹 소속 노드 0개면 group 노드 0개 (기존 fixture 회귀 — 그룹 노드 미생성 유지)
- [x] I(역·부정): 서로 다른 그룹키 2종 fixture에서 각 group 노드가 자기 자식만 경계에 포함 (교차 오염 없음)
- [x] C(교차확인): 각 group 노드의 경계(minX/minY/maxX/maxY)가 해당 그룹 자식 좌표에서만 산출됨
- Ordering: 해당 없음 (그룹 집합은 순서 무관, 단 위상정렬 X좌표 회귀는 기존 TC가 커버)
- DB 통합: 해당 없음 (프론트엔드 순수 함수)
- teardown: 해당 없음 (단위테스트 — 생성 리소스 없음)
