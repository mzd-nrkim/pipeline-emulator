# [FIX] FastAPI /stages root alias ↔ router sub-route 라우팅 충돌 검증·정리

> 상태: 머지완료-통테대기

`ui-backend/app/main.py`에 `@app.get("/stages")` root alias와 `stages.router`(prefix `/stages`) 아래 `GET /pii-stats` sub-route가 공존한다. FastAPI의 route 등록 순서 의존 우선순위 규칙에서 alias가 sub-route를 shadow할 수 있으며, 현재 실제 충돌 여부를 검증하지 않은 상태다.

---

## 목표

- `GET /stages` 와 `GET /stages/pii-stats` 가 각각 독립적으로 정상 동작한다
- root alias(`@app.get("/stages")`)가 `/stages/pii-stats` 를 가리지 않는다
- 불필요한 alias가 있으면 제거하거나 router 통합으로 정리한다

## 접근 방법

1. **실물 확인**: FastAPI는 route를 등록 순서대로 매칭한다 — `app.include_router(stages.router, prefix="/stages")` 후 `@app.get("/stages")` 가 선언된 순서 확인
2. **충돌 판별**: `/stages/pii-stats` 요청 시 root alias의 `get_stages_alias()`가 응답하는지, 아니면 `stages.router`의 `get_pii_stats_endpoint()`가 응답하는지 확인
3. **수정 전략** (충돌 있을 때):
   - `@app.get("/stages")` alias 제거 후 `stages.router`에 `GET ""` 엔드포인트로 통합 (현재 이미 존재: `@router.get("")`)
   - main.py의 `@app.get("/stages")` + import alias 제거
4. **수정 전략** (충돌 없을 때): 주석으로 "alias는 router 등록 이후 선언이므로 sub-route가 먼저 매칭됨" 명기하고 종료

## 실행 시 필수 고려사항

① **계획서 전제("alias가 sub-route를 shadow") 정정** — 실물상 `main.py`는 `include_router(stages.router, prefix="/stages")`(L9)를 `@app.get("/stages")`(L21)보다 **먼저** 등록하고, `stages.py`에는 `@router.get("")`(L22, `get_stages`)가 존재한다. FastAPI에서 `@app.get("/stages")`는 **exact-path 매칭**이라 `/stages/pii-stats`(별도 경로)를 애초에 shadow할 수 없다. 따라서 "sub-route shadow" 충돌은 라우팅 규칙상 성립하지 않는다 — `/stages/pii-stats`는 항상 `stages.router`의 `@router.get("/pii-stats")`가 처리한다.

② **실제 이슈 = alias dead code** — `/stages` GET은 먼저 등록된 `@router.get("")`가 매칭하므로 뒤에 선언된 `@app.get("/stages")` alias는 **도달 불가한 죽은 코드**다. 즉 A-2 판별 결과는 "비충돌"이지만, 그래서 B-2(주석 후 종료)가 아니라 **B-1(alias 제거)** 경로가 정답이다. 정적 판정만으로 제거 결정이 가능하며, A-3 실측(Z-post)은 회귀 확인용이다.

③ **회귀 대상 = alias 원 소비자** — alias는 "pipeline 페이지 호환" 주석이 달려 있다(L21). 제거 후 pipeline 페이지가 `/stages`(router "")로 동일 응답을 받는지 회귀 확인해야 한다. 응답 본문은 alias와 router 모두 `get_stages()`를 호출하므로 동일하다.

④ **읽기 전용** — 라우팅 정리만 수행. DB 레코드 생성/변경 없어 teardown 대상 없음.
