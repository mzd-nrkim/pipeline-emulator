# 인프라 뷰 force-directed 레이아웃 교체 (의존관계도 = 순서 아닌 연결 구조)

> 상태: 게이트통과-머지대기

인프라(의존관계) 뷰가 데이터 흐름도의 **순서 개념을 그대로 베껴** 계층을 줄세우고 있어, 실제 의존 구조와 어긋나고 엣지가 엉켜 보인다. 의존관계도는 순서(storage→…→serving)가 아니라 **"무엇이 무엇에 의존하는가"라는 연결 구조** 자체여야 하며, 데이터 흐름도의 배치와 **독립적**이어야 한다. 본 계획은 인프라 뷰 배치를 하드코딩된 계층 순서에서 **force-directed(d3-force) 네트워크 배치**로 교체한다.

## 현재 상태 근거

- `buildNodesAndEdges.ts:17` `INFRA_LAYER_ORDER = ['storage','broker','ingestion','processing','serving','other']` — 파이프라인 순서를 하드코딩.
- `buildNodesAndEdges.ts:21-40` `INFRA_LAYER_MAP` — 노드마다 계층을 수동 매핑.
- `buildNodesAndEdges.ts:53-79` `computeInfraPositions` — `y = layerIndex * INFRA_ROW_GAP`로 계층을 세로로 줄세움(= 데이터 흐름의 medallion 순서를 의존관계도에 투영).
- `buildNodesAndEdges.ts:137-140` — `view === 'infra'` 분기에서 `computeInfraPositions` 사용.
- 결과: `es → airflow`처럼 하위 계층에서 상위 계층으로 역류하는 엣지가 발생하고, L/R 핸들과 세로 계층 배치가 어긋나 엣지가 갈고리처럼 휘어 엉킴.

### 그래프 실제 구조 (dependency 엣지 8개, 노드 10개)

- 소스(in=0) 6개: es, mysql-container, valkey, seaweedfs, mock-api, zookeeper
- 싱크(out=0) 4개: airflow, debezium, nifi, kibana
- **airflow가 허브**(들어오는 의존 4개), mysql-container 부허브(2개). 사이클 없음(DAG), 깊이 사실상 2단.
- 순차 체인이 아니라 **허브형 네트워크** → 계층 줄세우기가 구조를 왜곡.

## 목표

- 인프라 뷰가 하드코딩된 계층 순서(`INFRA_LAYER_ORDER`/`INFRA_LAYER_MAP`) 없이, **의존 엣지 자체로부터 계산된 force-directed 배치**로 그려진다.
- 연결이 많은 허브(airflow·mysql-container)가 물리적으로 중앙에 몰려 **의존 중심성이 시각적으로 드러난다**.
- 배치가 **결정적(deterministic)** 이다 — 고정 토폴로지에 대해 새로고침·재렌더마다 동일한 좌표(시뮬레이션 1회 계산 후 freeze).
- 데이터 흐름 뷰(`view==='data'`)의 위상정렬 배치(`computeDepths`)는 **변경 없이 유지**되어, 두 뷰가 독립적으로 동작한다.
- infra 뷰의 채널 필터·고아 숨김·nodeclick 드릴다운·isInfra/animated 계약은 기존대로 동작한다.

## 접근 방법

1. **의존성 추가**: `d3-force`(및 타입) 1개를 `frontend`에 추가. xyflow는 순수 렌더러라 자동 배치 기능이 없으므로 좌표는 외부에서 계산해 넣는다.
2. **결정적 force 배치 유틸 신설** (`computeForceLayout`): `visibleNodes`·`visibleEdges`(dependency 채널)를 입력받아 d3-force 시뮬레이션을 **동기적으로 고정 tick 수만큼** 돌려 좌표를 반환한다.
   - 초기 좌표를 **인덱스 기반 원형(phyllotaxis/circle)으로 결정적 시드** → d3의 랜덤 초기화 회피(비결정성 제거).
   - `forceManyBody`(반발)·`forceLink`(엣지 인력, 거리 튜닝)·`forceCenter`(중심)·`forceCollide`(노드 겹침 방지) 조합. 노드 10개 규모에 맞춰 charge·linkDistance 튜닝.
   - `.stop()` 후 `for` 루프로 정해진 횟수 `.tick()` → 완료 시점 x/y를 좌표로 확정(freeze).
3. **infra 분기 교체** (`buildNodesAndEdges.ts:137-140`): `computeInfraPositions` 호출을 `computeForceLayout`로 교체. `computeInfraPositions`·`INFRA_LAYER_ORDER`·`INFRA_LAYER_MAP`·`getInfraLayer`·`InfraLayer` 타입 등 **계층 순서 하드코딩 자산을 제거**한다(data 뷰 경로는 무관).
4. **핸들 정합 검토**: force 배치에서는 노드가 임의 방향으로 놓이므로 L/R 고정 핸들(`ToolFlowNode.svelte:84,97`)이 엣지 방향과 어긋날 수 있다. xyflow가 `floating edge`(핸들을 노드 경계에 동적 부착)를 지원하는지 실측 → 지원 시 infra 엣지를 floating으로, 미지원/고위험이면 현행 L/R 유지하되 시각 허용 범위 확인(D-1 결정).
5. **테스트 재작성**: 계층 Y순서에 묶인 단위테스트(순서 전제)를 **구조 기반 단언으로 재작성**. e2e infra-view도 순서 좌표가 아닌 "노드 존재·엣지 연결·nodeclick 동작" 기준으로 재정렬.
6. **회귀 격리**: data 뷰 테스트·핸들 개수(`graph-lr-handle`)·모달(`node-detail-modal`)은 infra 배치와 무관하게 통과 유지 확인.

## 실행 시 필수 고려사항

### 순서 전제 테스트 재작성 (핵심 작업)

아래 단위테스트는 `computeInfraPositions`의 **계층 Y순서**를 직접 단언하므로 force 전환 시 반드시 재작성한다(단순 통과가 아니라 전제 교체):

- `buildNodesAndEdges.test.ts:157` "mysql-container Y < debezium/nifi (storage < ingestion)"
- `buildNodesAndEdges.test.ts:525` "es가 serving 계층 Y (kibana와 동일 Y)"
- `buildNodesAndEdges.test.ts:535` "es(serving) Y > debezium(ingestion)"
- `buildNodesAndEdges.test.ts:545` "mysql-container(storage) Y < es(serving)"

→ **구조 기반 재단언으로 교체**: (a) infra 뷰 노드/엣지 개수·채널 필터, (b) 배치 결정성(동일 입력 2회 호출 시 좌표 동일), (c) 허브(airflow)가 중심에 가까움(중심 대비 거리 비교) 등 순서에 의존하지 않는 성질로 검증. `isInfra`·`animated=false`·고아 숨김 테스트는 유지.

### 결정성 보장

d3-force는 초기 좌표 미지정 시 내부적으로 배치를 잔들며(랜덤성 유발), tick마다 상태가 변한다. **초기 좌표를 인덱스 결정적으로 시드 + 고정 tick 수 + 시뮬레이션 freeze**로 결정성을 확보해야 새로고침마다 동일 배치가 보장된다. 비결정 배치는 e2e 스냅샷·시각 검증을 깨뜨린다.

### 재설계 계획과의 동일 파일 충돌

`2026-07-20.실행주체구조_캔버스_재설계_airflow오케스트레이션.md`가 `buildNodesAndEdges.ts`의 노드/엣지 빌드와 `topology.ts`를 대규모로 바꾼다. 본 계획도 같은 파일(`buildNodesAndEdges.ts`)의 infra 분기를 수정 → **동일 파일 병렬 편집 충돌**. 두 계획을 함께 실행하면 순차 배치(본 계획 먼저 머지 권장 — infra 배치만 국소 수정이라 재설계의 선행이 되기 쉬움), 또는 재설계 착수 시 본 변경을 반영해 진행.

### 뷰 독립성 불변식

data 뷰(`computeDepths`)는 절대 건드리지 않는다. force 배치는 `view==='infra'` 경로에만 적용. 두 뷰 왕복(`buildNodesAndEdges.test.ts:167` data→infra→data) 후 data 뷰 엣지 수 동일 불변식 유지.
