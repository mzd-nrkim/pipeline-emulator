# Bug: polling/trigger_adapter.py register_bronze_event 시그니처 불일치

> 상태: 미시작

`polling_adapter.py`·`trigger_adapter.py`가 내부 래퍼 `register_bronze_event(record, change_operation)`를 통해 `scripts.ingest.register_bronze_event`를 `_register(change_operation=..., record=...)` 로 호출하나, 실제 시그니처는 `(conn, table_name, batch_id, s3_path, record_count, change_operation)` — `record` 파라미터가 없어 런타임 `TypeError` 발생.

---

## 목표

- `polling_adapter.py`·`trigger_adapter.py`의 Bronze 등록 호출을 `debezium_adapter.py` 참조 구현(conn+table_name+batch_id+s3_path+record_count+change_operation)과 계약 일치시킨다.
- NiFi ExecuteStreamCommand 실행 환경에서 MySQL 연결·S3 경로 생성·Parquet 업로드까지 포함한 end-to-end 경로가 실제로 동작한다.

## 접근 방법

1. **현황 파악**: `debezium_adapter.py`의 `register_bronze_event` 호출부(L265) 참조 — `get_mysql_connection`, `_make_s3_path`, `record_count=1` 패턴 확인
2. **래퍼 제거**: `polling_adapter.py`·`trigger_adapter.py`의 로컬 `register_bronze_event` 래퍼 함수를 삭제, `scripts.ingest`에서 `get_mysql_connection`·`register_bronze_event`를 직접 import
3. **호출부 재작성**: 각 어댑터의 메인 루프에서 conn 생성·table_name 고정(`source_cft_problem_history`)·batch_id(UUID 또는 PK)·s3_path(`_make_s3_path` 또는 동등 함수)·record_count(=1) 를 구성해 올바르게 호출
4. **단위테스트 갱신**: `test_polling_adapter.py`·`test_trigger_adapter.py`의 `register_bronze_event` mock 대상을 실제 경로(`scripts.ingest.register_bronze_event`)로 수정

## 실행 시 필수 고려사항

### xyflow 세션과의 충돌 — 없음 (프런트엔드 파일 0개)

편집 대상은 `scripts/cdc/`·`scripts/cdc/tests/`뿐. `frontend/`·Svelte·xyflow 파일을 전혀 건드리지 않는다. xyflow 업그레이드 세션과 **완전 독립** — 지금 즉시 병렬 실행 가능.

### 참조 구현 계약 (debezium_adapter.py L265-272)

올바른 호출은 **키워드 인자**로 `conn, table_name, batch_id, s3_path, record_count, change_operation` 전부 전달한다(`scripts/ingest.py` L42-49 시그니처, `change_operation` 기본값 `"snapshot"`). `record` 인자는 존재하지 않는다 — 현재 두 어댑터의 래퍼가 `_register(change_operation=..., record=...)`로 호출해 `TypeError` 발생.

### mock patch 대상 — 계획서 접근 4의 "scripts.ingest 경로로 변경"은 재검토 필요 (실물 확인 후 결정)

Python `unittest.mock`의 원칙은 **"사용되는 곳(where it's looked up)을 patch"**다. 래퍼 제거 후 어댑터가 `from scripts.ingest import register_bronze_event`로 이름을 자기 모듈 네임스페이스에 바인딩하면, 올바른 patch 대상은 여전히 **`scripts.cdc.polling_adapter.register_bronze_event`**(모듈 내부 참조)이지 `scripts.ingest.register_bronze_event`가 아니다 — import 후 원본 모듈을 patch하면 이미 바인딩된 이름에는 반영되지 않아 mock이 무력화될 수 있다. 현재 테스트가 이미 `patch("scripts.cdc.trigger_adapter.register_bronze_event")`(L47)를 쓰고 있으므로, **patch 대상은 그대로 두거나 어댑터 모듈 경로로 유지**하는 것이 안전하다. import 스타일(모듈 참조 vs 이름 바인딩)에 따라 달라지므로 실제 코드 확인 후 확정한다.

### 테스트 격리 — 단위테스트는 워크트리(Z-pre) 실행 가능

`test_polling_adapter.py`·`test_trigger_adapter.py`는 `redis`·`pymysql`·`scripts.ingest`를 `sys.modules` 수준에서 mock → **DB 없이 격리 실행 가능**. 따라서 pytest는 Z-pre에서 통과 확인한다. 반면 실 DB·S3·NiFi 런타임이 필요한 스모크는 Z-post.
