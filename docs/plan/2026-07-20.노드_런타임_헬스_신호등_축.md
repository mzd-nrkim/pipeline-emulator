# 캔버스 노드 런타임 헬스 신호등 — deployStatus와 직교하는 runtimeHealth 축

> 상태: 머지완료-통테대기

현재 캔버스 노드는 **정적 배포 계획 축**(`deployStatus: 'active'|'planned'|'absent'`, `done/2026-07-20.노드_배포상태_표시_deployStatus.md`)만 시각화한다. 이는 "실측 프로덕션 파이프라인에 배포되어 있는가"라는 **설계 시점** 정보이지, "지금 이 순간 살아 있는가"라는 **런타임 상태**가 아니다. 사용자가 요청한 노드별 신호등(🟢🔴🟡)은 후자 — **런타임 헬스**다. 본 계획은 이를 `deployStatus`와 **직교하는 별도 축 `runtimeHealth`**로 도입해, 두 축이 한 필드에 뭉개지지 않게 한다(이 프로젝트가 이미 `outOfTeamScope` vs `deployStatus`에서 확립한 축 분리 원칙의 연장).

---

## 핵심 설계 질문 해소: "실배포 안 됨(absent)인데 무엇에 연동하나?"

`realTopology.ts`에서 `node-debezium`·`node-nifi`·`node-valkey`는 `deployStatus: 'absent'`다. 그러나 이는 **실측 프로덕션(hyundaimotor-lllm) 축**에서 "없음"이라는 뜻이고, **로컬 docker-compose 에뮬레이터에는 이 컨테이너들이 실재**한다(`docker-compose.yml`에 `debezium`·`nifi`·`valkey` 서비스 정의 존재).

즉 두 축은 **연동 대상이 다르다**:

| 축 | 진리 원천 | debezium 예 |
|----|-----------|-------------|
| `deployStatus` (정적) | 실측 프로덕션 파이프라인 | `absent` (프로덕션 미배포) |
| `runtimeHealth` (동적, 신규) | **로컬 에뮬레이터 docker 컨테이너 상태** | 컨테이너 실행 중이면 `up` |

→ **"absent인데 어떻게 연동?"의 답**: 헬스 신호등은 프로덕션 배포 여부가 아니라 **에뮬레이터 컨테이너의 실행/헬스 상태**에 연동한다. 프로덕션엔 없어도(absent) 에뮬레이터에서 컨테이너가 돌면 신호등은 초록일 수 있고, 두 정보는 노드 위에 **공존**한다("프로덕션엔 미배포지만 로컬에선 살아 있음"). 반대로 **컨테이너 자체가 없는 노드**(`node-dam`=외부 API, `node-kibana`=compose 부재, `node-docling`·`node-presidio`·`node-kure`=Airflow task이지 독립 컨테이너 아님)는 폴링 대상이 없어 `runtimeHealth: 'unknown'`(신호등 미표시 또는 회색 점등)으로 남는다.

## 목표

- `ToolNode`/`FlowNode.data`에 옵셔널 `runtimeHealth?: 'up'|'down'|'degraded'|'unknown'` 축이 추가되고, 미지정 시 `unknown`으로 폴백되어 **기존 전 노드 회귀 0**이다.
- ui-backend가 `GET /health/services`로 각 compose 서비스의 실행/헬스 상태를 집계 반환하고, 프론트가 이를 주기 폴링해 노드 신호등을 갱신한다.
- 신호등은 `deployStatus` 배지·`outOfTeamScope` grayout과 **시각적으로 독립**(별 위치·별 CSS 축)이며, 세 축이 한 노드에 과중첩 없이 공존한다.
- 컨테이너에 매핑되지 않는 노드는 신호등을 표시하지 않거나 `unknown` 회색 점등으로 남아 오해를 유발하지 않는다.
- 헬스 조회는 **read-only**(조회만) — 컨테이너 제어(start/stop)는 본 계획 범위 밖(형제 계획 `외부의존_서비스_ON_OFF_제어.md`).

## 접근 방법

1. **축 타입 추가** (`frontend/src/lib/api/types.ts`): `ToolNode`에 `runtimeHealth?: 'up'|'down'|'degraded'|'unknown'` 옵셔널 필드. 미지정 폴백 = `unknown`. `deployStatus`와 완전 독립.

2. **node → compose service 매핑 테이블** (신규, 프론트 또는 백엔드 SSOT): `node-debezium→debezium`, `node-nifi→nifi`, `node-mysql-container→mysql`·`node-mysql→mysql`, `node-es→elasticsearch`·`node-es-search→elasticsearch`, `node-seaweedfs→seaweedfs`·`node-s3-bronze→seaweedfs`, `node-valkey→valkey`, `node-zookeeper→zookeeper`, `node-mock-api→mock-api`, `node-airflow→airflow`. 매핑 없는 노드는 신호등 없음. **매핑을 백엔드·프론트 어디에 둘지 plan-review에서 확정**(형제 ON/OFF 계획과 공유 SSOT 권장).

3. **ui-backend 헬스 집계 엔드포인트** (`GET /health/services`): compose 서비스별 상태를 반환. 획득 방식 후보(plan-review 확정):
   - (a) `docker compose ps --format json` / `docker ps` subprocess 파싱 — docker.sock 불요 시 호스트 docker CLI 접근 방식 검토.
   - (b) docker Engine API(`/var/run/docker.sock`) 직접 조회 — 형제 ON/OFF 계획이 어차피 마운트하므로 인프라 공유.
   - **healthcheck 없는 서비스**(debezium·zookeeper)는 헬스 상태 부재 → 실행 여부만으로 `up`/`down`, 상세 헬스는 `unknown` 강등.
   - 매핑: 컨테이너 `running`+`healthy`→`up`, `running`+`unhealthy`→`degraded`, `exited`/부재→`down`, 조회 불가→`unknown`.

4. **프론트 폴링 + 신호등 렌더**:
   - `real-adapter.ts`에 `fetchServiceHealth()` 추가, 스토어에서 주기 폴링(주기·중단 조건 plan-review 확정 — 기본 5~10s, 탭 비활성 시 중단).
   - 폴링 결과를 node별 `runtimeHealth`로 매핑해 `FlowNode.data`에 주입(`buildNodesAndEdges.ts`가 정적 topology + 동적 health 병합).
   - `ToolFlowNode.svelte`에 신호등 인디케이터(🟢 up / 🔴 down / 🟡 degraded / ⚪ unknown)를 `deployStatus` 배지와 **다른 위치**(예: 카드 좌상단 점)로 렌더. 기존 배지·grayout CSS 축과 클래스 분리.

5. **회귀 안전**: `runtimeHealth` 미지정·매핑 없는 노드는 `unknown`으로 신호등 미표시 → 폴링/백엔드 미가동 시에도 화면은 기존과 동일(신호등만 빠짐).

## 실행 시 필수 고려사항

### 3축 직교 (deployStatus · outOfTeamScope · runtimeHealth)

세 축이 한 노드에 공존한다. opacity 과중첩(회색 곱연산) 방지 원칙(`deployStatus` 계획 "두 축 CSS 독립")을 3축으로 확장 — 신호등은 opacity를 건드리지 않고 **점(dot)/색 인디케이터**로만 표현해 grayout 축과 물리적으로 분리. 조합 매트릭스(예: `absent`+`up` = 흐린 카드 + 초록 점)를 plan-review 표로 확정.

### docker 상태 획득 방식 = 형제 ON/OFF 계획과 인프라 공유

헬스 조회(read-only)와 ON/OFF 제어(read-write)는 **동일한 docker 접근 경로**를 쓴다. **plan-review 확정(D-1): (b) docker.sock + `docker` SDK 경로**로 통일한다((a) host CLI subprocess는 컨테이너 내부에서 CLI·소켓 둘 다 필요해 실익 없음). 따라서 **형제 #2 `외부의존_서비스_ON_OFF_제어.md`의 I-1(sock 마운트 + docker SDK requirements)이 본 계획의 공유 선행**이다.

**plan-run 배치 순서**(중요): #2의 I-1(인프라)을 두 계획 착수 전 공유 기반으로 먼저 반영하거나, 두 계획을 동일 워크트리/브랜치에서 함께 진행한다. 헬스만 독립 머지하려면 #2의 sock 마운트 커밋을 헬스 브랜치에 선반영해야 한다. 매핑 SSOT(D-2, ui-backend 상수)도 #2와 공유하므로 **매핑 정의를 헬스 계획이 먼저 도입하고 #2가 재사용**한다(중복 정의 금지).

### 폴링 부하·수명주기

주기 폴링은 백엔드·docker에 부하를 준다. 탭 비활성 시 중단, 최소 주기 하한, 실패 시 backoff, `unknown` 폴백을 plan-review에서 확정. SSE(`sse.py` 기존 존재) 재사용 여부도 검토.

### healthcheck 부재 서비스

debezium·zookeeper는 `healthcheck:`가 없어 "실행 중이지만 정상 여부 불명"이다. 이 경우 `up`으로 단정하지 말고 `running=up(헬스 불명)` 취지로 처리하거나 별도 표기. 신호등이 거짓 초록으로 오해를 주지 않게.

### Node 정적/e2e 게이트 위치 (Z-post)

`frontend/package.json` 의존(check/build/vitest/playwright)은 gitignored `node_modules` 필요 → 워크트리 신뢰 불가 → **머지 후 원본 main(Z-post)**. ui-backend Python 테스트는 DB·docker 미의존 부분만 pre-gate, docker 실연동은 Z-post.
