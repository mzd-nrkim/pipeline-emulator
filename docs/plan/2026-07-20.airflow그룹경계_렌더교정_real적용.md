# Airflow 그룹 경계 렌더 교정 + real 모드 적용

데이터흐름 그래프에서 "Airflow가 한 스텝으로 돌고 끝난 뒤 presidio 등 스테이지가 이어지는" 것처럼 보이는 문제. 실행 실체는 presidio·docling·kure가 Airflow worker 태스크 안에서 도는 스테이지(`silver_2_masking.py:65` in-process `pii_engine` 호출, 별도 서비스 아님)인데, 그래프는 이를 반영하지 못한다. 재설계(`done/2026-07-20.실행주체구조_캔버스_재설계_airflow오케스트레이션.md`)가 sample(`mockTopology`)에만, 그것도 **렌더가 깨진 채로** 부분 착지했고, real(`realTopology`)은 옛 선형 peer 모델 그대로다.

현 상태(근거 확인 완료):
- **sample** (`frontend/src/lib/mock/topology.ts`): presidio·docling·kure에 `parentId:'node-airflow-group'` 있음(그룹 합성됨) → 그러나 **자식 노드가 서로 겹쳐 렌더**. 원인은 `buildNodesAndEdges.ts`가 자식에 **절대 위상좌표**(`:94` `x:d*COL_GAP`)를 주면서 `parentId`+`extent:'parent'`(`:155`)를 붙이는데, xyflow는 자식 `position`을 **부모 상대좌표로 재해석**한다 → 부모 원점 차감(상대변환) 누락 + 작은 그룹 박스(`:216-220`)의 `extent:'parent'` 클램프로 자식 3개가 박스 우측 경계에 포개진다.
- **sample** 추가 결함: 독립 `node-airflow` 노드(`:116-126`, 가상 `dagId:'lllm_pipeline'`)가 체인에 남아 `s3-bronze→node-airflow→docling`(`:280,283`)로 배선 → airflow가 "한 스텝"(허구)과 "경계 박스"(실체)로 **이중 표현**.
- **real** (`frontend/src/lib/mock/realTopology.ts`): `parentId`/`group` **0건**. 순수 평면 선형 체인(`:272-283` `s3-bronze→node-airflow→docling→presidio→kure→mock-api→{es→es-search, mysql}`). airflow 경계화 미적용.
- 부수: `node-mock-api.tool`이 두 파일 모두 `'presidio'` 오분류(sample `:252`, real `:244`) — 실제는 enrichment.

## 목표

- sample 데이터흐름 그래프에서 Airflow 그룹 박스 안 자식 노드(docling·presidio·kure)가 **겹치지 않고 가로로 나란히** 렌더된다.
- 독립 `node-airflow` "한 스텝" 노드가 제거(또는 그룹 경계로 흡수)되어, airflow가 그래프에서 **오직 경계 박스로만** 표현된다 — 이중 표현 제거.
- `realTopology`에도 동일한 그룹 경계 모델이 적용되어, real 모드 데이터흐름도 presidio·docling·kure를 Airflow 경계 안 스테이지로 표시한다.
- `node-mock-api.tool` 오분류(`presidio` → 실제 enrichment tool)를 두 토폴로지에서 정정한다.
- 위 변경 후에도 nodeclick 드릴다운·infra 뷰·엣지 viaTable 라벨이 정상 동작하고, 관련 테스트가 재설계 구조 기준으로 통과한다.

## 접근 방법

1. **좌표 상대변환 교정 (`buildNodesAndEdges.ts` — 핵심 버그)**
   - 자식 노드(`parentId` 보유)의 `position`을 **부모 그룹 원점 기준 상대좌표**로 변환한다. 그룹 박스(minX/minY)를 먼저 확정한 뒤 자식 `position.x -= groupX`, `position.y -= groupY` 재계산, 또는 그룹 내부 전용 레이아웃(자식만 별도 0-origin 가로 배치)으로 산출한다.
   - 그룹 박스 width/height를 **상대좌표 기준 자식 실측 폭 + 노드 크기(≈200×?) + 패딩**으로 재계산해 `extent:'parent'` 클램프가 자식을 접지 않도록 한다.
   - depth 계산(`computeDepths`)이 그룹 자식을 전체 그래프 depth에 넣는지 / 그룹 내부에서 재정렬하는지 결정(아래 D-1).

2. **독립 `node-airflow` 처리 (양 토폴로지)**
   - `node-airflow` 노드를 제거하고 진입 엣지를 `s3-bronze → docling`(그룹 첫 자식)으로 재배선. dependency 엣지(mysql/seaweedfs/valkey/es → node-airflow)는 **그룹 노드(`node-airflow-group`) 대상**으로 옮기거나 대표 스테이지로 재연결(아래 D-2).
   - 그룹 노드 메타(`buildNodesAndEdges.ts:206` `Airflow (CeleryExecutor)`)가 trigger·config를 표현하도록 필요한 필드를 그룹 노드로 이전(트리거 버튼이 airflow 노드에 걸려 있었다면 그룹으로).

3. **`realTopology` 그룹 모델 적용**
   - real 노드 presidio·docling·kure에 `group:'airflow'`+`parentId:'node-airflow-group'` 부여, 진입 엣지 재배선(2와 동일 패턴). real 고유 노드(`node-es-search` 등)·실제값(executor=SequentialExecutor, dagId 실제 목록, recognizers=phone,rrn,email,bank_account, nlpEngine=ko_core_news_lg)은 보존.
   - gold_4(mock-api)·gold_5(es)의 그룹 포함 여부는 D-3 결정.

4. **`node-mock-api.tool` 오분류 정정** — 두 토폴로지에서 `tool:'presidio'`를 실제 enrichment tool id로 교체(카탈로그에 적절 엔트리 확인, 없으면 displayNameOverride 유지 + tool id 정정). 아이콘/셀렉터 회귀 확인.

5. **테스트·기대값 재작성** — `buildNodesAndEdges.test.ts`(그룹 좌표·자식 상대위치·그룹 박스 크기), nodeclick/infra/graph-handle 관련 e2e를 변경 구조 기준으로 갱신. 단순 통과가 아니라 상대좌표·경계 배선 기대값 명시.

## 결정 항목 (착수 전 확정)

- **D-1 그룹 내부 레이아웃**: (a) 전체 그래프 depth 유지하되 상대변환만(그룹 폭이 전체 depth 간격에 종속) vs (b) 그룹 자식을 0-origin에서 가로 재배치(그룹 내부 독립 레이아웃). → 기본안 (b): 겹침 근본 차단, 그룹 폭 예측 가능.
- **D-2 dependency 엣지 재대상**: node-airflow 제거 시 `mysql/seaweedfs/valkey/es → node-airflow` dependency 엣지를 (a) `node-airflow-group`으로 vs (b) 유지 위해 airflow 노드를 그룹 노드로 대체. → 기본안 (a).
- **D-3 gold_4·gold_5 그룹 포함**: mock-api=외부 HTTP 서비스(경계 밖 + HTTP dependency 유지), es=gold_5 airflow DAG(경계 안 후보). → 기본안: mock-api 경계 밖 유지, es는 경계 밖 유지(색인 sink 성격)하되 재설계 원안과 정합 재확인.
- **D-4 xyflow group 노드 렌더**: `AirflowGroupNode.svelte` 이미 존재(`:19`) — 자식 핸들 좌표계·nodeclick 버블링이 상대변환 후 정상인지 실측 필요.
