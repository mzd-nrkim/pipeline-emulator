# infra 뷰 의존성 배선(dependency 채널) 교정 — docker-compose 실측 정합

> 상태: 미시작

`frontend/src/lib/mock/topology.ts`의 `dependency` 채널 엣지(= infra 뷰 그래프)가 `docker-compose.yml`의 실제 서비스 의존성과 어긋난다. 실측 대조 결과 **누락 2건·허위 1건·계층 오분류 1건**을 확인했다. 본 계획은 infra 뷰를 **이 repo의 실제 배선(compose ground truth)** 에 정합시킨다. 데이터 흐름(`data` 채널)·노드 카탈로그·노드 렌더(ToolFlowNode 재디자인)는 **본 계획 범위 밖**이며 건드리지 않는다.

> 별도 계획서 `2026-07-20.toolflownode_실제_n8n_룩_재디자인.md`(노드 시각 룩)는 그대로 유지된다. 본 계획은 **데이터(토폴로지) 레이어의 정확성**만 다룬다.

## 배경: 실측 대조표

`docker-compose.yml`의 `depends_on` + host env(`*_HOST`/`*_ADDRESS`/`*_ENDPOINT`/`*_URL`)를 provider→consumer 의존성으로 환산해, topology.ts의 `dependency` 채널 엣지와 대조했다.

| # | 엣지 | topology 현재 | compose 실제 근거 | 판정 |
|---|------|--------------|------------------|------|
| 1 | `node-valkey → node-debezium` | 없음 | debezium `depends_on: valkey(service_healthy)` + `DEBEZIUM_SINK_REDIS_ADDRESS: valkey:6379` + `DEBEZIUM_SOURCE_SCHEMA_HISTORY_INTERNAL_REDIS_ADDRESS: valkey:6379` | **누락 → 추가** |
| 2 | `node-es → node-airflow` | 없음 | airflow `depends_on: elasticsearch(service_healthy)` + `ES_HOST: elasticsearch` | **누락 → 추가** |
| 3 | `node-mysql-container → node-nifi` | 있음(264행) | nifi `depends_on: zookeeper`만. nifi env에 mysql 참조 **0건** | **허위 → 제거** |
| 3b | `node-zookeeper → node-nifi` | 없음 | nifi `depends_on: zookeeper` + `NIFI_ZK_CONNECT_STRING: zookeeper:2181` | **누락 → zookeeper 노드+엣지 신설** (D-2 확정) |
| 4 | `node-valkey` 계층 = `storage` | `INFRA_LAYER_MAP`에서 `storage` | topology 주석 `broker`, 카탈로그 성격도 브로커/캐시(영속 스토리지 아님) | **오분류 → 재분류** |
| — | `node-valkey → node-airflow` | 없음 | airflow `AIRFLOW__CORE__EXECUTOR: SequentialExecutor`(브로커 미사용) | **없음이 정답 → 유지·명문화** |

> **고아 노드 숨김은 기존 정상 동작**(`buildNodesAndEdges.ts:124-130`, 주석 "고아 노드 숨김"). 뷰별 채널 필터 후 엣지가 0인 노드를 숨긴다. 부수효과: (a) valkey는 현재 dependency 엣지가 없어 infra 뷰에서 **이미 숨겨져 있고**, 엣지 #1 추가로 비로소 나타난다(긍정적). (b) 허위 엣지 #3 제거 시 nifi가 고아화되나, #3b(zookeeper→nifi)로 실제 의존을 모델링해 nifi 가시성을 유지한다.

### 핵심 사실 (오해 정정)

- valkey→airflow가 없는 것은 **"airflow가 단일 노드라서"가 아니라** airflow가 `SequentialExecutor`라 메시지 브로커(Redis/valkey)를 쓰지 않기 때문이다. valkey를 브로커로 소비하려면 `CeleryExecutor`여야 하는데 이 repo는 그 구성이 아니다.
- 이 repo에서 valkey의 실제 소비자는 **Debezium 하나**다(Redis 싱크 + 스키마 히스토리 저장소).

### 추가 갭 (2026-07-20 후속 리뷰 — INFRA_LAYER_MAP·카탈로그 정합)

dependency 엣지(위 표) 외에 `INFRA_LAYER_MAP`(`buildNodesAndEdges.ts`)·카탈로그 라벨에서 4건을 추가 확인했다. G-1/G-2는 "죽은 매핑" 정리(삭제), G-3/G-4는 결정 선행(D-3/D-4).

| # | 항목 | 현재 | 판정 | 근거 |
|---|------|------|------|------|
| G-1 | `INFRA_LAYER_MAP['s3']` | `storage` 매핑 존재 | **삭제** | `s3`라는 node id 없음(노드는 `node-s3-bronze`/`node-seaweedfs`) — 어떤 노드와도 매칭 안 되는 죽은 키 |
| G-2 | `node-presidio`/`node-docling`/`node-kure` 매핑 | `processing` 매핑 존재 | **삭제(또는 방치)** | 세 도구는 airflow DAG **내부 실행**(SSOT 주석)이라 독립 compose 서비스 0건 → dependency 엣지 없어 infra 뷰에 절대 미표시. 매핑은 죽은 항목 |

## 목표

- infra 뷰 dependency 그래프가 `docker-compose.yml` 실제 의존성과 **1:1로 정합**한다(누락·허위 0건).
- `node-valkey → node-debezium` dependency 엣지가 추가되어 valkey의 실제 소비 관계가 그래프에 드러난다.
- `node-es → node-airflow` dependency 엣지가 추가되어 airflow의 ES 의존이 드러난다.
- `node-mysql-container → node-nifi` 허위 엣지가 제거되고, nifi의 실제 의존 대상 **zookeeper가 캔버스 노드로 신설**되어 `node-zookeeper → node-nifi` 엣지로 대체된다(D-2 확정). nifi는 infra 뷰에 계속 보인다.
- valkey가 `storage`가 아니라 **broker/messaging** 성격으로 재분류되어 infra 계층 배치가 의미상 정확해진다.
- valkey→airflow 부재가 **의도된 정답**임이 topology.ts 주석에 명문화된다(SequentialExecutor 근거).
- `data` 채널·카탈로그·ToolFlowNode 렌더는 **불변**(회귀 0).

## 접근 방법

1. **엣지 추가·제거는 `topology.ts`의 `edges` 배열 한 곳에서만** 수행한다(258~272행 dependency 블록). data 채널 엣지(242~257행)는 손대지 않는다.
   - 추가: `{ from: 'node-valkey', to: 'node-debezium', channels: ['dependency'] as (...)[] }`
   - 추가: `{ from: 'node-es', to: 'node-airflow', channels: ['dependency'] as (...)[] }`
   - 제거: `{ from: 'node-mysql-container', to: 'node-nifi', ... }`(264행).
   - 추가: `{ from: 'node-zookeeper', to: 'node-nifi', channels: ['dependency'] as (...)[] }` (D-2 확정 — nifi 실제 의존).
   - 각 엣지에 근거 주석(compose 라인 참조)을 단다(기존 주석 스타일 준수, 267~272행처럼).

1-b. **zookeeper 캔버스 노드 신설** (D-2): `topology.ts`의 `nodes` 배열에 `node-zookeeper`(tool: `zookeeper`) 추가. `toolCatalog.ts`에 `zookeeper` 엔트리 신설(displayName `Apache ZooKeeper 3.9`, vendor `Apache`, category `source`/coordination, icon·accent — 아이콘은 별도 아이콘 계획 전까지 이모지 폴백 유지). 카탈로그 무변경 원칙의 **유일한 예외**이며 신규 1건 추가에 한정, 기존 엔트리는 불변.

2. **valkey 계층 재분류** (`buildNodesAndEdges.ts`):
   - `INFRA_LAYER_ORDER`에 `broker` 계층을 도입하거나(기본안), valkey를 기존 계층 중 의미상 맞는 곳으로 이동. 기본안: `['storage', 'broker', 'ingestion', 'processing', 'serving', 'other']`로 확장하고 `INFRA_LAYER_MAP['node-valkey'] = 'broker'`.
   - 계층 추가 시 `computeInfraPositions`의 y축 layerIndex 배치가 자동 반영되는지 확인(빈 계층은 `continue`로 건너뛰므로 순번 보정 불필요 — 기존 로직 재사용).

3. **valkey→airflow 부재 명문화**: topology.ts dependency 블록에 주석 추가 — `/* airflow는 SequentialExecutor라 valkey(브로커) 의존 없음 — 의도적 부재 */`. 향후 "누락 아니냐" 재질문 차단.

4. **infra 뷰 렌더 계약 불변**: 노드 data shape·핸들·채널 필터 로직(`buildNodesAndEdges`의 `view==='infra'` 분기)은 건드리지 않는다. 엣지 집합과 계층 매핑만 바뀐다.

## 실행 시 필수 고려사항

### 데이터 소스만 변경, 렌더 계약 불변

핵심 변경은 `topology.ts`의 `edges`·`nodes` 배열과 `buildNodesAndEdges.ts`의 `INFRA_LAYER_MAP`/`INFRA_LAYER_ORDER` + `toolCatalog.ts`에 zookeeper 엔트리 **1건 신설**이다. `FlowNode.data` shape·`type:'tool'`·핸들 개수·채널 필터 로직·기존 카탈로그 엔트리는 불변 → 두 뷰(data/infra) 렌더 회귀 표면 최소.

### D-0 결정 항목 (착수 전 확정)

1. **infra 뷰 기준**: 이 repo 실제 배선(compose) vs 일반 아키텍처 교육용. → **기본안: 실제 배선(compose ground truth)**. 사용자 확정(2026-07-20): 실제 배선 기준.
2. **mysql-container → nifi(허위 엣지) 처리** (D-2): **확정 = 제거 + zookeeper 노드 신설 + zookeeper→nifi**. compose `depends_on: zookeeper` + `NIFI_ZK_CONNECT_STRING`이 nifi의 실제 의존이므로, 허위 mysql 엣지를 제거하고 zookeeper를 캔버스 노드로 신설해 정확히 모델링한다. → nifi는 infra 뷰에 계속 보이고 고아화되지 않는다.
3. **valkey 계층** (D-1): 신규 `broker` 계층 추가 vs 기존 계층 재배치. → **기본안: `broker` 계층 신설**.

### 회귀 표면 실측 (plan-review에서 확인 — 위험 낮음)

두 테스트 표면을 실측한 결과 이번 변경의 회귀 위험은 **낮다**:

- **단위테스트 `buildNodesAndEdges.test.ts`는 독립 합성 fixture(`sampleTopology`)를 사용** — 실제 `topology.ts`가 아니다. 따라서 T 섹션(실제 topology.ts 엣지·노드 편집)은 단위테스트를 **직접 건드리지 않는다**. data 뷰 단언(`toHaveLength(11)`·`toHaveLength(10)`·`position.x % 280`)은 data 채널·COL_GAP 기준이라 dependency 변경·계층 변경과 무관. infra 뷰 단언은 "빈 topology 크래시 없음"(no-throw)뿐 — **infra 위치 단언 없음** → B 섹션(INFRA_LAYER_ORDER에 broker 추가)도 단위테스트를 깨지 않는다.
  - 단, 이 fixture 주석(14행)·엣지(60행)에 구(舊) `mysql-container → nifi`와 zookeeper 부재가 남아 있다. **fixture는 실제 topology와 별개라 수정 불필요**(합성 데이터). 일관성을 위해 갱신할 수는 있으나 본 계획 범위 밖(선택).
- **e2e `infra-view.spec.ts`·`pipeline-canvas.spec.ts`는 exact count가 아니라 견고한 단언 사용** — `edgeCount/markerCount > 0`(20·32행), `infraNodeCount !== dataNodeCount`(pipeline-canvas 50행), `esY > storageY`·`esY !== storageY`(74·116행) 등 상대 비교·부등호. 엣지 ±2·노드 +1·broker 계층 추가로도 이 단언들은 유지된다(broker 계층은 ingestion 이하 y를 한 행 밀지만 `esY>storageY` 관계는 불변). 하드코딩 카운트 단언 없음 → **조정 불필요 예상**. Z-post에서 실측 재확인.

핸들·채널 필터 계약은 불변이므로 구조 셀렉터는 전부 유지된다.

### 고아 노드 가시성 (기존 정상 동작)

`buildNodesAndEdges`는 **가시 그래프에서 연결된 노드만** 렌더한다(`connectedIds`, 124-130행, 주석 "고아 노드 숨김") — 신규 도입 아닌 **기존 의도된 동작**이다. 뷰별 채널 필터 후 엣지 0인 노드를 숨긴다. 본 계획의 관련 부수효과:
- valkey는 현재 dependency 엣지가 없어 infra 뷰에 **이미 숨겨져 있음** → 엣지 #1(valkey→debezium) 추가로 처음 나타난다.
- mysql-container→nifi 제거만 하면 nifi가 고아화되어 사라지므로, **zookeeper 노드+엣지(#3b)로 실제 의존을 모델링**해 nifi 가시성을 유지한다(D-2 확정).
- D-2로 nifi 고아화는 해소되나, plan-run 시 **다른 노드의 의도치 않은 고아화가 없는지** infra/data 양쪽 뷰 노드 수를 실측 확인한다.

### data 채널 불변 검증

valkey는 `data` 채널에서 `kure → valkey → {es, mysql}` 파이프로 등장한다(253·256·257행). 본 계획은 이 3개 data 엣지를 **건드리지 않는다**. dependency 채널만 변경.

### Node 정적/e2e 게이트 위치 (Z-post)

`frontend/package.json` 의존(`npm run check`/`build`/`vitest`/`playwright`)은 gitignored `node_modules` 필요 → 워크트리(Z-pre)에서 신뢰 불가 → **머지 후 원본 main(Z-post)** 실행. DB·마이그레이션 변경 없음(순수 mock 데이터 + 배치 로직).
