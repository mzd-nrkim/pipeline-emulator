# @xyflow/svelte v1 업그레이드 + n8n 스타일 커스텀 노드

> 상태: 머지완료-통테대기

pipeline-emulator 그래프를 n8n 에디터 수준으로 개선한다. 현재 `@xyflow/svelte` **0.1.39**의 기본 노드(`type: 'default'`)에 아이콘·이름·역할을 텍스트 한 줄로 욱여넣어 밋밋하고, handle이 노드 상·하단에 붙어 좌→우 흐름과 어긋난다. 표현에 필요한 데이터(`icon`·`accent`·`vendor`·`displayName`·`role`·`trigger`·`configFields.applyMode`·엣지 `condition`)는 `toolCatalog.ts`에 **이미 완비**되어 있고, 없는 것은 "노드 위에 그려주는 뷰"뿐이다. 프로젝트는 이미 Svelte 5이므로 1.x(runes 네이티브, 최신 1.6.2)로 올려 커스텀 노드를 제대로 구현한다.

> **기존 계획서 대체 관계**: `docs/2026-07-20.그래프_엣지_LR_handle_변경.md`(미시작)는 0.1.39 유지 + handle 좌/우 배치만 하는 최소안이다. 본 계획이 그 목표(좌 입력·우 출력 handle)를 **포함·대체**한다 — 커스텀 노드 `ToolFlowNode`가 좌/우 handle을 기본 배치하기 때문. 본 계획 완료 시 기존 LR-handle 계획서는 폐기(archive) 대상. **폐기 여부는 사용자 확인 후 처리**한다.

---

## 목표

- `@xyflow/svelte`가 **1.x(1.6.2)** 로 업그레이드되어 Svelte 5 runes(`$state.raw`·`bind:`) 네이티브 API를 사용한다.
- 그래프의 모든 노드가 **큰 아이콘·벤더 브랜드색(accent) 헤더·displayName/vendor** 를 갖는 커스텀 노드로 렌더링된다(n8n 스타일).
- 노드의 **입력(좌)·출력(우) handle이 라벨과 함께** 표시되어, 각 연결이 무슨 역할인지 보인다. switch/route 노드(`airflow-branch`)는 `cases`별 **다중 출력 handle**을 갖는다.
- 노드 위에서 **조건(엣지 condition)·가용성(applyMode: 🟢실시간/🟡재기동/🔵코드/🔴읽기전용)·trigger 가능 여부·실행 상태**가 배지로 드러난다.
- 기존 기능(노드 클릭 → 상세 모달, data/infra 뷰 토글, 실행 이력·비교)이 **회귀 없이** 동작한다.

## 접근 방법

1. **버전 업그레이드 (선행)**: `frontend/package.json`의 `@xyflow/svelte`를 `^1.6.2`로 올리고 `npm install`. 0.1.x와 1.x는 breaking change가 있어 코드 마이그레이션과 한 묶음으로 진행한다.

2. **v1 API 마이그레이션 (2개 컴포넌트)**: 공식 마이그레이션 가이드 기준 —
   - `writable<Node[]>` / `writable<Edge[]>` → `let nodes = $state.raw([...])` / `let edges = $state.raw([...])` (불변 취급, 재할당으로 갱신).
   - `<SvelteFlow nodes={store} edges={store}>` → `<SvelteFlow bind:nodes bind:edges>`.
   - `on:nodeclick={handler}`(CustomEvent) → `onnodeclick={({ node }) => ...}` 콜백. `event.detail.node` → `node`.
   - `$effect`로 topology→nodes/edges를 재계산하던 부분은 `$state.raw` 재할당으로 유지.
   - 대상: `ToolCanvasView.svelte`, `PipelineGraphView.svelte`.

3. **커스텀 노드 컴포넌트 `ToolFlowNode.svelte` 신규**:
   - `let { data }: NodeProps = $props()` — `data`에 `icon`·`accent`·`vendor`·`displayName`·`role`·`trigger`·`applyMode`·상태 필드를 담아 전달.
   - 레이아웃: accent 색 좌측 보더/헤더 바 + 큰 아이콘(이모지) + `displayName`(굵게)·`vendor`(작게) + 하단 배지 행(role/가용성/trigger/status). 기존 `StageNode.svelte` 카드 패턴 재활용, DESIGN.md 토큰(`--surface`·`--border`·`--primary`·`status-*`) 준수.
   - `<Handle type="target" position={Position.Left} />`(입력, 좌) + `<Handle type="source" position={Position.Right} />`(출력, 우). 필요 시 handle 옆 라벨.
   - **다중 출력**: route 계열 노드는 `cases`(또는 엣지 condition 집합)별로 `id`가 다른 source handle을 여러 개 렌더. 엣지는 `sourceHandle`로 특정 handle에 연결.

4. **`buildNodesAndEdges.ts` 정리**:
   - `type: 'default'` → `type: 'tool'`.
   - 인라인 `style` 문자열(ROLE_STYLE·getNodeStyle) 방식은 커스텀 노드가 스타일을 담당하므로 **accent/role/trigger/applyMode를 `data`로 전달**하도록 전환(색은 노드 컴포넌트 내부에서 결정). infra 뷰 스타일 분기도 `data`+뷰 플래그로 이관.
   - route 노드용 `sourceHandle` 매핑이 필요하면 엣지 생성부에서 condition→handle id 연결.

5. **nodeTypes 등록**: 두 뷰 컴포넌트의 `<SvelteFlow>`에 `nodeTypes={{ tool: ToolFlowNode }}` 추가.

6. **테스트/회귀 방어**: e2e 6개 파일이 `.svelte-flow__node`·`.svelte-flow__edge` 등 xyflow **내부 CSS 클래스**와 nodeclick 동작에 의존. 이 클래스는 1.x에서도 유지되지만, 노드 내부 텍스트 구조가 바뀌므로 텍스트 매칭 셀렉터(`allInnerTexts` 등)는 조정이 필요할 수 있다 — 실제 실행으로 확인.

## 실행 시 필수 고려사항

### 버전 업그레이드는 단일 선행 Phase

`package.json` 수정 + `npm install`은 이후 모든 코드 마이그레이션의 전제. 반드시 **먼저 단독 완료**한 뒤 컴포넌트 마이그레이션을 시작한다(0.1.39 API로 짜면 install 후 깨짐).

### 동일 파일 편집 충돌

`ToolCanvasView.svelte`·`PipelineGraphView.svelte` 각각은 (① v1 API 마이그레이션 ② nodeTypes 등록 ③ import 추가)가 **같은 파일**을 건드린다 — 파일별로 **한 에이전트가 순차 처리**한다. plan-run이 별개 Phase로 쪼개 동시 편집하지 않도록 한다.

### Phase 의존성

- 커스텀 노드를 import·등록하는 Phase는 `ToolFlowNode.svelte` 생성 Phase 완료 후 실행(없으면 `npm run check` 실패).
- `buildNodesAndEdges.ts`의 `type`·`data` 변경과 `ToolFlowNode`가 기대하는 `data` 필드는 **계약(shape)이 일치**해야 함 — 두 작업의 인터페이스를 먼저 확정한다.

### Node 정적/e2e 게이트 위치

`frontend/package.json` 존재 → `npm run check`·`build`·`vitest`·`playwright`는 gitignored `node_modules` 의존이라 **워크트리(Z-pre)에서 신뢰 불가 → 머지 후 원본 main(Z-post)** 에서 실행한다.

### 현재 코드 실측 — 계획서 전제와의 차이 (plan-review 실물 확인 결과)

- 현재 노드 `type`은 `'default'`가 **아니라 이미 `'lrnode'`**이고, `frontend/src/lib/components/LRFlowNode.svelte`(10줄, 좌 target·우 source handle)가 **이미 존재**한다. 즉 "handle 상·하단" 최소안(LR-handle 계획서)은 이미 반영됐다. 본 계획의 실질 작업은 (a) v1 API 마이그레이션, (b) `LRFlowNode` → 리치 `ToolFlowNode` 교체, (c) 배지·다중출력 추가다. 목표/접근의 "type:'default'→'tool'"·"상·하단 handle" 표현은 이 실측으로 대체해 읽는다.
- `buildNodesAndEdges.ts`는 이미 `data`에 `displayName/vendor/icon/accent/role/trigger`를 채운다(라인 173–179). 따라서 C의 "data 전달"은 대부분 완료 상태이고, 남은 것은 (i) 인라인 `style`/`getNodeStyle`(라인 247)·`ROLE_STYLE`(라인 4–13) 제거, (ii) 배지용 신규 필드(`applyMode`·`status`) 추가다.
- 엣지 `condition`은 현재 `FlowEdge.label = e.condition`(라인 189)로만 쓰인다. route용 **`sourceHandle` 매핑은 미구현** → C-3은 신규 작업이며 `FlowEdge` 인터페이스(라인 110)에 `sourceHandle` 필드 추가가 선행된다.

### 배지 데이터 소스 갭 (실물 확인 후 결정)

- **applyMode**: `toolCatalog`의 `applyMode`는 `configFields[]` **필드별** 값(runtime/restart/code/readonly)이다. 노드 단위 배지로 쓰려면 대표값 산출 규칙(예: 가장 제약 강한 값 우선)을 정해야 한다 → 작업 목록 C-0 결정 항목.
- **status(실행 상태)**: 현재 `buildNodesAndEdges(topology, view)` 시그니처·`data` 어디에도 노드 실행 상태가 없다. status 배지를 그리려면 실행 이력/결과를 노드에 매핑해 주입하는 경로가 필요하다 → 소스 확보 방법 또는 **이번 범위 제외**를 실물(실행 이력 연동 코드) 확인 후 결정(C-0).

### 기존 LRFlowNode·e2e handle count 회귀

- `ToolCanvasView.svelte`·`PipelineGraphView.svelte` 모두 `LRFlowNode`를 import하고 `nodeTypes={{ lrnode: LRFlowNode }}`로 등록한다. `ToolFlowNode` 도입 시 두 뷰의 import·nodeTypes 키·`buildNodesAndEdges`의 `type` 문자열을 **동시에** 바꿔야 한다(하나라도 불일치 시 노드 미렌더). 미사용된 `LRFlowNode.svelte`는 삭제한다(F-1).
- `e2e/graph-lr-handle.spec.ts`는 `.svelte-flow__handle-left/right` **개수**를, `route-split.spec.ts`는 route 분기를 단언한다. `ToolFlowNode`는 좌 target 1개 + 우 source(≥1)를 유지해야 하며, route 다중출력으로 우 handle 수가 늘면 count 단언이 회귀할 수 있다 → Z-post e2e에서 실측 조정.
