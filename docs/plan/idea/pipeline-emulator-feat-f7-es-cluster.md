## 목표

ES 단일 노드를 3-master 클러스터로 확장해 HA·부하 분산을 시연한다.

## 전환 트리거

고가용성/부하가 시연 대상이 될 때. **단, F1 검색 서빙이 켜져 있어야 유효.**

## 실행 시 필수 고려사항

- **착수 보류 전제**: 이 계획은 **보류**(우선순위 최하위) — F1(검색 서빙 `on`) 전제 + 노트북 3-master 리소스 부담. 명시적 HA/부하 시연 요구가 생길 때만 착수.
- **의존성**: F1 미구현 시 무의미 → 착수 전 F1 `on` 확인이 **선행 게이트**(설정 메뉴 규칙: F1 off면 토글 조작 불가).
- **환경 전제**: Z-post e2e는 3-master 컨테이너 기동 전제(리소스 큼, 워크트리 불가).
- **미선택 결정**: 없음(방향 = 보류 유지).

### Z-post. push 후 (앱 기동 환경)

- [ ] `ES=single` 복귀 정상(회귀)

- [ ] **C(에러/HA)**: 노드 1개 다운 → 검색 지속(무중단), 클러스터 yellow 허용·red 아님.

## 리스크 & 헤지

| 리스크 | 헤지 |
|--------|------|
| 노트북 리소스 부담 큼 | **낮은 우선순위** — 데모 가치 대비 무거움. 실제 요구 발생 시에만 착수 |
| F1 미구현 시 무의미 | 의존성 명시 — F1 `on`이 아니면 토글 조작 불가(설정 메뉴 규칙) |

## 비고

전체 로드맵에서 우선순위 최하위. 노트북 환경에서 3-master는 리소스 부담이 크므로, 고가용성/부하가 명확한 시연 요구가 될 때만 착수한다.
