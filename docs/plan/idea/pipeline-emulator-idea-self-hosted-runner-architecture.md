# [아이디어·장기] Self-Hosted Runner 아키텍처 (호스티드 컨트롤플레인 + 아웃바운드 로컬 에이전트)

> ⚠️ 상태: **아이디어 레벨 — 지금 당장 실행하지 않는다.**
> 이 문서는 방향·설계 질문을 박제하는 용도다. 원자 작업·체크박스는 착수 결정 이후 정식 실행계획서로 승격할 때 상세화한다. **plan-run 대상이 아니다.**

멀티 고객·팀·이력 중앙화까지 감당하는 "제품" 단계의 목표 아키텍처. 컨트롤 플레인(오케스트레이션·UI·이력)은 서버가 호스팅하고, 각 고객 로컬에는 **아웃바운드로 서버에 상시 연결되는 에이전트(runner)**가 실행 플레인(로컬 docker)을 구동한다. GitHub Actions self-hosted runner · Temporal worker · GitLab Runner와 동일 계열 패턴.

> 전제: 이 제품의 판매 포인트 = **원본/PII 데이터가 고객 로컬을 벗어나지 않는 data residency.** 이 경계는 어떤 설계에서도 불변.

## 왜 장기이고 왜 아직 실행하지 않는가
- 단기 목표(데모·단일 사용자 실동작)는 `...control-execution-plane-split.md`(모델 A/B1)로 충분히 달성된다.
- B2(아웃바운드 에이전트)는 페어링·보안·에이전트 배포/버전관리·연결관리 등 **운영 복잡도가 크고**, 실제 멀티고객 수요가 확인된 뒤 착수하는 게 비용 대비 옳다.
- 따라서 지금은 **설계 질문을 정리만** 하고, 단기 계획이 만든 트러스트 경계·분리 배포를 승격 발판으로 재사용한다.

## 개념 구조 (아이디어)
```
[서버: 컨트롤 플레인]                         [고객 로컬: 실행 플레인]
 UI · 오케스트레이션 · 이력DB   ◀── 아웃바운드 상시연결 ──  에이전트(runner)
      │  (명령/DAG config/상태 폴링)                     │  docker-compose
      │                                                  │  airflow/presidio/...
   메타데이터·상태만 수신  ◀───────────────────────────  원본데이터는 로컬 상주
```

## 풀어야 할 설계 질문 (착수 시 정식 plan으로 전개)
- **연결 방식**: websocket vs gRPC 스트림 vs long-poll. 아웃바운드 전용(인바운드 홀 없음) 원칙.
- **페어링/등록**: 조직·사용자별 등록 토큰, 에이전트 인증·회수(revoke), 재연결.
- **명령 프로토콜**: 서버→에이전트 잡(트리거/설정적용) 스키마, 에이전트→서버 상태/로그 스트림 스키마.
- **트러스트 경계 강제**: 원본/PII가 서버로 새지 않도록 payload 화이트리스트·검증(단기 계획 A-1 분류표 재사용).
- **멀티테넌시**: 고객별 격리, 이력·설정 분리, 권한.
- **에이전트 배포·업데이트**: 설치(installer/docker one-liner/homebrew), 버전 호환, 자동 업데이트, 롤백.
- **관측성**: 에이전트 하트비트·연결 상태·잡 실패 알림.
- **오프라인/열화**: 에이전트 단절 시 큐잉·재시도·UI 표기.
