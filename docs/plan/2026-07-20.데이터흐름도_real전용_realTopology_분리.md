# 데이터흐름도 real 모드 전용 realTopology 분리 — sample=이상 / real=실측

`/real/pipeline` 데이터흐름도의 노드 config가 실제 인프라(docker-compose·DAG·mock-api·pii_engine)와 다수 어긋난다(사실오류: 임베딩 차원 768→실측 384, index `lllm-docs`→실측 `pdis_cft`, `ml-node-1` 허위, executor CeleryExecutor→실측 SequentialExecutor 등). 원인은 **real 모드도 `mockTopology`를 그대로 반환**하기 때문이다(`real-adapter.ts:61-64`). 이 mockTopology는 목표 프로덕션 아키텍처(hyundaimotor-lllm: KURE·docling·lllm 네이밍)를 모사한 것이라 로컬 emulator 실구현과 별개다.

방향(사용자 확정): **sample(mock) 모드 = 이상 아키텍처(현행 mockTopology 유지), real 모드 = 실측(docker-compose/DAG 실제값)으로 분리**. 어댑터 분기(`+layout.ts:6`)는 이미 모드별로 나뉘어 있어 real-adapter만 별도 `realTopology`를 반환하면 된다.

> 조사 근거: `docs/dataflow-topology-fidelity-audit.md`

## 목표

- real 모드(`/real/pipeline`) 데이터흐름도가 **신규 `realTopology`**를 사용해 실측값(index=pdis_cft, dim=384, MiniLM, SequentialExecutor, DB=pipeline_emulator 등)을 표시한다.
- sample 모드(`/sample/pipeline`)는 **현행 `mockTopology`(이상 아키텍처: KURE·docling·lllm·dam)를 무변경**으로 유지한다.
- real 모드에서 실제 ES 색인 DAG(gold_6_es_indexing)가 데이터흐름도 노드로 트리거 가능해진다(node-es-search 추가 또는 node-es 재배선).
- 실측 일치 노드(debezium·seaweedfs·zookeeper·valkey·mock-api)와 렌더 계약(toolCatalog·buildNodesAndEdges·ToolFlowNode)은 **불변**이다.

> **분리 축은 뷰가 아니라 모드다.** 데이터흐름 뷰·인프라 뷰는 별도 토폴로지가 아니라 **하나의 `CanvasTopology`를 채널(`data`/`dependency`)로 필터**한 두 뷰다(`buildNodesAndEdges.ts:116-121`). 따라서 real-adapter가 `realTopology`를 반환하면 real 모드의 **데이터흐름 뷰·인프라 뷰가 동시에** realTopology를 쓴다. 이 계획의 교정 대상은 **노드 config 값 + data 채널 엣지 + 가상 노드**이며, **dependency 채널 엣지(인프라 뷰 배선)는 mockTopology에서 이미 docker-compose depends_on/env 기반 실측 반영**(`topology.ts:276-294`)이라 realTopology에서도 **원칙적으로 동치 유지**한다(dam 제거 등 노드 삭제에 따른 엣지 정리만 반영). 인프라 뷰를 별도로 손대지 않는다.

## 접근 방법

1. **`realTopology.ts` 신규 작성** (`frontend/src/lib/mock/realTopology.ts` 또는 `$lib/api/realTopology.ts`): `mockTopology`를 복제한 뒤 위 실측 표대로 노드 config를 교정한다. 타입은 기존 `CanvasTopology` 재사용(스키마 변경 없음).
2. **real-adapter 배선 교체** (`real-adapter.ts:59-64`): import를 `mockTopology`→`realTopology`로 바꾸고 `fetchCanvasTopology()`가 이를 반환. mock-adapter는 손대지 않는다 → sample 회귀 0.
3. **사실오류 교정(값)**: 임베딩(model/dim/위치), ES(index/mlNode/색인 DAG), executor, PII(recognizer/nlpEngine), DB명 등을 realTopology 노드 config에만 반영.
4. **구조 갭 해소(node-es-search)**: `ui-backend`에 이미 존재하는 `node-es-search→gold_6_es_indexing` 매핑(`airflow.py:37`)에 대응하는 노드를 realTopology에 추가하거나, node-es의 dagId를 gold_6으로 재지정한다. 두 안의 트레이드오프는 아래 "실행 시 필수 고려사항" 참조.
5. **가상/profile-gated 노드 처리**: DAM(가상), nifi·debezium·valkey(profile-gated)는 real 모드에서 `deployStatus: 'absent'`(이미 존재하는 필드, `types.ts:110`)로 표기하거나 노드 제거 — 결정은 고려사항 참조.

## 실행 시 필수 고려사항

### 사용자 결정 필요 (plan-review 전 확정 권장)
- **A. node-es 재배선 vs node-es-search 추가**: (a) node-es.dagId를 gold_6로 바꾸면 노드 1개로 색인까지 커버되나 gold_5(staging) 트리거 경로가 사라진다. (b) node-es-search 노드를 추가하면 staging(gold_5)·indexing(gold_6)이 분리 표현되고 ui-backend 매핑과 1:1이나 노드가 늘어 흐름도가 복잡해진다. → **권장 (b)**: 백엔드 STAGE_DAG_MAP과 정합.
- **B. DAM/profile-gated 소스**: real 기본 프로파일에서 dam은 아예 없고 nifi·debezium·valkey는 `COMPOSE_PROFILES` 옵션이다. real 흐름도에서 (a) 제거 vs (b) `deployStatus:'absent'/'planned'`로 흐리게 표기. → **권장 (b)**: 인프라 존재 가능성을 표현하되 실측 상태 구분.

### 렌더 계약·카탈로그 불변
`realTopology`도 기존 `CanvasTopology`/`ToolNode` 타입을 그대로 쓰므로 `toolCatalog.ts`·`buildNodesAndEdges.ts`·`ToolFlowNode.svelte` 수정 불필요. 노드 `tool` 값이 기존 카탈로그 엔트리 id와 일치해야 카드가 정상 렌더된다(신규 tool 도입 금지 — 값만 교정).

### sample 회귀 0
mock-adapter는 `mockTopology`를 계속 반환(무변경). realTopology는 real-adapter만 참조 → sample 모드 렌더·config 회귀 표면 없음. 두 파일이 topology를 공유하지 않도록 import 출처를 분리한다.

### ui-backend 정합 (T3)
node-es-search 추가 시 노드 id는 반드시 `node-es-search`(백엔드 `airflow.py:37` 키와 동일)여야 트리거가 STAGE_DAG_MAP을 탄다. 신규 id를 만들면 트리거 404(`nodes.py:22`)가 난다.
