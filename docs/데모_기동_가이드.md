## 1. 가장 중요한 사실 — 샘플 모드는 백엔드가 필요 없다

발표에서 쓰는 화면은 `/sample/pipeline`이고, 이건 **mock-adapter**를 쓴다.

```ts
// frontend/src/routes/[mode=mode]/pipeline/+page.svelte:16
const currentAdapter = page.params.mode === 'real' ? realAdapter : mockAdapter;
```

`mock-adapter.ts`에는 `fetch`가 한 줄도 없다. 전부 하드코딩된 데이터를 반환한다.

```ts
export async function fetchStages() { return stages; }          // 고정 데이터
export async function triggerNode(nodeId, _conf) {
  return { dag_run_id: `mock-run-${nodeId}-${Date.now()}` };     // 가짜 ID 즉시 반환
}
export async function fetchServiceHealth() {
  return { 'node-airflow': 'up', 'node-mysql': 'up', 'node-es': 'up' };  // 무조건 up
}
```

**따라서 Docker가 전부 꺼져 있어도 시연 화면은 완벽히 정상이다.** 서비스 상태 표시도 초록으로 나온다.

시연에 필요한 것은 **vite dev 하나뿐**이다. 장애 리스크가 사실상 없다는 뜻이므로, 발표 전에 Docker를 붙들고 씨름할 이유가 없다.

| | `/sample/pipeline` (발표용) | `/real/pipeline` |
|---|---|---|
| 데이터 | 하드코딩 mock | ui-backend → MySQL/Airflow |
| 트리거 | 가짜 run_id 즉시 반환 | 실제 DAG 실행 |
| Docker | **불필요** | 필수 |
| 기동 시간 | ~10초 | 3~5분 |

## 3. 시연 중 말할 때의 정직성 (중요)

샘플 모드는 **고정된 예시 데이터**다. 기능 자체는 `pii_engine`·DAG에 실제로 구현되어 있지만, 화면에서 지금 돌고 있는 것은 아니다. 이 구분이 무너지면 신뢰가 무너진다.

**안전한 표현**
- "이 파이프라인은 문서를 받아서 마스킹하고 청킹·색인까지 흘립니다" (시스템이 하는 일)
- "마스킹 단계에서는 전화번호·주민번호·이메일·계좌번호를 이렇게 가립니다" (기능 설명 + 예시 화면)

**위험한 표현**
- "지금 실행된 결과입니다" / "방금 돌린 run입니다" → 샘플 모드에서는 사실이 아니다
- 서비스 상태 초록을 가리키며 "전부 살아 있습니다" → mock이 무조건 up을 반환한다

**대본 부록 A의 정직성 기준 (그대로 유지)**

실제로 구현된 것 — 자신 있게 말해도 됨
- Bronze→Silver→Gold 7개 DAG
- PII 정규식 마스킹 4종 (전화·주민·이메일·계좌)
- 캔버스 · 실행이력 · 로그 · run 비교 UI

과장 금지
- Debezium·NiFi는 **선택 프로필**이라 기본 경로에 없다 → "실시간 CDC가 지금 돌고 있다"고 하지 않는다
- 설정 토글 일부(검색 서빙·Presidio·분산 실행·ES 클러스터)는 **"다음 계획" 배지가 붙은 mock**
- 이름·주소 같은 비패턴 PII는 Presidio Layer2 = **계획 단계**

미리 밝히고 갈 것
- 데모 스택(MySQL·Elasticsearch)이 제품 코어(Trino·Iceberg)와 다른 것은 **의도된 대역**이다. 먼저 말하면 강점, 지적당하면 약점.

샘플 모드는 의존성이 vite 하나뿐이라 고장날 구석이 거의 없다.

| **노트북이 버벅임** | Docker 컨테이너가 떠 있으면 정리: `docker compose down` (샘플 모드엔 불필요) |

## 부록 A. 실제 모드 (`/real/pipeline`) — 지금 발표에는 쓰지 않음

진짜 DAG를 돌려 실제 데이터로 시연하려면 이쪽이다. 스크립트는 `scripts/demo/real-mode/`에 보존해 두었다.

```bash
./scripts/demo/real-mode/start-demo.sh      # Docker 6개 + 헬스 대기 + 프론트엔드
./scripts/demo/real-mode/check-demo.sh      # 컨테이너·엔드포인트 진단
./scripts/demo/real-mode/fallback-demo.sh   # 폴백 (영상/스크린샷)
```

### 준비물과 함정

- 백엔드 6개: mysql · elasticsearch · seaweedfs · mock-api · ui-backend · airflow
- **기동만으로는 화면이 채워지지 않는다.** 2026-07-24 확인 시점 기준 실제 데이터는 비어 있었다:

  ```
  /stages → docsIn:0, docsOut:0, status:"none"
  /runs   → []
  ```

  화면을 채우려면 DAG를 실제로 실행해 데이터를 만들어야 한다. 리허설 시간을 넉넉히 잡을 것.
- 첫 DAG 실행이 가장 느리다(파싱·커넥션 초기화). 미리 1회 소진해 둔다.
