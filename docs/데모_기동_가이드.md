## 1. 발표 30분 전 — 기동

```bash
cd ~/workspace_mzd/pipeline-emulator
./scripts/demo/start-demo.sh
```

이 스크립트가 하는 일:
1. Docker 데몬 확인
2. **포트 충돌 검사** (다른 프로젝트 컨테이너와 겹치는 사고가 잦다)
3. `docker compose up -d` — 백엔드 6개
4. 각 서비스가 응답할 때까지 대기 (Airflow는 최대 4분)
5. 프론트엔드(vite dev) 기동 — 이미 5173이 떠 있으면 건드리지 않음
6. 시연 URL 출력

정상 종료 후 브라우저에서:

```
http://localhost:5173/sample/pipeline
```

### 리허설에서 반드시 할 것

**캔버스에서 `masking-task` 트리거를 1회 실제로 돌린다.**

첫 실행이 가장 느리다(DAG 파싱·커넥션 초기화). 리허설에서 미리 소진해 두면 본 시연에서 체감 속도가 크게 다르다. 이것만은 생략하지 말 것.

## 2. 포트 맵

| 서비스 | 포트 | 확인 URL |
|---|---|---|
| Frontend (PipeScale) | 5173 | http://localhost:5173/sample/pipeline |
| UI Backend (FastAPI) | 8001 | http://localhost:8001/health |
| Airflow | 8080 | http://localhost:8080 |
| Mock API | 8000 | http://localhost:8000/docs |
| MySQL | 3306 | — |
| SeaweedFS (S3) | 8333 | — |
| Elasticsearch | 9200 | http://localhost:9200 |
| Valkey | 6379 | *선택 프로필(`celery`)에서만 사용* |

> **주의**: 이 노트북에는 `airflow-tutorial`, `airflow_demo`, `kubeflow` 등 **다른 프로젝트 컨테이너가 상시 떠 있다.** 8080을 쓰는 스택이 여럿이라 충돌 가능성이 있다. `start-demo.sh`가 자동 검사하지만, 수동 확인은:
>
> ```bash
> lsof -nP -iTCP:8080 -sTCP:LISTEN
> ```

## 3. 증상별 대처

| 증상 | 즉시 조치 |
|---|---|
| **캔버스가 안 열림 (5173 응답 없음)** | `cd frontend && npm run dev` — 가장 빠른 복구. 백엔드는 대개 멀쩡하다 |
| **캔버스는 뜨는데 노드 데이터가 비어 있음** | UI Backend 문제. `docker compose restart ui-backend` (약 10초) |
| **트리거를 눌러도 run_id가 안 나옴** | Airflow가 아직 기동 중. http://localhost:8080/health 확인. 기동 완료까지 2~3분 걸린다 |
| **트리거 후 medallion 증거가 0건** | MySQL 집계 지연. 노드를 다시 클릭해 패널을 새로 연다 |
| **Docker 데몬이 죽음** | Docker Desktop 재실행 → 안정까지 1~2분. **이 시점이면 폴백이 정답** |
| **포트 충돌** | 충돌 컨테이너 정지: `docker stop <이름>` — 단, 남의 스택을 끄는 것이므로 발표 전에만 |
| **전부 정상인데 느림** | 첫 실행이면 정상. 해설을 이어가며 기다린다 (대본 부록 A 톤 유지) |

### 폴백 시 멘트

> "환경 문제로 라이브 대신 녹화로 보여드리겠습니다. 내용은 동일합니다."

- **사과를 길게 하지 않는다.** 한 문장으로 넘기고 바로 재생.
- 영상 중에도 해설은 그대로: 문서 유입 → 구조화 → 마스킹 → 청킹·색인.
- 강조점도 그대로: **"코드 없이 모듈을 갈아끼워도 파이프라인이 흐른다."**

하지 말 것: 복구하며 침묵하기 / "원래는 되는데" 반복하기 / 터미널 에러 화면 보여주기.

## 5. 시연 중 지켜야 할 정직성 (대본 부록 A에서)

라이브든 폴백이든 **말할 수 있는 것과 없는 것**은 동일하다.

**실제로 동작 — 자신 있게 말해도 됨**
- Bronze→Silver→Gold 7개 DAG
- PII 정규식 마스킹 4종 (전화·주민·이메일·계좌)
- 캔버스 · 실행이력 · 로그 · run 비교 UI

**과장 금지 — 물으면 솔직히**
- Debezium·NiFi는 **선택 프로필**이라 기본 데모 경로에 없다 → "실시간 CDC가 지금 돌고 있다"고 말하지 않는다
- 설정 토글 중 일부(검색 서빙·Presidio·분산 실행·ES 클러스터)는 **"다음 계획" 배지가 붙은 mock** → 토글 시연은 실동작 축(마스킹 등) 위주로
- 이름·주소 같은 비패턴 PII는 Presidio Layer2 = **계획 단계**

**미리 밝히고 갈 것**
- 데모 스택(MySQL·Elasticsearch)이 제품 코어(Trino·Iceberg)와 다른 것은 **의도된 대역**이다. 먼저 말하면 강점이 되고, 지적당하면 약점이 된다.

## 6. 발표 후 정리

```bash
cd ~/workspace_mzd/pipeline-emulator
docker compose down                        # 백엔드 정지
kill $(cat .demo-logs/frontend.pid)        # 프론트엔드 정지 (start-demo.sh로 띄운 경우)
```

컨테이너를 계속 켜 두면 노트북이 느려진다. 발표 끝나면 정리할 것.
