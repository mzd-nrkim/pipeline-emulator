# mock-api runtime 라벨 ↔ 구현 격차 해소

`frontend/src/lib/canvas/toolCatalog.ts`의 mock-api 노드는 `responseTemplate`·`latencyMs`·`statusCode` 3필드를 `applyMode: 'runtime'`으로 표시하지만, 이를 실제로 적용할 경로가 없다. `ui-backend/app/api/nodes.py`의 `config_node`는 `RUNTIME_KEYS = {"is_paused", "variable"}`만 처리해 mock-api 키가 오면 전부 `readonly`("이 항목은 런타임 변경 불가")로 반려한다. 게다가 `mock_api/` 서비스는 `/chunk`·`/enrich`·`/health`만 노출할 뿐 설정 변경 엔드포인트가 없고, 이 서비스는 "청킹/엔리치 목"이지 응답을 통째로 커스터마이즈하는 범용 목 HTTP가 아니다 — 즉 세 필드는 서비스 실체와도 어긋난다. **라벨과 동작을 일치시켜야 한다.**

---

## 목표

- mock-api의 configField `applyMode`가 **실제 ui-backend 동작과 일치**한다 — UI가 "runtime 적용 가능"이라 표시하고 실제론 readonly 반려하는 모순이 사라진다.
- 사용자가 mock-api 설정을 저장했을 때의 결과(적용됨/불가)가 **예측 가능**해진다.
- (옵션 A 채택 시) mock_api 서비스가 런타임 설정 변경 엔드포인트를 노출하고 nodes.py가 이를 중계한다.

## 접근 방법

두 옵션 중 택일한다. **권장: 옵션 B(라벨 강등)로 즉시 정합화**, 옵션 A(실동작 구현)는 데모 가치가 확인되면 후속으로.

### 옵션 B — 라벨 강등으로 문서-코드 일치 (권장)

**근거**: mock_api 서비스는 청킹/엔리치 목이라 `responseTemplate`(범용 응답 템플릿)이 서비스 동작과 매핑되지 않는다. 세 필드는 "이 목 API가 이렇게 응답한다"를 **서술하는 표시용 메타데이터**에 가깝다. 무리하게 runtime을 구현하기보다 실제 성격(변경 불가 or 재기동 반영)으로 강등하는 것이 정직하고 저위험이다.

1. `toolCatalog.ts` mock-api 정의(292~294행)에서:
   - `responseTemplate` → `readonly` (목 응답은 라우트 코드에 하드코딩; 런타임 변경 개념 없음).
   - `latencyMs`·`statusCode` → `restart` 또는 `readonly` 중 택1. 환경변수로 주입 가능하게 만들 여지가 있으면 `restart`, 순수 코드 정의면 `readonly`. **판단**: 현재 mock_api에 이 값을 읽는 코드가 없으므로 `readonly`가 정확(작업 중 mock_api 코드 재확인).
2. nodes.py는 무변경(이미 mock-api 키를 readonly로 반려하므로 라벨만 맞추면 동작 일치).
3. `tool-api-survey.md`의 mock-api runtime 표기도 동반 수정(문서 갱신 계획서와 조율 — 중복 편집 방지 위해 순서 합의).

### 옵션 A — 실동작 구현 (후속/선택)

**범위가 커지고 서비스 성격 재정의가 필요**하므로 데모 시나리오상 "런타임에 목 응답을 바꿔 파이프라인 반응을 시연"할 가치가 있을 때만 채택한다.

1. **mock_api에 설정 상태 도입**: `mock_api/`에 in-memory 설정 스토어(`state.py` — `latency_ms`, `status_code`, `response_template`) 추가. `latencyMs`·`statusCode`는 `/chunk`·`/enrich` 응답에 미들웨어로 주입(지연·상태코드 오버라이드). `responseTemplate`은 적용 대상이 모호하므로 이 옵션에서도 제외 검토.
2. **설정 엔드포인트**: `POST /config` 라우트 추가 — body의 필드로 스토어 갱신, 재기동 없이 반영.
3. **nodes.py 중계**: `config_node`에서 node가 mock-api일 때 mock_api 서비스 `POST /config`로 프록시. `RUNTIME_KEYS`를 노드 타입별로 분기(airflow: is_paused/variable, mock-api: latencyMs/statusCode)하도록 구조 확장. 하드코딩 set 대신 노드→핸들러 매핑 테이블 도입 고려.
4. **테스트**: mock_api `POST /config` 후 `/chunk` 응답 지연·상태코드가 반영되는지 통합 테스트.

### 옵션 트레이드오프

| | 옵션 B (강등) | 옵션 A (구현) |
|---|---|---|
| 정합성 | 즉시 달성 | 달성(구현 후) |
| 작업량 | 소 (라벨 3줄 + 문서) | 중~대 (서비스 상태·엔드포인트·중계·테스트) |
| 데모 가치 | 없음(현상 유지) | 있음(런타임 목 제어 시연) |
| 위험 | 낮음 | 중(서비스 성격 재정의·회귀) |
| responseTemplate 처리 | readonly로 정리 | 여전히 매핑 모호 |

**권장**: 지금은 **옵션 B**로 모순을 제거하고, 옵션 A는 별도 기능 요청으로 승격될 때 이 계획서를 근거로 재개한다.

## Verification

- **옵션 B**: mock-api 노드 상세에서 `responseTemplate`/`latencyMs`/`statusCode`가 runtime 배지가 아닌 readonly(또는 restart)로 표시된다. 해당 필드 저장 시도 시 UI가 "런타임 변경 불가"로 일관 안내(더 이상 runtime 라벨과 반려 응답이 모순되지 않음).
- **테스트/빌드**: `cd frontend && npm run test` 통과, `npm run build`·`npm run check` 오류 없음.
- **문서 일치**: `tool-api-survey.md`의 mock-api applyMode가 `toolCatalog.ts`와 일치.
- **(옵션 A 채택 시)**: mock_api `POST /config`로 latencyMs 설정 후 `/chunk` 응답 지연이 실제 반영(통합 테스트 수치 근거), nodes.py `config_node`가 mock-api 키를 `applied`로 반환.
