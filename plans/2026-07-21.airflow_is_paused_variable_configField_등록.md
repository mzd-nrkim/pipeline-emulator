# apache-airflow is_paused·variable configField 등록

ui-backend에는 `set_paused()`·`set_variable()`(`app/services/airflow.py`)와 이를 중계하는 `config_node`의 `RUNTIME_KEYS = {"is_paused", "variable"}`(`app/api/nodes.py`)가 **이미 구현**돼 있으나, 프론트엔드 `toolCatalog.ts`의 apache-airflow configField에는 이 두 런타임 경로가 **등록돼 있지 않다**. 결과적으로 backend가 지원하는 runtime 기능을 UI에서 노출·조작할 수 없다. `tool-api-survey.md`도 이를 "향후 추가 후보"로 명시하고 있다. 두 필드를 configField로 등록해 backend 능력을 UI로 끌어올린다.

---

## 목표

- apache-airflow 노드 상세에서 `is_paused`(DAG 일시정지)와 `variable`(Airflow Variable set)을 **UI로 조작**할 수 있고, 저장 시 기존 `POST /{node_id}/config` 경로로 실제 반영된다.
- 두 필드가 `applyMode: 'runtime'`으로 표시되어 즉시 적용 가능함이 UI 배지에 드러난다.
- `variable`의 key-value(dict) 구조가 UI에서 입력 가능한 형태로 처리된다(backend가 기대하는 dict payload와 일치).
- `tool-api-survey.md`의 "configFields 미등록" 비고가 해소된다.

## 접근 방법

### A. is_paused 필드 등록 (단순)

1. `toolCatalog.ts` apache-airflow configFields(86~101행 영역)에 추가:
   `{ key: 'is_paused', label: 'DAG 일시정지', type: 'boolean', group: '동작', applyMode: 'runtime' }`.
2. 저장 흐름: 프론트 저장 핸들러가 `POST /{node_id}/config` body의 `config`에 `{ is_paused: <bool> }`를 실어 보내면 `config_node`가 `set_paused(dag_id, bool(value))` 호출 → 이미 동작. **backend 변경 불필요.**
3. `_mapped_dag_id(node_id)`가 apache-airflow 노드에 대해 유효한 dag_id를 반환하는지 확인(STAGE_DAG_MAP). 반환 없으면 is_paused 적용 불가 → 매핑 보강 필요 여부 판단.

### B. variable 필드 등록 (key-value 구조 — 필드 타입 결정 필요)

`config_node`는 `variable` 값이 **dict**이길 기대(`isinstance(value, dict)` 검사, 아니면 에러)한다. 현 `FieldType`은 text·number·select·boolean뿐이라 dict 입력 수단이 없다. 두 방안 중 택1:

1. **(권장) JSON text 필드 + 파싱**: `{ key: 'variable', label: 'Airflow Variable(JSON)', type: 'text', placeholder: '{"key":"value"}', group: '동작', applyMode: 'runtime' }`. 저장 핸들러가 이 text를 `JSON.parse`해 dict로 만들어 `config.variable`에 넣는다. 파싱 실패 시 UI 검증 에러. **장점**: FieldType 확장 불필요, 최소 변경. **단점**: 사용자가 JSON을 직접 써야 함.
2. **(대안) 신규 `keyvalue` 필드 타입 도입**: `FieldType`에 `'keyvalue'` 추가 + 전용 key-value row 에디터 컴포넌트. `ConfigField` 인터페이스·렌더러·저장 직렬화까지 확장. **장점**: UX 우수. **단점**: 컴포넌트·타입·직렬화 전반 손대 범위 큼. 데모 목적엔 과할 수 있음.

**권장**: 방안 1(JSON text)로 최소 침습 등록. keyvalue 에디터는 별도 UX 개선 과제로 분리.

### C. 저장 핸들러 정합 확인

- 프론트의 config 저장 핸들러(노드 상세 모달/패널)가 `applyMode: 'runtime'` 필드를 `POST /config`로 보내는 경로가 이미 있는지 확인. `conf`(기존 runtime 필드)가 어떻게 전송되는지 추적해 동일 경로 재사용.
- 응답의 `applied`/`readonly`/`errors` 배열을 UI가 표시하는지 확인(부분 성공 피드백). 없으면 최소 토스트/인라인 표시 추가.

### D. 문서 동반 갱신

- `tool-api-survey.md` apache-airflow 표에 `is_paused`·`variable` 행을 정식 등록(비고의 "향후 추가 후보" → "등록됨"). 문서 갱신 계획서와 순서 조율(중복 편집 방지).

## Verification

- **is_paused**: apache-airflow 노드 상세에서 일시정지 토글을 켜고 저장 → 응답 `applied`에 `{"key":"is_paused","status":"ok"}`, 실제 Airflow에서 해당 DAG가 paused 상태로 전환(REST `GET /dags/{id}` 또는 UI 확인).
- **variable**: JSON `{"foo":"bar"}` 입력·저장 → 응답 `applied`에 `variable/foo` 성공, Airflow Variable `foo=bar` 생성(REST `GET /variables/foo` read-back). 잘못된 JSON 입력 시 UI가 저장 전 검증 에러.
- **배지**: 두 필드가 노드 상세에서 runtime 배지로 표시되고, 노드 applyMode 우선순위(`buildNodesAndEdges`)에 runtime이 반영된다.
- **테스트/빌드**: `cd frontend && npm run test` 통과, `npm run build`·`npm run check` 오류 없음. backend 무변경 확인(`git diff` 범위가 frontend + 문서에 한정, ui-backend 코드 변경 없음).
- **문서**: `tool-api-survey.md`에서 is_paused·variable이 "미등록"이 아닌 정식 configField로 기재.
